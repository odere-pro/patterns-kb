Performance

Serving more requests, faster, without the cost growing in step: spread work across instances, keep hot data close to the request, and spend less memory getting there.

Theme

The question

Every system that gets used enough eventually hits the same wall: the same query, the same computation, the same object, done again and again for every request. The naive answer is more hardware — but a bigger machine or a bigger fleet only postpones the wall if the underlying work is still repeated in full each time. Performance as a theme is the discipline of not doing that: do the work once and reuse the result, do the work closer to the requester, or don't do the work at all if you can rule it out cheaply.

That reframes "make it faster" into three separate questions. Where is the load concentrated, and can it be spread across more machines? What is recomputed on every request that didn't actually change, and can it be cached or precomputed instead? And where is memory or allocation itself the bottleneck, independent of any network call? Each question points at a different family of patterns, and conflating them is how teams end up caching a problem that was actually a routing problem, or scaling out a problem that was actually a memory-churn problem.

The trade-space

Almost every performance pattern buys speed by spending something else. Caching buys latency with freshness — a cached answer is a bet that the world hasn't changed since it was computed, and the bet is occasionally wrong. Precomputation buys read speed with storage and update complexity — a materialized view is fast to read and someone has to keep it in sync with its source. Pooling and sharing buy allocation savings with statefulness — a reused object can leak state between callers if it isn't reset carefully.

Spreading load geographically or across instances buys throughput with routing complexity: a load balancer or a consistent-hashing ring has to keep working correctly as the pool of nodes changes underneath it, or the cure becomes the outage. None of these trades are free, which is why the first move is always diagnosis — find which bottleneck you actually have before reaching for the pattern that fixes it.

Patterns that implement the choice

Load Balancer

Distributes incoming requests across a pool of interchangeable instances so no single node absorbs all the traffic. It turns "one machine is too slow under load" into an operational problem — add capacity, and the balancer spreads the new headroom across it.

Cache-Aside

The application checks the cache first, falls back to the source of truth on a miss, and populates the cache with the result. It's the default read-cache strategy precisely because the cache can be empty, wrong, or evicted at any moment without breaking correctness — the source of truth is still there.

Read-Through

Moves the cache-aside logic into the caching layer itself: callers ask the cache, and the cache asks the backing store on a miss. Every caller stops reimplementing the same fetch-or-populate dance.

Write-Behind

Acknowledges a write the moment it lands in the cache and flushes it to the backing store asynchronously. Writes feel instant to the caller; the durability guarantee shifts to whatever drains the flush queue.

CDN

Moves static and semi-static content to points of presence near the requester, so the round trip is single-digit milliseconds instead of crossing a continent. It's caching applied at the network layer rather than the application layer.

Materialized View

Runs the expensive join or aggregation once, ahead of time, and serves the result as a plain table read. The cost of the computation moves to whoever keeps the view in sync with its source.

Consistent Hashing

Maps keys and nodes onto the same ring, so adding or removing a node only reshuffles the keys nearest to it. It keeps cache hit rates high across a changing pool of instances instead of resetting them on every deploy.

Object Pool

Keeps a set of costly-to-construct objects — connections, threads, buffers — ready to hand out and reuse instead of allocating and discarding one per request. It trades a little bookkeeping for the allocation and GC cost it removes.

Flyweight

Splits an object into shared, immutable intrinsic state and small per-instance extrinsic state, so a million logical objects can share one copy of the heavy part. It's memory-side performance — fewer resident bytes means more of the working set fits in cache.

Bloom Filter

A compact, probabilistic structure that answers "definitely not present" or "maybe present" in constant time. Checking it before an expensive lookup turns wasted round trips on misses into a cheap in-memory check.
