Singleton

Guarantees a class is instantiated exactly once and hands every caller that same shared object through a single global access point.

GoF Creational

What it is

A singleton is a class that permits only one instance of itself and provides a well-known way to reach it. The class hides its own constructor so no one can call new from outside, keeps the sole instance in a static field, and exposes a static accessor — classically getInstance() — that creates the instance on first request and returns the same object forever after.

The forces it resolves are uniqueness and reach. Some resources are genuinely singular — a hardware device, an OS-level handle, one shared connection pool or cache — and having two of them is a bug, not a convenience. At the same time, that one object is needed all over the codebase, and threading it manually through every constructor and call is tedious. The singleton answers both at once: it enforces the single instance and makes it reachable from anywhere.

That very convenience is also its trap. A singleton is a controlled global variable, and global mutable state couples code invisibly, resists isolation in tests, and hides its dependencies. The pattern is disciplined enough to be defensible for truly singular resources and seductive enough to be badly overused everywhere else.

How it works

Variations

Eager initialization
The instance is built at class-load time. Simplest and inherently thread-safe, but the object is created even if it's never used.
Lazy initialization
The instance is created on the first getInstance() call, deferring expensive setup — at the cost of needing care under concurrency.
Thread-safe / double-checked locking
Guard the lazy creation with a lock and re-check inside it, so two concurrent callers can't each build an instance.
Monostate (Borg)
Allow many instances but back them all with shared static state, so every object behaves as one — uniqueness of state without uniqueness of object.
Module-level singleton
In module systems the module is cached after first load, so a plain module-level value is a de facto singleton with none of the boilerplate.

Trade-offs

Pros

  • One authoritative instance for genuinely singular resources — a config, a connection pool, a logger.
  • Global reach without threading the object through every constructor and call.
  • Lazy creation defers expensive setup until the instance is first actually needed.
  • Controls instantiation outright — it's impossible to accidentally spin up a second one.

Cons

  • Hidden global state — every user is coupled to the singleton without declaring it.
  • Hard to test: shared mutable state leaks between cases, and swapping in a mock is awkward.
  • Lifetime is process-wide; you can't cleanly scope, reset, or replace it per request.
  • Lazy creation invites concurrency races if the guarding is done carelessly.
  • Frequently a disguised global variable, reached for where explicit wiring would be clearer.

When to use it

Reach for it when

  • The resource is single by nature — a hardware device, an OS handle, one log sink.
  • An expensive, shared object must be reused everywhere, like a connection pool or cache.
  • You need exactly one coordination point and passing it around by hand is impractical.

Avoid when

  • You mainly want convenient access — prefer Dependency Injection so dependencies stay explicit and testable.
  • The object holds mutable state that tests or individual requests need to isolate.
  • Multiple configurations or scopes are plausible later — per-tenant, per-request, per-thread.

Code sketch

TypeScript — a lazy, single-instance config 
class Config {
  private static instance: Config | null = null;
  readonly port: number;

  // Private constructor blocks `new Config()` from outside.
  private constructor() {
    this.port = Number(process.env.PORT ?? 8080);
  }

  static getInstance(): Config {
    if (Config.instance === null) {
      Config.instance = new Config(); // built on first request
    }
    return Config.instance;
  }
}

// Every caller shares the one instance.
const a = Config.getInstance();
const b = Config.getInstance();
console.log(a === b); // true
