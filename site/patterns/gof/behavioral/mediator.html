Mediator

Replaces a tangle of objects calling each other directly with a single coordinator that owns how they interact — colleagues talk to the mediator, never to one another.

GoF Behavioral

What it is

A mediator is an object that encapsulates how a set of other objects — its colleagues — interact. Instead of each colleague holding references to the others and calling them directly, every colleague knows only the mediator. When something happens, a colleague notifies the mediator, and the mediator decides who else needs to react.

The problem it resolves is tangled coupling. Let a group of objects talk to each other freely and you get a many-to-many web: every object references every other, the number of connections grows with the square of the count, and the rules governing their interaction end up smeared across all of them. Changing one interaction means editing several classes, and no single object can be reused without dragging its neighbours along.

The mediator turns that web into a hub-and-spoke. Colleagues depend only on the mediator; the mediator depends on the colleagues. The interaction logic lives in exactly one place and can be changed without touching the colleagues at all.

The canonical GoF example is a dialog box: a director coordinates widgets so that toggling a checkbox enables a button, or selecting a list item fills a text field — without any widget knowing the others exist.

How it works

Variations

Message / event bus
A fully decoupled mediator: colleagues publish typed messages and subscribe to the ones they care about, so senders and receivers never name each other.
Request dispatcher
An in-process mediator that maps a request object to exactly one handler (the MediatR style), keeping controllers thin and handlers isolated.
GUI director
The classic dialog mediator: one object owns the rules that link widgets, so each widget stays dumb about its siblings.
Hub / broker
A network-scale mediator — a chat server or air-traffic controller — where clients coordinate only through the central hub, never peer-to-peer.

Trade-offs

Pros

  • Decouples colleagues — they depend on the mediator, not on each other.
  • Interaction logic lives in one place instead of being smeared across many classes.
  • Collapses many-to-many references into a hub, cutting coupling from n² toward n.
  • Colleagues get simpler and more reusable in other contexts.

Cons

  • The mediator can swell as it absorbs every rule until it does too much.
  • Centralized control becomes a bottleneck and a single point of failure.
  • Indirection hides who actually triggers what, making a flow harder to trace.
  • Overkill when the objects barely interact or only talk one way.

When to use it

Reach for it when

  • A set of objects reference each other in complex, tangled ways.
  • Reusing an object is hard because it is wired directly to many others.
  • Interaction rules spread across classes are hard to change or understand.

Avoid when

  • Objects interact little, or only in one direction — a broadcast fits better.
  • There is a natural one-way flow that needs no central coordinator.
  • The mediator would only relay calls without owning any real logic.

Kept lean, it stops scattered coordination from congealing across your classes — but a mediator that hoards every rule becomes exactly the God Object it was meant to prevent.

Code sketch

TypeScript — a form director 
interface Mediator {
  notify(sender: Component, event: string): void;
}

abstract class Component {
  constructor(protected mediator: Mediator) {}
}

class Checkbox extends Component {
  checked = false;
  toggle() {
    this.checked = !this.checked;
    this.mediator.notify(this, "toggled");
  }
}

class SubmitButton extends Component {
  enabled = false;
  setEnabled(on: boolean) { this.enabled = on; }
}

class FormMediator implements Mediator {
  constructor(private terms: Checkbox, private submit: SubmitButton) {}

  notify(sender: Component, event: string) {
    // one place owns the rule: submit follows the terms checkbox
    if (sender === this.terms && event === "toggled") {
      this.submit.setEnabled(this.terms.checked);
    }
  }
}

In the wild

MediatRA .NET library named for the pattern: controllers send a request object to the mediator, which routes it to exactly one handler neither side names.
