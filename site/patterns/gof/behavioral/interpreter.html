Interpreter

Turns each rule of a small language into its own class, so a sentence becomes a tree of objects that knows how to evaluate itself.

GoF Behavioral

What it is

The Interpreter pattern gives a small language its own object model. You define one class for each rule in the grammar, then represent a sentence as an abstract syntax tree built from those classes. Every node exposes a single operation — interpret(context) — and evaluating an expression means calling interpret on the root and letting the call recurse down through the tree.

There are two kinds of node. Terminal expressions — literals and variables — sit at the leaves. Nonterminal expressions — rules composed of sub-expressions, like and, sequence, or repetition — sit at the branches and delegate to their children. That branch-and-leaf shape is the Composite pattern applied to a grammar.

The problem it resolves is a little language whose parsing and evaluation logic has sprawled into tangled if/switch code where every new rule touches everything. Interpreter trades that for a clean one-rule-one-class mapping: the grammar is the type hierarchy, a new rule is a new class, and the tree can be walked more than once — evaluated, pretty-printed, or optimized. It pays off only for simple, stable grammars; a real language belongs to a parser generator.

How it works

Variations

Terminal vs. nonterminal expressions
The core split: leaf nodes that resolve directly (a number, a variable) versus composite nodes whose result is defined in terms of their children.
Context object
External state — variable bindings, an input cursor, accumulated output — is threaded through as an explicit Context argument rather than hidden in the nodes.
Visitor
Move the interpret logic out of the node classes into a separate visitor, so you can add operations (evaluate, print, type-check) without editing every expression.
Flyweight terminals
Terminal symbols repeat constantly across an expression; share single instances of them to shrink the tree, as GoF pairs Interpreter with Flyweight.

Trade-offs

Pros

  • Grammar rules map one-to-one to classes — the code reads like the grammar.
  • Adding a new rule is adding a class; existing rules stay untouched.
  • The tree is reusable: evaluate it, print it, or transform it as data.
  • Each rule's behavior lives in one small, focused, testable class.

Cons

  • Class explosion — every rule is a class, so a rich grammar sprawls fast.
  • Recursive object trees are slow to build and walk over large inputs.
  • Only practical for simple, stable grammars, not evolving real languages.
  • Changing the node interface ripples across every expression class.

When to use it

Reach for it when

  • The language is small and well-bounded — a search filter, a rules engine, arithmetic.
  • The grammar is stable, and clarity matters more than raw throughput.
  • You reinterpret the same expressions repeatedly or manipulate them as trees.

Avoid when

  • The grammar is large or changing quickly — reach for a parser generator like ANTLR.
  • Performance over big inputs is critical; object trees won't keep up.
  • A regular expression or a simple lookup already does the job.

Code sketch

TypeScript — a boolean expression interpreter 
type Context = Record<string, boolean>;

interface Expr {
  interpret(ctx: Context): boolean;
}

class Var implements Expr {                    // terminal
  constructor(private name: string) {}
  interpret(ctx: Context) { return ctx[this.name] ?? false; }
}

class And implements Expr {                    // nonterminal
  constructor(private left: Expr, private right: Expr) {}
  interpret(ctx: Context) {
    return this.left.interpret(ctx) && this.right.interpret(ctx);
  }
}

class Not implements Expr {                    // nonterminal
  constructor(private expr: Expr) {}
  interpret(ctx: Context) { return !this.expr.interpret(ctx); }
}

// (x AND NOT y)
const tree = new And(new Var("x"), new Not(new Var("y")));
tree.interpret({ x: true, y: false });         // => true

In the wild

Common Expression Language (CEL)A non-Turing-complete expression language with no unbounded loops, so evaluation is guaranteed to terminate. Kubernetes runs CEL in CRD validation rules (x-kubernetes-validations) and ValidatingAdmissionPolicy; Envoy uses it for RBAC and access-log filters. Expressions parse to a protobuf AST that can be type-checked against a declared environment before running against a runtime activation.
Spring Expression LanguageParses expression strings into an AST evaluated against an EvaluationContext holding variables and a root object. SpelCompilerMode (OFF, IMMEDIATE, MIXED) can compile hot expressions to bytecode for speed, and SimpleEvaluationContext narrows what an expression may reach when the input is not fully trusted.

In production

Tuning knobs

  • Max nesting depth — A cap on how deeply an expression tree may nest, bounding recursion so a deeply nested input cannot exhaust the call stack of the evaluating thread.
  • Evaluation budget — A ceiling on time or step count for interpreting one expression, guarding against runaway or adversarial input, especially when the grammar allows repetition.
  • AST caching — Whether parsed expression trees are cached and reused or rebuilt on every evaluation. Reparsing per call is the usual throughput cliff.
  • Context scope — What variables, functions, and objects an expression may reach through its context. A restricted context is the guard when the expressions come from untrusted input.

Signals to watch

  • Evaluation latency — p99 time to walk one expression tree; it grows with tree size and nesting depth and is the first sign that an input is pathological.
  • Parse-to-evaluate ratio — How often expressions are parsed versus evaluated. A parse count that tracks the evaluate count means the AST cache is missing or absent.

Failure modes under load

  • Stack overflow on deep nesting — An adversarial deeply nested expression recurses past the stack limit and crashes the evaluating thread.
  • Runaway evaluation — An expression with large repetition or expansion runs far longer than intended, tying up a thread with no budget to stop it.
  • Reparse under load — Building the AST on every call instead of caching it collapses throughput once evaluation volume rises.

Readiness checklist

  • Expression nesting depth is capped before evaluation.
  • Untrusted expressions run under a time or step budget and against a restricted context.
  • Parsed ASTs are cached and reused rather than rebuilt per evaluation.
  • The grammar is confirmed simple and stable enough that an object tree beats a parser generator.
