Observer

When one object's state changes, every dependent that registered interest hears about it automatically — no polling, no manual wiring, just a single notify that fans out.

GoF Behavioral

What it is

The Observer pattern defines a one-to-many dependency between objects. A subject (the observable) keeps a list of observers and notifies them automatically whenever its state changes — typically by calling an update method that each observer implements.

The problem it resolves is this: you hold a piece of data that many parts of a system care about — a view, a logger, a cache, an analytics sink — and you don't want the object owning that data to know about every one of them. Hard-wiring the subject to each dependent couples them tightly, and forces you to edit the subject every time a new consumer appears.

Observer inverts that dependency. The subject exposes subscribe and unsubscribe, and depends only on an abstract observer interface — never on concrete dependents. Listeners come and go at runtime; the subject just broadcasts. You trade explicit, traceable calls for loose coupling and the freedom to add reactions without touching the source.

How it works

Variations

Push vs. pull
The subject either pushes the changed data into update(payload), or just pings observers and lets each pull what it needs via getState(). Push is convenient, pull keeps the subject ignorant of what observers want.
Publish-Subscribe
Insert a broker or event channel between subject and observers so neither holds a reference to the other — the same idea stretched across modules or processes.
Typed / per-aspect events
Notify with a topic or property name so observers subscribe to parts of the state and skip changes they don't care about.
Weak references / auto-unsubscribe
Hold observers weakly, or return a disposer from subscribe, so a listener that forgets to detach doesn't keep itself alive forever.

Trade-offs

Pros

  • Loose coupling — the subject knows only an interface, not its concrete dependents.
  • Observers can be added and removed at runtime without touching the subject.
  • One change broadcasts to many listeners with a single call.
  • Open/closed: new observer types plug in without editing the source.

Cons

  • Notification order is unspecified — observers can't assume they run in sequence.
  • Cascading updates: one notify triggers another, and the chain is hard to trace.
  • Lapsed listeners that never unsubscribe leak memory and fire on dead state.
  • Implicit, indirect control flow makes stepping through the code harder.

When to use it

Reach for it when

  • Several objects must react whenever another object's state changes.
  • You want the subject decoupled from the concrete set of dependents.
  • The set of listeners varies at runtime — they come and go.

Avoid when

  • There is exactly one dependent that never changes — a direct call is simpler and clearer.
  • Update ordering or transactional consistency across listeners matters.
  • Notifications must cross process boundaries — reach for Publish-Subscribe and a broker instead.

Code sketch

TypeScript — a minimal subject 
interface Observer<T> {
  update(value: T): void;
}

class Subject<T> {
  private observers = new Set<Observer<T>>();
  private value: T;

  constructor(initial: T) { this.value = initial; }

  subscribe(o: Observer<T>): () => void {
    this.observers.add(o);
    return () => this.observers.delete(o); // disposer to unsubscribe
  }

  get state(): T { return this.value; }

  set state(next: T) {
    if (next === this.value) return;       // no change, no notify
    this.value = next;
    for (const o of this.observers) o.update(next);
  }
}

In the wild

DOM addEventListenerAn element keeps a list of listeners per event type and calls each one when the event fires, with removeEventListener as the detach.
Node.js EventEmitterThe on/emit contract underlying streams and servers: emitters broadcast named events to any number of in-process subscribers.
RxJSBuilds the whole library on Observable and Subscriber, adding operators and an explicit unsubscribe to the basic notify contract.
