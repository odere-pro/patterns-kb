Chain of Responsibility

Hands a request down a line of handlers — each free to deal with it or pass it along — so the sender never needs to know who finally answers.

GoF Behavioral

What it is

The Chain of Responsibility strings a set of handler objects into a line. A request enters at the head; each handler either deals with it or forwards it to its successor. The sender fires the request into the chain without knowing — or caring — which handler ultimately answers.

The force it resolves is coupling the sender to the receiver. When several objects could service a request, wiring the caller to all of them produces a growing if/else ladder and a rigid, hard-to-reorder set. You want to issue the request once and let the runtime decide who takes it.

The pattern breaks that coupling by giving each handler a single reference to the next. Handlers stay small, focused, and independently orderable, and the chain itself becomes a list you can rebuild at runtime. The price: a request can fall off the end unhandled unless you install a catch-all at the tail.

How it works

Variations

Pure vs. impure chain
In the pure form exactly one handler consumes the request and stops; in the impure form several handlers may act on it as it flows past.
Middleware / interceptor pipeline
Each handler does part of the work then calls next() — the shape behind servlet filters, Express middleware, and ASP.NET pipelines.
Hierarchical bubbling
The successor link points up a tree instead of along a list; DOM event bubbling and exception propagation are chains that climb a hierarchy.
Command
Wrap each request as a Command object and let it travel the chain; handlers execute it or forward it untouched.

Trade-offs

Pros

  • Decouples the sender from the concrete handler that serves the request.
  • Each handler stays small and single-purpose — easy to test in isolation.
  • The chain is assembled at runtime: reorder, add, or drop handlers without touching callers.
  • Honors open/closed — a new handler slots in without editing the existing ones.

Cons

  • No guarantee a request is handled; it can silently fall off the end.
  • Behavior is spread across a dynamic list, so flow is harder to trace and debug.
  • A long chain adds call-stack depth and latency to every request.
  • A bad order — or a handler that forgets to forward — breaks the chain quietly.

When to use it

Reach for it when

  • More than one object can handle a request and the right one isn't known until runtime.
  • You want to issue a request to one of several handlers without naming the receiver.
  • The set of handlers, and their order, should be configurable.

Avoid when

  • Exactly one object always handles the request — a direct call is clearer.
  • Every request must be handled — a chain's fall-through becomes a silent bug.
  • The ordering never changes — an explicit sequence beats the indirection.

Code sketch

TypeScript — an abstract handler and two links 
abstract class Handler {
  private next?: Handler;

  setNext(handler: Handler): Handler {
    this.next = handler;        // return it for fluent chaining
    return handler;
  }

  handle(req: Request): Response | null {
    return this.next ? this.next.handle(req) : null;
  }
}

class AuthHandler extends Handler {
  handle(req: Request): Response | null {
    if (!req.token) return { status: 401 };   // consumed here
    return super.handle(req);                  // otherwise pass on
  }
}

class RateLimitHandler extends Handler {
  handle(req: Request): Response | null {
    if (overLimit(req.ip)) return { status: 429 };
    return super.handle(req);
  }
}

// wire the chain, then feed it a request
const chain = new AuthHandler();
chain.setNext(new RateLimitHandler());
chain.handle(incoming);

In the wild

Express.js middlewareEach middleware function either ends the request or calls next() to pass it to the following handler in the stack.
Java Servlet FiltersA filter chain wraps every request, with each filter calling doFilter to forward it down the chain.
ASP.NET Core middlewareThe request pipeline is an ordered chain of middleware components, each free to short-circuit or invoke the next.
