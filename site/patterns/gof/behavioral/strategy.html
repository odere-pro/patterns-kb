Strategy

Wraps a family of interchangeable algorithms behind one interface, so a program can swap which one it runs at runtime without touching the code that calls it.

GoF Behavioral

What it is

The Strategy pattern defines a family of algorithms, puts each one behind a common interface, and lets a client pick which member to use. The object that needs the work — the context — holds a reference to a strategy and delegates to it, never knowing or caring which concrete implementation it is talking to.

The force it resolves is a method that keeps sprawling into conditionals. A single routine grows a switch over a "mode" or "type", each branch a different way of computing the same result. Every new variant means editing that method and risking the others. Strategy pulls each branch out into its own class or function, so the variation lives in separate, self-contained units instead of tangled together.

Because the choice is a plain object reference, it can change at runtime: swap the strategy on the context and the next call behaves differently. That makes behavior configurable — by a setting, a feature flag, or the shape of the request — without recompiling or branching in the hot path.

How it works

Variations

Function strategies
In languages with closures, a strategy is just a function passed in — no interface or class ceremony, only a callable with the right signature.
Null Object default
A do-nothing strategy stands in when none is configured, so callers never have to branch on a missing one.
Strategy registry
Register named strategies in a map and select one by key from config, a feature flag, or the request itself.
Policy-based design
Bind the strategy at compile time through generics or templates, trading runtime flexibility for zero dispatch overhead.

Trade-offs

Pros

  • Switch between algorithms while the program runs, without changing the code that calls them.
  • Replaces one sprawling if/switch with small, single-purpose units.
  • Each algorithm is isolated, so you can test it on its own.
  • Add a new algorithm without editing the existing ones (open for extension).

Cons

  • More types and indirection than a simple inline branch when there are only two trivial cases.
  • The caller has to know enough to pick the right algorithm.
  • Algorithms that share data need a common context object, or awkward parameter passing.
  • Overkill when the set of algorithms is fixed and will never grow.

When to use it

Reach for it when

  • You have several interchangeable ways to do one job and choose between them while running.
  • A method is turning into a growing switch or if/else over a "kind" or "mode".
  • You want to add new behaviors without editing the code that invokes them.

Avoid when

  • There is only one algorithm, or two that will never grow into more.
  • The variants differ only in a single constant — pass a parameter instead.
  • The behavior really tracks an object's lifecycle changes — that is State, not Strategy.

Code sketch

TypeScript — swappable pricing strategies 
interface PricingStrategy {
  price(subtotal: number): number;
}

// Interchangeable algorithms behind one interface
const listPrice: PricingStrategy = { price: (s) => s };
const memberPrice: PricingStrategy = { price: (s) => s * 0.9 };

class Checkout {
  #strategy: PricingStrategy;

  constructor(strategy: PricingStrategy) {
    this.#strategy = strategy;
  }

  use(next: PricingStrategy): void {   // swap at runtime
    this.#strategy = next;
  }

  total(subtotal: number): number {
    return this.#strategy.price(subtotal);
  }
}

const cart = new Checkout(listPrice);
cart.use(memberPrice);   // now 10% off
cart.total(100);         // => 90

In the wild

java.util.ComparatorSorting order is passed in as an interchangeable object, so one sort routine — Collections.sort, Arrays.sort, Stream.sorted — handles any ordering the caller supplies. Combinators like comparing, thenComparing and reversed compose new orderings without writing a new class.
Passport.jsCalls its authentication mechanisms strategies outright: each provider — local, OAuth, JWT — is a plug-in object registered with passport.use() that the middleware delegates to via passport.authenticate(name), never knowing which one it holds.
