Objects & Classes · Structural

Bridge

Splits a class into two hierarchies — an abstraction and its implementation — joined by a reference, so each can evolve on its own without dragging the other along.

GoF Structural

What it is

The Bridge pattern takes one class that mixes two independent concerns and cleaves it into two separate hierarchies: an abstraction that clients talk to, and an implementation it delegates to. The abstraction holds a reference to an implementation object — that reference is the bridge. Because they meet only across an interface, either side can be extended without touching the other.

The problem it resolves is subclass explosion. When a type varies along two orthogonal dimensions — say a shape (circle, square) that must render on two platforms (raster, vector) — a single inheritance tree forces one class per combination: RasterCircle, VectorCircle, RasterSquare, and so on. Add a third shape or a third platform and the count multiplies. Bridge turns that multiplication into addition: m abstractions plus n implementations, wired together at runtime.

The two forces it balances are the desire to extend behaviour freely and the cost of a rigid, combinatorial class tree. It sides with composition over inheritance: what the object is and how it does its work become two things you can vary separately.

How it works

Variations

Pimpl idiom
C++'s pointer-to-implementation: a class exposes a stable public face while its private members live behind an opaque pointer — a compiler firewall that keeps ABI and build dependencies in check.
Handle / Body
The classic name for the same split — a lightweight handle that clients pass around, backed by a heavier body object it can share or swap.
Runtime-swappable implementor
The abstraction chooses or replaces its implementation at runtime — selecting a platform driver, a storage backend, or a rendering engine based on configuration or environment.
Hierarchical implementor
The implementation side itself grows a small tree (a base implementor with specialised variants), letting a family of related backends share behaviour behind one interface.

Trade-offs

Pros

  • Two dimensions vary independently — combinations become additive, not multiplicative.
  • Implementations can be selected or swapped at runtime, not baked in at compile time.
  • Hides implementation details from clients, acting as a compiler and dependency firewall.
  • Open for extension on either side without editing the other.

Cons

  • Adds a layer of indirection and more moving parts before any benefit appears.
  • Overkill when there is only one implementation and no sign of a second.
  • The split raises conceptual overhead — readers must follow a delegation hop.
  • The seam has to be designed early; retrofitting a bridge onto a fused class is costly.

When to use it

Reach for it when

  • A type varies along two orthogonal dimensions that each grow on their own.
  • You want to switch implementations at runtime rather than choosing one at build time.
  • You need to hide a platform or backend behind a stable, published interface.

Avoid when

  • There is only one implementation and no realistic prospect of a second.
  • The two dimensions are not actually independent — they change together.
  • A plain strategy object or simple composition already covers the variation.

Code sketch

TypeScript — remote and device across the bridge 
// Implementation side — the "how"
interface Device {
  setVolume(pct: number): void;
  power(on: boolean): void;
}

class Tv implements Device {
  setVolume(pct: number) { /* drive TV hardware */ }
  power(on: boolean) { /* ... */ }
}
class Radio implements Device {
  setVolume(pct: number) { /* drive radio hardware */ }
  power(on: boolean) { /* ... */ }
}

// Abstraction side — the "what", holding the bridge
class RemoteControl {
  constructor(protected device: Device) {}
  togglePower() { this.device.power(true); }
  volumeUp()    { this.device.setVolume(10); }
}

class AdvancedRemote extends RemoteControl {
  mute() { this.device.setVolume(0); }   // extends abstraction, any Device works
}

new AdvancedRemote(new Tv()).mute();      // mix and match freely
new AdvancedRemote(new Radio()).volumeUp();

In the wild

JDBCApplication code writes against the fixed java.sql interfaces while a per-vendor driver supplies the implementation, so databases and query code vary independently.
