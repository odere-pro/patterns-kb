Decorator

Wraps an object in another that shares its interface — layering new behavior on at runtime, one transparent shell at a time, without touching the original class or the code that already uses it.

GoF Structural

What it is

A decorator is an object that wraps another object of the same interface, forwards calls to it, and adds behavior before or after. Because the wrapper implements the same interface as the thing it wraps, callers can't tell the two apart — they hold a Component, unaware whether it's a plain component or one dressed in several layers.

The problem it resolves is adding responsibilities to individual objects rather than to a whole class, and adding them dynamically. Subclassing is static: behavior is fixed at compile time, and every optional feature you add by inheritance multiplies the class count — a separate subclass for each combination of features. Decorator sidesteps that explosion by letting you compose features at runtime, stacking one small wrapper on another.

Each decorator is a thin shell responsible for a single concern — logging, buffering, compression, access checks — so behavior is built up from composable parts instead of baked into a monolithic class hierarchy.

How it works

Variations

Transparent (interface-preserving)
The classic form: the wrapper exposes exactly the wrapped interface, so it stays substitutable and stackable and callers never change.
Function / higher-order decorator
Wrap a function in another function instead of an object — memoize, throttle, retry, or time a call. Same idea, no class hierarchy needed.
Language-level decorators
Python @decorator syntax and TypeScript/Java annotations augment a declaration at definition time. Related in spirit, but they rewrite the target rather than wrap a live instance.
Stackable, order-sensitive
Several wrappers compose into a pipeline where order is significant — compress-then-encrypt behaves differently from encrypt-then-compress.

Trade-offs

Pros

  • Adds behavior at runtime without editing the wrapped class or its other users.
  • Composes features by stacking small wrappers instead of a class per combination.
  • Honours open/closed and single responsibility — each decorator does one thing.
  • Fully substitutable: callers keep depending on the interface, not the concrete shell.

Cons

  • A deep stack of thin wrappers is hard to trace — one call passes through many layers.
  • The order in which decorators are applied matters and is easy to get wrong.
  • Object identity breaks: a wrapped object is not the original, so === and type checks fail.
  • Many tiny classes, and the base decorator must forward every method faithfully.

When to use it

Reach for it when

  • You need to add responsibilities to individual objects, not to the whole class.
  • Optional features combine many ways and you want to pick the mix at runtime.
  • Subclassing would explode into a separate class for every feature combination.

Avoid when

  • The interface is large — hand-forwarding every method through each wrapper is brittle.
  • Downstream code needs the object's concrete type or identity.
  • A single flag or a strategy would express the variation more simply.

Code sketch

TypeScript — a transparent, stackable decorator 
interface DataSource {
  read(): string;
  write(data: string): void;
}

class FileSource implements DataSource {
  private contents = "";
  read() { return this.contents; }
  write(data: string) { this.contents = data; }
}

// Base decorator: same interface, forwards to the wrapped source.
abstract class SourceDecorator implements DataSource {
  constructor(protected readonly inner: DataSource) {}
  read() { return this.inner.read(); }
  write(data: string) { this.inner.write(data); }
}

class Encrypted extends SourceDecorator {
  read() { return decrypt(this.inner.read()); }        // add behavior on the way out
  write(data: string) { this.inner.write(encrypt(data)); } // and on the way in
}

// Stack it: encryption over a raw file, transparent to callers.
const source: DataSource = new Encrypted(new FileSource());
source.write("secret");   // stored encrypted, read back plain
