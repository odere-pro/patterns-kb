Value Object

Defined only by its attributes, with no identity of its own — two instances holding the same data are simply the same value, immutable, freely comparable, and safe to share without a second thought.

DDD Lens

What it is

A value object models a concept whose full meaning is its attributes — a sum of money, a date range, a coordinate, an email address. It carries no identifier field, no lifecycle, and no notion of "the same one I had yesterday." Two value objects with equal attributes are equal, by definition, no matter where each was constructed. Once built, a value object never changes; any operation that looks like a mutation instead returns a new instance.

The force it resolves is the identity question that Domain-Driven Design asks of every concept: does this thing have a continuous existence I must track through change, or is it fully described by what it currently holds? A customer is tracked — the same customer after a name change is still that customer. Five dollars is not tracked — any five dollars is interchangeable with any other. Conflating the two, by giving every concept an id and a mutable setter, is primitive obsession and identity bloat: validation logic scatters across the codebase, and objects that should be freely shared instead need defensive copies to guard against surprise mutation.

Value objects push validation and behavior to sit with the data. A Money class that owns its currency and rejects negative cents makes an invalid amount unrepresentable, rather than trusting every call site to check it. Immutability follows for free: with no identity to preserve, there is nothing wrong with handing the same instance to ten callers, caching it, or comparing it structurally instead of by reference.

How it works

Variations

Immutability
Fields are set once at construction and never reassigned; every "change" produces a new instance instead. This is less an optional variant than the core discipline that makes the rest of the pattern safe.
Self-validating value
The constructor enforces its own invariants — a Money that rejects negative cents, an Email that checks its format — so an invalid instance simply cannot exist.
Value object with behavior
Operations like add(), overlaps(), or format() live on the value itself and return new instances, keeping domain logic beside the data it acts on.
Structural (composite) value
Built from other value objects — an Address composed of Street, City, and PostalCode — with equality defined recursively across every part.

Trade-offs

Pros

  • Equality by value eliminates whole classes of aliasing bugs — equal data means interchangeable.
  • Immutability makes instances safe to share, cache, and pass across threads without defensive copies.
  • Validates its own invariants at construction, so an invalid instance can never exist.
  • Keeps domain logic — formatting, arithmetic, comparison — next to the data instead of scattered in services.

Cons

  • Every "change" allocates a new instance — real churn and GC pressure at scale if overused.
  • Equality and hashing must compare every field; forgetting one silently reintroduces identity bugs.
  • Persistence layers built around row identity often force embedding or serialization workarounds.
  • Easy to under-use — a thin primitive wrapper feels like ceremony, so primitive obsession creeps back in.

When to use it

Reach for it when

  • A concept is fully described by its attributes, and no two with the same attributes should be told apart — money, a date range, a coordinate.
  • You want equality, comparison, and hashing based on content rather than reference.
  • You're validating the same primitive — an email, a percentage, a currency amount — in more than one place.

Avoid when

  • The concept has a continuous lifecycle that must be tracked even as its attributes change — that's an Entity.
  • Persistence or a client needs to reference this exact instance independent of its current data.

Code sketch

TypeScript — an immutable Money value 
class Money {
  private constructor(
    readonly amountCents: number,
    readonly currency: string,
  ) {
    if (!Number.isInteger(amountCents)) {
      throw new Error("amount must be whole cents");
    }
  }

  static of(amountCents: number, currency: string): Money {
    return new Money(amountCents, currency);
  }

  add(other: Money): Money {
    if (other.currency !== this.currency) throw new Error("currency mismatch");
    return new Money(this.amountCents + other.amountCents, this.currency); // new instance
  }

  equals(other: Money): boolean {
    return this.amountCents === other.amountCents && this.currency === other.currency;
  }
}

const a = Money.of(500, "USD");
const b = Money.of(500, "USD");
a.equals(b); // true  — same value, no shared identity
a === b;     // false — different objects, and that's fine

In the wild

java.timeLocalDate, Duration and Instant are documented value-based classes: immutable, self-validating, and compared by their components rather than by reference. Their Javadoc warns against identity-sensitive operations such as == comparison or synchronizing on an instance, since implementations may cache or intern values.
Java recordsA language-level value aggregate, final since Java 16: the compiler makes the components final and derives equals, hashCode and toString from them, giving component-wise value equality without hand-written boilerplate.
