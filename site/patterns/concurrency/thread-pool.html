Lens · Concurrency

Thread Pool

Reuses a fixed set of worker threads to run a stream of tasks, so the cost of spinning up and tearing down a thread is paid once instead of on every request — and the number of threads running at once stays bounded no matter how much work arrives.

Concurrency Lens

What it is

A thread pool keeps a fixed (or bounded) collection of worker threads alive and a queue of pending tasks in front of them. When work arrives it's placed on the queue; whenever a worker is idle it pulls the next task off, runs it to completion, and loops back for another — it never exits between tasks. Callers get a handle back (often a future or promise) and move on without waiting for the thread to be created.

The force it resolves is the cost and risk of thread creation. Spinning up an OS thread means allocating a stack, registering it with the scheduler, and paying a real, measurable cost — repeating that for every incoming request ties your concurrency directly to arrival rate, not to what the hardware can actually run in parallel. Under load that means unbounded thread counts, memory pressure from thousands of stacks, and context-switch thrashing that makes everything slower, not faster.

The pool decouples submission from execution. Pool size becomes the one dial that governs real parallelism, and the queue absorbs bursts by making callers wait their turn instead of translating every burst directly into more live threads.

How it works

Variations

Fixed-size pool
N threads created up front and reused for the life of the process — the simplest and most predictable default, and the one most runtimes ship as their standard executor.
Cached / elastic pool
Grows toward a cap as demand rises and retires idle threads after a timeout, trading a hard resource ceiling for lower latency under bursty, uneven load.
Work-stealing pool
Each worker owns its own task deque and steals from a busy neighbor's when its own is empty, instead of every thread contending on one shared queue — how ForkJoinPool-style runtimes scale many small, unevenly sized tasks.
Bulkhead pools
One dedicated pool per tenant, endpoint, or workload type instead of a single shared pool, so a caller that saturates its own pool can't starve every other caller's threads too.

Trade-offs

Pros

  • Amortizes thread creation cost — threads are built once and reused across many tasks.
  • Bounds live concurrency to a known number, protecting memory and the scheduler.
  • The queue absorbs bursts instead of turning every spike into more live threads.
  • One number — pool size — is a single, tunable knob for real parallelism.

Cons

  • Sizing is genuinely hard: too few threads underuses the hardware, too many causes context-switch thrashing.
  • One long-running or blocking task can occupy a worker indefinitely and starve every other queued task.
  • An unbounded queue hides backpressure — tasks pile up in memory instead of the caller ever seeing "busy."
  • Tasks share threads, so leftover thread-local state or an uncaught exception in one task can bleed into the next.

When to use it

Reach for it when

  • You have a stream of many short, similar units of work — requests, jobs, callbacks — to run concurrently.
  • Creating a thread per task would be measurably expensive relative to the task itself.
  • You need a hard, known ceiling on how many threads can be running at once.

Avoid when

  • Tasks block indefinitely or run for a very long time — they'll occupy a worker and starve the rest of the queue.
  • The runtime already gives you a cheaper concurrency unit — an event loop, a coroutine, or a lightweight actor — where dedicating an OS thread per unit of work is overkill.
  • The workload is purely CPU-bound and what you actually want is one thread per core, not a generically sized pool.

Code sketch

TypeScript — a minimal fixed-size pool 
type Task<T> = () => Promise<T>;

class ThreadPool {
  private queue: Array<() => void> = [];
  private idle: number;

  constructor(private readonly size = 4) {
    this.idle = size; // all workers start idle
  }

  submit<T>(task: Task<T>): Promise<T> {
    return new Promise((resolve, reject) => {
      this.queue.push(() => {
        this.idle--;
        task()
          .then(resolve, reject)
          .finally(() => { this.idle++; this.drain(); });
      });
      this.drain();
    });
  }

  private drain() {
    while (this.idle > 0 && this.queue.length > 0) {
      this.queue.shift()!(); // hand the next task to an idle worker
    }
  }
}

const pool = new ThreadPool(4);
const results = await Promise.all(items.map((i) => pool.submit(() => process(i))));
