Lens · Concurrency

Reactor

A single event loop watches every open socket, timer, and signal at once, and the instant one is ready it dispatches straight to the handler that registered for it — no thread parked per connection, no one blocked waiting.

Concurrency Lens

What it is

A reactor is a single-threaded event loop that owns one job: ask the OS which of many registered event sources is ready — a socket that can be read, a timer that has fired, a signal that arrived — and dispatch each ready one to the handler that registered interest in it. The asking is done through a demultiplexer (select, poll, epoll, kqueue); the loop blocks there, wakes with a list of ready descriptors, and calls each handler in turn, synchronously, on the loop's own thread.

The force it resolves is the cost of thread-per-connection concurrency. Give every connection its own OS thread and each idle one still pins a stack and a scheduler entry — thousands of connections mean thousands of threads mostly doing nothing but costing context-switches and memory. Reactor decouples "wait for something to happen" from "do work when it happens": one thread (or a handful, one per core) can watch tens of thousands of descriptors for the price of a single blocking call, and only pays for CPU when there is real work to dispatch.

The catch is exactly what makes it cheap: there is one loop. A handler that blocks — a slow disk read, a synchronous DNS lookup, a tight CPU loop — stalls every other connection sharing that thread. The pattern trades away preemption; it demands that handlers give control back quickly.

Doug Schmidt formalized it in the ACE framework in the mid-1990s, and it now sits underneath most high-concurrency network servers: nginx's worker processes, Redis's single command-processing thread, and Node.js's event loop over libuv are all reactors.

How it works

Variations

Single-threaded reactor
One loop runs both the demultiplexing and every handler. Simplest, no locking anywhere, but a slow handler stalls all other connections.
Worker-pool reactor
The loop only demultiplexes and dispatches; the actual handler work runs on a bounded thread or task pool, so one slow handler no longer blocks ingestion of new events.
Multi-reactor (thread-per-core)
Run N independent loops, each pinned to a core and owning its own slice of connections, sharing nothing — the shape behind nginx workers and modern thread-per-core servers.
Reactor vs. Proactor
Reactor notifies "this descriptor is ready, go read it yourself"; Proactor notifies "your read already completed, here's the data" — the OS or an async layer performs the I/O, as in Windows IOCP.

Trade-offs

Pros

  • No thread-per-connection overhead — one loop scales to tens of thousands of idle sockets.
  • No locking inside the loop: handlers on the same thread never race each other.
  • Low, predictable memory footprint per connection compared to a parked thread and stack.
  • Dispatch order within the loop is explicit and easy to reason about.

Cons

  • A single blocking or CPU-heavy handler stalls every other event on that loop.
  • One loop uses one core — real parallelism needs multiple reactor instances.
  • Callback-driven control flow is harder to step through than straight-line, blocking code.
  • Every handler must be written non-blocking, which pushes async style through the whole call chain.

When to use it

Reach for it when

  • You're serving many concurrent, mostly-idle I/O connections — sockets, pipes, timers.
  • Thread-per-connection overhead (memory, context switches) is the actual bottleneck.
  • Work is I/O-bound and can be expressed as short, non-blocking handlers.

Avoid when

  • The workload is CPU-bound — a reactor's single loop just serializes the crunching.
  • Your runtime already gives you cheap, preemptible concurrency (lightweight threads, coroutines) — reintroducing an explicit loop adds complexity for no gain.
  • You depend on libraries that only offer blocking calls and can't be made async or offloaded to a pool.

Code sketch

TypeScript — a minimal single-threaded reactor 
type Handler = (fd: number) => void;

class Reactor {
  private handlers = new Map<number, Handler>();

  register(fd: number, handler: Handler): void {
    this.handlers.set(fd, handler);
  }

  unregister(fd: number): void {
    this.handlers.delete(fd);
  }

  // The loop: block until the OS says something is ready,
  // then dispatch each ready fd to its handler. A handler
  // must never block — that stalls every other connection.
  run(): void {
    while (true) {
      const ready = demultiplex([...this.handlers.keys()]); // epoll/kqueue
      for (const fd of ready) {
        this.handlers.get(fd)?.(fd);
      }
    }
  }
}

In the wild

nginxEach worker process runs its own epoll/kqueue event loop over thousands of connections, which is why it serves idle keep-alives so cheaply.
Node.js / libuvThe event loop demultiplexes socket, timer, and completion events and dispatches each to its JavaScript callback on a single thread.
RedisA single-threaded loop processes every command in turn, which is exactly why commands are atomic and why one slow command stalls the server.
