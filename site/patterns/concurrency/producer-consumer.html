Producer-Consumer

Decouples generating work from processing it — producers append items to a shared, bounded buffer and consumers drain it independently, so each side runs at its own pace without ever calling the other directly.

Concurrency Lens

What it is

In the producer-consumer shape, one or more producers generate items and place them on a shared buffer; one or more consumers take items off that same buffer and process them. Neither side calls the other directly — the buffer is the entire interface between them, and it's usually bounded, so it can hold only so many items before a producer must wait.

The force it resolves is a rate mismatch. Producers and consumers rarely run at the same speed: a producer might burst (a batch upload, a flood of sensor readings), while a consumer does something slower per item (a database write, a network call). Without a buffer between them, you're left with two bad options — couple them synchronously so the producer blocks on every single item, or let unconsumed work pile up somewhere uncontrolled. The buffer absorbs the mismatch, and its fullness becomes the signal that tells a producer to slow down and a consumer that there's nothing left to do.

Correctly coordinating access to that shared buffer is the whole exercise: producers must block (or be told to wait) when it's full, consumers must block when it's empty, and both need mutual exclusion around the buffer itself so two writers or a writer and a reader don't corrupt it. Get that wrong and you get lost updates, busy-waiting that pegs a CPU, or a deadlock where everyone is asleep.

It's tagged a lens here because it isn't one library or one API — it's the basic shape underneath a thread pool's work queue, an actor's mailbox, an OS pipe, and a streaming pipeline's internal channel. Once you see it, you recognize it inside far more specific-looking patterns.

How it works

Variations

Bounded vs. unbounded buffer
A bounded buffer caps memory use and forces a policy for what happens when it fills; an unbounded one defers that decision until the process runs out of memory.
Multiple producers, multiple consumers
Any number of producers and consumers can share one buffer as long as access is synchronized — the classic bounded-buffer problem scales cleanly in both directions.
Backpressure
When the buffer is full, block the producer, drop the newest or oldest item, or reject outright — the full-policy is the pattern's central design decision.
Message Queue
Swap the in-process buffer for a durable, out-of-process queue and the same shape survives a crash and spans separate services.

Trade-offs

Pros

  • Decouples producer and consumer rates — each runs at its own pace.
  • Smooths bursts, absorbing spikes instead of dropping or blocking work immediately.
  • Scales each side independently: add consumers to drain faster, add producers without touching consumers.
  • The buffer's fullness is a built-in backpressure signal, not something bolted on afterward.

Cons

  • The buffer is shared mutable state — wrong synchronization causes races, lost wakeups, or deadlock.
  • A bounded buffer forces a full-policy decision (block, drop, reject) that's easy to get wrong or skip entirely.
  • An unbounded one hides unchecked memory growth until the process falls over.
  • Adds a moving part — buffer size, wakeups, queue depth — that must be sized and monitored.

When to use it

Reach for it when

  • Producers and consumers naturally run at different, variable rates.
  • Work should keep being generated even while processing catches up, or vice versa.
  • You need to smooth bursty input before it overwhelms downstream processing.

Avoid when

  • Work is a one-off, synchronous call — invoking the consumer directly is simpler and clearer.
  • Strict per-item ordering across producers is required and the buffer can't preserve it.
  • Producer and consumer speeds are already matched, so a buffer only adds cost.

Code sketch

TypeScript — an async bounded queue 
class BoundedQueue<T> {
  private buffer: T[] = [];
  private consumers: Array<(item: T) => void> = [];
  private producers: Array<() => void> = [];

  constructor(private readonly capacity: number) {}

  async put(item: T): Promise<void> {
    while (this.buffer.length >= this.capacity) {
      await new Promise<void>((resolve) => this.producers.push(resolve));
    }
    const waiting = this.consumers.shift();
    if (waiting) waiting(item);          // hand off directly
    else this.buffer.push(item);
  }

  async take(): Promise<T> {
    if (this.buffer.length > 0) {
      const item = this.buffer.shift()!;
      this.producers.shift()?.();         // wake one blocked producer
      return item;
    }
    return new Promise<T>((resolve) => this.consumers.push(resolve));
  }
}

In the wild

Go channelsA buffered channel make(chan T, n) is a bounded queue: sends block when full, receives block when empty, close signals no more values, and range drains until closed. The classic bounded-buffer hand-off built into the language.
java.util.concurrent.BlockingQueueArrayBlockingQueue is bounded and array-backed with a single lock; LinkedBlockingQueue uses separate put and take locks for higher throughput and is unbounded unless a capacity is given. put/take block, offer/poll take timeouts.
Unix pipesA shell pipeline gives each stage a fixed-size kernel buffer (64 KB on Linux by default, resizable via fcntl F_SETPIPE_SZ) that stalls the writer when full and the reader when empty; writes up to PIPE_BUF (4096 bytes) are atomic.

In production

Tuning knobs

  • Queue capacity — The bound on the shared buffer. It sets how far producers may run ahead before they block and caps the memory the backlog can consume.
  • Consumer pool size — Number of consumer threads or workers draining the queue. Too few and the queue backs up; too many and they contend or sit idle for work.
  • Full-queue policy — What an insert does when the buffer is full: block (ArrayBlockingQueue.put), time out (offer with a timeout), or reject the item.
  • Batch / prefetch size — How many items a consumer takes per wakeup. Batching amortizes lock and wakeup cost against added latency.

Signals to watch

  • Queue depth — Sustained near-full means consumers are the bottleneck; sustained near-empty means producers are.
  • Producer block time — Time producers spend blocked on a full queue.
  • Throughput vs consumer idle time — Items processed per second against the time consumers wait on an empty queue.

Failure modes under load

  • Unbounded queue to OOM — An unbounded buffer, such as a LinkedBlockingQueue with no capacity, lets a fast producer grow it until the process runs out of memory.
  • Deadlock on shutdown — Consumers blocked on take() never learn that producers are done unless a shutdown or poison-pill protocol wakes them.
  • Consumer starvation — If every consumer blocks on a downstream dependency while producers keep filling the queue, the whole pipeline stalls.

Readiness checklist

  • Bound the queue; an unbounded buffer only defers the failure to an OOM.
  • Size the consumer pool to the consumer service rate, not the producer rate.
  • Define an explicit shutdown or drain protocol (poison pills or channel close) so blocked consumers exit.
  • Instrument queue depth to see which side is the bottleneck.
