Semaphore

A counter of N permits that bounds concurrent access to a resource: every acquire takes a permit and blocks when none are left, every release returns one and wakes a waiting thread.

Concurrency

What it is

A semaphore is a counter with two operations and one rule. It is created holding N permits. acquire() takes a permit before a thread proceeds; if the count is already zero it blocks the caller and parks it until a permit appears. release() returns a permit and wakes one waiting thread. Everything else about the pattern follows from those two moves.

The force it resolves is scarcity, not correctness: there simply is not enough of some resource for everyone who wants it at the same instant. A downstream API tolerates ten concurrent callers before it sheds load; a box has 500 MB of buffer memory to share across worker threads; a pool holds ten database connections against a thousand requests a second. In each case the danger is overrun rather than a data race — issue more than the resource can absorb and it degrades or fails for everyone. The semaphore makes that ceiling explicit and enforces the wait: at the limit, the next caller blocks instead of piling on.

Two shapes recur. In the first, one permit is one slot — N permits cap how many callers may be inside a section at once, whatever the total thread count. In the second, a permit is a unit of a divisible budget — a caller acquires as many permits as it will consume, one per megabyte say, so the sum of all outstanding work can never exceed the total. A semaphore with a single permit degenerates into a plain lock, which is why the two ideas are so often conflated; but where a mutex only asks "is anyone inside?", a semaphore counts "how many, and is there room for one more?" That counting is the whole point. Because a blocked acquire is undone only by a matching release, that release belongs in a finally block or a with/using scope — miss it on an exception and the permit is gone for good, so the ceiling ratchets down until the semaphore is dead at zero.

How it works

Variations

Counting semaphore
The general form: N permits let up to N threads hold a resource at once. This is the default you reach for when the cap is greater than one — throttling concurrent operations without dedicating a worker per slot.
Binary semaphore (N = 1)
A single permit makes it behave like a lock. The difference from a mutex is ownership: a semaphore records no holder, so any thread may release it — useful for signalling between threads (one waits, another posts), but wrong as a mutual-exclusion lock, where you want the acquirer to be the only releaser.
Permits as a budget
Instead of one-permit-per-caller, one permit represents one unit of a divisible resource — a megabyte, a byte of bandwidth, a token. A caller acquires as many permits as it will use and releases exactly that many, so aggregate consumption is bounded even though each operation's size varies.
Timed / try-acquire
Rather than block forever, acquire with a timeout (or a non-blocking attempt) and fail when no permit arrives in time. Essential on request paths: it converts an indefinite hang into a fast, visible rejection the caller can turn into a 503 or a retry.
Fair vs. barging
When a permit frees, does the longest-waiting thread get it (FIFO fairness) or may a newly arriving thread barge in ahead of the queue? Barging gives higher throughput; strict fairness prevents a waiter from being starved indefinitely under heavy contention, at some cost in throughput.

Trade-offs

Pros

  • One explicit number caps concurrency or consumption — a single dial for how much of a scarce resource is in use at once.
  • Bounds in-flight work independent of thread count — any thread may hold a permit, so you throttle load without dedicating a worker per unit.
  • Blocking is OS-efficient — a waiter parks and is woken on release, with no spin-polling or sleep-and-retry.
  • Permits-as-a-budget generalizes past counts — the same primitive throttles megabytes of memory or bandwidth, not just how many operations run.

Cons

  • A leaked permit never comes back — miss the release on an exception and the ceiling ratchets down until the semaphore deadlocks at zero.
  • A bare acquire blocks forever — on a request path a starved caller waits indefinitely, so users see timeouts while monitoring shows no errors; a timed acquire is needed to fail fast.
  • It grants permission, not objects — a semaphore says "there is room" but hands out no connection or buffer; when callers need the actual resource you still need a pool behind it.
  • No ownership or reentrancy — unlike a mutex, any thread can release a permit it never took, so a double-release or a wrong initial count silently lifts the real ceiling above the intended one.
  • Counting is manual for variable-size budgets — the caller must acquire and release exactly the right number of permits, and an off-by-one in that arithmetic is a slow resource leak.

When to use it

Reach for it when

  • You need to let at most N callers into a section at once — more than one, but still bounded — such as capping concurrent calls to a downstream that sheds load past a threshold.
  • You're bounding total consumption of a divisible resource — megabytes of buffer, in-flight bytes, disk I/O — where each operation uses a variable amount.
  • You want to throttle in-flight work without a fixed worker pool — any thread can hold a permit, so the cap rides on top of whatever concurrency you already have.

Avoid when

  • The limit is one and you want plain mutual exclusion — a lock or monitor states that intent more clearly and gives you ownership and reentrancy.
  • Callers need the actual resource object, not just permission — a connection, a GPU handle — where a blocking-queue resource pool that dispenses the objects fits better.
  • You need to wait until a specific state holds, not merely for a free slot — that is condition-variable territory, handled by a monitor object.

Code sketch

TypeScript — a counting semaphore gating outbound calls 
class Semaphore {
  private permits: number;
  private readonly waiters: Array<() => void> = [];

  constructor(permits: number) {
    this.permits = permits; // N slots free up front
  }

  async acquire(): Promise<void> {
    if (this.permits > 0) {
      this.permits--;               // a slot was free — take it
      return;
    }
    await new Promise<void>((resolve) => this.waiters.push(resolve)); // park
  }

  release(): void {
    const next = this.waiters.shift();
    if (next) next();               // hand the permit straight to a waiter
    else this.permits++;            // no one waiting — return it to the pool
  }
}

// Cap outbound calls to a fragile service at 5 concurrent.
const gate = new Semaphore(5);

async function callDownstream(req: Request): Promise<Response> {
  await gate.acquire();
  try {
    return await fetch(req);
  } finally {
    gate.release();                 // always release, even on throw
  }
}

In the wild

Java java.util.concurrent.SemaphoreThe canonical counting semaphore on the JVM: constructed with a permit count, acquire()/release() move the counter, tryAcquire(timeout, unit) fails instead of blocking forever, and a boolean fairness argument switches wake order from barging to FIFO. acquire(n)/release(n) take several permits at once for the budget idiom, and any thread may release.
Python threading.Semaphore / asyncio.SemaphoreBound concurrent work in either threaded or async code; both act as context managers (with / async with) so the permit is released even on exception. BoundedSemaphore raises if released more times than acquired, catching the double-release miscount that silently raises the ceiling.
Go buffered channel as a counting semaphoreA buffered channel of capacity N is the idiomatic counting semaphore — send an empty struct to take a slot, receive to release; a full channel blocks the sender until a slot frees. The golang.org/x/sync/semaphore package adds a weighted variant whose Acquire(ctx, n) requests several units at once for the budget idiom.
