Render Props

A component owns some state or behavior and calls a function you hand it to produce the output — inverting control over rendering so one piece of logic can drive many different UIs.

Frontend

What it is

Sometimes the same stateful behavior — tracking the mouse, fetching data, holding an on/off flag — needs to power components that look nothing alike. The awkward answer is to hoist that behavior into a base class and inherit from it, coupling unrelated views through a rigid hierarchy.

Render props inverts the arrangement. One component owns the behavior and computes the state, but it does not decide what to draw. Instead it accepts a function — the render prop — and calls it with the current state, delegating the markup entirely to the caller. The behavior lives in one place; the rendering is supplied fresh at every use site. Because the function is just a value, the same logic component can drive a list in one screen, a chart in another, and a debug readout in a third, with no shared base class between them.

How it works

Variations

Render prop / function-as-child
A prop whose value is a function that receives the state and returns markup. When that prop is children, the call site reads as nested JSX — the "function as a child" idiom — but it is the same mechanism.
Higher-Order Component (HOC)
Instead of calling a function you pass in, a function wraps your component and injects the behavior as extra props. The logic reuse is the same; the wiring happens at composition time rather than at render time.
Hooks
The modern replacement. A custom hook extracts the same stateful logic into a plain function call inside the component, with no wrapper component and no nesting. Preferred for most new code.

Trade-offs

Pros

  • Reuses stateful logic across unrelated views without a class-inheritance hierarchy.
  • Flexible composition — the same behavior component drives any markup the caller supplies.
  • Keeps behavior and view decoupled, so each can change without disturbing the other.

Cons

  • Combining several render props nests callbacks into "wrapper hell" that is hard to read.
  • Indirection can obscure where a given value actually comes from.
  • Hooks now supersede it for most new code, expressing the same reuse without the nesting.

When to use it

Reach for it when

  • You must share cross-cutting stateful behavior — mouse position, data-fetching, a toggle — across views that render completely differently.
  • The reused logic must stay independent of any particular markup.

Avoid when

  • A Hook expresses the same reuse more simply — the case for most new code.
  • Several render props would nest deeply into unreadable wrapper hell.

Code sketch

TypeScript — a Toggle that owns the flag and lets the caller render it 
import { useState, type ReactNode } from "react";

type ToggleProps = {
  // The render prop: Toggle owns the state, the caller owns the markup.
  children: (on: boolean, toggle: () => void) => ReactNode;
};

function Toggle({ children }: ToggleProps) {
  const [on, setOn] = useState(false);
  const toggle = () => setOn((v) => !v);
  return <>{children(on, toggle)}</>; // hand state to the caller's function
}

// One behavior, two completely different views:
function Settings() {
  return (
    <Toggle>
      {(on, toggle) => (
        <button onClick={toggle}>
          Notifications: {on ? "on" : "off"}
        </button>
      )}
    </Toggle>
  );
}
