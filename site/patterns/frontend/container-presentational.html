Container / Presentational

A container component owns data-fetching and state; a presentational component below it only takes props and renders — so rendering stays pure, reusable, and easy to test.

Frontend

What it is

A component that both fetches its own data and renders the result is hard to test and hard to reuse: every test has to stub a network call, and the markup is welded to one particular data source. This pattern splits those two concerns into a pair of components. A container knows where the data comes from — it talks to a store, a cache, or an API, holds the loading and error state, and owns the side effects. A presentational component knows how it looks — it receives plain props from the container and renders them, nothing more.

The force it resolves is the tangle of I/O and view logic in one place. By pushing all the fetching up into the container and keeping the view a pure function of its props, the view becomes trivial to test in isolation and free to be reused with any source of data.

How it works

Variations

Hooks-era split
Instead of a wrapping component, a custom hook holds the "container" logic — the fetch, the state, the effects — and returns plain data; a presentational component calls the hook and renders what it gets. Same boundary, less nesting. This is the idiomatic modern form.
Route / page-level container
The container sits at a route or page boundary, loading everything the page needs and distributing props to several presentational children below it. Keeps data-loading at the edges of a feature.
Higher-Order-Component container
A HOC wraps a presentational component and injects the fetched data as props (the classic connect() shape). Older style, but still the mechanism behind many data-binding wrappers.

Trade-offs

Pros

  • Pure views are trivially testable — pass props, assert on output, no network stubs.
  • Presentational components are reusable with any data source, not welded to one.
  • A clear data-flow boundary: fetching lives above, rendering lives below.

Cons

  • More files and a layer of indirection to trace through.
  • Overkill for trivial components that neither fetch nor hold state.
  • The split is a convention, not enforced by the framework — discipline keeps it honest.

When to use it

Reach for it when

  • A component mixes fetching and rendering and has become hard to test.
  • You want one view reused with different data sources or in different contexts.
  • You need a clean seam to render the same UI from mock data in tests or stories.

Avoid when

  • The component is trivial — it neither fetches nor holds meaningful state.
  • A single hook already colocates the loading logic cleanly and nothing needs reuse.

Code sketch

TypeScript — a container fetches, a pure view renders 
interface User { id: string; name: string; email: string; }

// Presentational: a pure function of its props. No fetching, no state.
function UserView({ user }: { user: User }) {
  return (
    <article>
      <h2>{user.name}</h2>
      <p>{user.email}</p>
    </article>
  );
}

// Container: owns the data source and the loading/error state.
function UserContainer({ id }: { id: string }) {
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    fetchUser(id).then(setUser);
  }, [id]);

  if (!user) return <Spinner />;
  return <UserView user={user} />;   // hand plain props to the pure view
}
