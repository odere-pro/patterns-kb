Secure Logger

Captures the events an audit trail needs — who did what, when — while masking or stripping the passwords, tokens, and personal data that should never survive into a log file.

Security Lens

What it is

A secure logger is the single path every security-relevant event travels before it reaches a log sink. Instead of letting each call site format and write its own log line, code hands the logger a structured event — an actor, an action, a target, some metadata — and the logger applies a sensitivity policy, masks or redacts anything that shouldn't leave the process in the clear, and only then writes the record to an append-only, access-controlled store.

The force it resolves is the collision between two real requirements that pull in opposite directions. Auditors, incident responders, and regulators need a trustworthy record of who did what and when — logins, permission checks, admin actions, data access. But the objects flowing through a typical request already carry passwords, session tokens, card numbers, and PII, and a naive console.log(request) or a stack trace dump copies that data verbatim into a log aggregator that is usually far less protected than the system it came from. Plenty of real breaches start exactly there: a secret sitting in plaintext in Splunk, CloudWatch, or a support ticket's attached log excerpt.

Secure Logger is one of the classic J2EE-era security patterns, catalogued alongside Intercepting Validator and Single Access Point. Its job is narrow on purpose: centralize the one place where "does this field get written down" is decided, so that decision doesn't depend on every engineer remembering it at every call site.

How it works

Variations

Wire Tap
Intercept an existing message flow with a non-invasive tap and route a copy into the secure logger, instead of instrumenting every call site by hand.
Allowlist vs. denylist redaction
Denylist known-sensitive keys like password or token and mask anything matching, or allowlist only fields explicitly cleared for logging. Allowlists survive schema drift better but need active upkeep as fields are added.
Tokenization and hashing
Replace an identifier with a stable hash or token instead of blanking it outright, so an investigator can still correlate events for the same user without ever seeing the raw value.
Tamper-evident / hash-chained entries
Chain each entry's hash to the previous one, or sign it, so an after-the-fact edit or deletion of a log record is detectable — protecting integrity, not just confidentiality.

Trade-offs

Pros

  • Centralizes masking and redaction in one place instead of trusting every call site to remember it.
  • Produces an audit trail auditors and incident responders can actually trust for non-repudiation.
  • Makes it safe to ship logs to lower-trust systems — SIEMs, third-party aggregators, support tools.
  • Tamper-evident variants turn the log itself into forensic evidence, not just a debugging aid.

Cons

  • Adds latency and CPU to the logging hot path — scanning and redacting nested objects isn't free.
  • A denylist can't catch what it doesn't know about; a new field carrying a raw secret slips through silently.
  • Over-aggressive redaction can strip the very detail an investigation needed, trading audit value for false safety.
  • Only as strong as its adoption — one stray console.log bypasses it entirely.

When to use it

Reach for it when

  • You log security-relevant events — authentication, authorization decisions, admin actions, data access.
  • Requests or domain objects carry passwords, tokens, session IDs, or PII that must never land in a general-purpose log store.
  • Compliance requires a tamper-evident, access-controlled audit trail — PCI-DSS, HIPAA, SOC 2, GDPR.

Avoid when

  • Nothing sensitive ever flows through the log path — plain structured logging is enough and redaction is dead weight.
  • Logs are purely local debug output that never leaves the developer's machine.
  • You already write to a single, encrypted, access-controlled store — field-level redaction may be belt-and-braces.

Code sketch

TypeScript — a redacting audit logger 
interface AuditEvent {
  actor: string;
  action: string;
  target?: string;
  metadata?: Record<string, unknown>;
}

const SENSITIVE_KEYS = new Set(["password", "token", "ssn", "cardnumber"]);

function redact(value: unknown): unknown {
  if (Array.isArray(value)) return value.map(redact);
  if (value && typeof value === "object") {
    const out: Record<string, unknown> = {};
    for (const [k, v] of Object.entries(value)) {
      out[k] = SENSITIVE_KEYS.has(k.toLowerCase()) ? "[REDACTED]" : redact(v);
    }
    return out;
  }
  return value;
}

class SecureLogger {
  constructor(private readonly sink: (line: string) => void) {}

  audit(event: AuditEvent): void {
    const safe = { ...event, metadata: redact(event.metadata) };
    this.sink(JSON.stringify({ ts: new Date().toISOString(), ...safe }));
  }
}

const logger = new SecureLogger(line => auditStore.append(line));
logger.audit({ actor: "user:42", action: "login", metadata: { password: "hunter2" } });
// written: metadata.password becomes "[REDACTED]"
