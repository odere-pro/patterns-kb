Network · Routing & Scale

API Gateway

A single front door for a constellation of backend services — every request lands here first, gets authenticated, rate-limited, and routed to the right service, so clients never learn the internal topology behind it.

Distributed Routing & Scale

What it is

An API gateway sits between every external client and a set of backend services, and is the only thing those clients ever talk to directly. It accepts each incoming request, decides — by path, host, header, or version — which backend service should handle it, forwards the call, and hands the response back to the caller. Along the way it typically also absorbs the concerns every service would otherwise have to reimplement: authentication, rate limiting, request logging, response caching, and translating between the protocol a client speaks and whatever the backend speaks internally.

The forces it resolves are fan-out and duplication. In a system decomposed into many small services, a client that talks to each one directly has to know where every service lives, hold a connection to each, and repeat the same security and observability logic for every call it makes. That couples clients to an internal topology that keeps changing as services are split, merged, or rehomed, and it means an auth bug gets fixed — or missed — independently in a dozen places. The gateway collapses that fan-out into one seam: clients see a single stable surface, and cross-cutting policy lives in one deployable instead of copy-pasted across services.

It is not just a reverse proxy with a different name. A reverse proxy forwards bytes; a gateway makes routing decisions grounded in the request's meaning — which API version, which tenant, which feature flag — and layers policy on top of that. That extra intelligence is also its hazard: pushed too far, the gateway accretes business logic that belongs in the services it's meant to be routing to, not in the routing layer itself.

How it works

Variations

Gateway Routing
Pure request routing by path, host, or header, with no aggregation or reshaping — the thinnest form of the pattern.
Gateway Aggregation
Fans a single client request out to several backend calls and composes the results into one response, cutting round trips for slow or mobile clients.
Gateway Offloading
Moves shared infrastructure concerns — TLS termination, compression, caching, auth — out of every service and into the gateway itself.
Backend-for-Frontend
One gateway per client type — web, mobile, partner — instead of a single universal gateway trying to satisfy all of their conflicting needs at once.
Streaming Gateway
Carries a long-lived streaming response — server-sent events, chunked transfer, or a WebSocket upgrade — rather than buffering one payload and returning it. It shifts the gateway's job from request/response to holding an open flow, which forces different read timeouts, disables response buffering, and often needs connection affinity (see Sticky Session) so the stream isn't cut mid-flight by a rebalance.
Agent-Facing Gateway
Exposes backing capabilities as discoverable, self-describing tools an autonomous AI agent can enumerate and invoke at runtime, instead of fixed endpoints a human wired up ahead of time. The agent is just another client class — a Backend-for-Frontend for software that decides its own calls. Anthropic's Model Context Protocol (MCP) is the current real-world example of this tool-server shape.

Trade-offs

Pros

  • One place to enforce auth, rate limiting, and logging instead of duplicating it in every service.
  • Clients depend on a stable surface, not the internal service topology, which is free to change behind it.
  • Central point for protocol translation and API versioning as backends evolve independently.
  • Aggregation can cut chatty round trips for clients on slow or high-latency networks.

Cons

  • A single point of failure and a potential bottleneck unless it's made resilient and scaled itself.
  • Prone to becoming a dumping ground for business logic that belongs in the services behind it.
  • Adds a network hop and a layer of latency to every single request.
  • Yet another highly-available deployable to build and operate, and a shared release cadence every team routing through it now depends on.

When to use it

Reach for it when

  • Clients — web, mobile, partners — need one stable entry point across many backend services.
  • You need auth, rate limiting, or logging enforced consistently without repeating it per service.
  • Backend services are being split, merged, or migrated and clients shouldn't feel any of that churn.

Avoid when

  • There's a single backend service — there's no fan-out to front, just call it directly.
  • Client types have sharply different aggregation needs — one shared gateway becomes a tangle of conflicting requirements; split it with a Backend-for-Frontend per client instead.
  • The team can't yet run one more highly-available, horizontally-scaled service — an under-resourced gateway just becomes the new single point of failure.

Code sketch

TypeScript — a minimal routing gateway 
type Route = { prefix: string; target: string };

const routes: Route[] = [
  { prefix: "/orders", target: "http://orders-svc:3000" },
  { prefix: "/users", target: "http://users-svc:3000" },
  { prefix: "/inventory", target: "http://inventory-svc:3000" },
];

app.use(authenticate);              // once, at the edge
app.use(rateLimit({ windowMs: 60_000, max: 100 }));

app.use(async (req, res, next) => {
  const route = routes.find((r) => req.path.startsWith(r.prefix));
  if (!route) return res.status(404).json({ error: "no route" });

  try {
    const upstream = await fetch(route.target + req.path, {
      method: req.method,
      headers: req.headers as HeadersInit,
      body: req.method === "GET" ? undefined : req.body,
    });
    res.status(upstream.status).send(await upstream.text());
  } catch {
    next(new Error("upstream unavailable")); // let error middleware handle it
  }
});

In the wild

KongBuilt on NGINX and OpenResty; routing, authentication, and rate limiting are Lua plugins applied per route or service, with configuration stored in a database or declared in a YAML file for DB-less mode.
AWS API GatewayA managed edge that maps REST or HTTP routes onto Lambda functions, HTTP endpoints, or other AWS services, and handles authorizers, request throttling via usage plans, and per-stage response caching.
EnvoyThe C++ proxy used as the data plane in Istio and Envoy Gateway; it terminates external TLS, routes on path, header, or authority, and load-balances across endpoints with health checking and outlier detection.

In production

Tuning knobs

  • Rate-limit thresholds — Requests per window per client, route, or key, and the burst allowance before requests are rejected.
  • Upstream timeout and retry — How long the gateway waits on a backend and whether it retries, so one slow service does not hang the caller.
  • Upstream connection pool — Max concurrent connections the gateway holds to each backend, bounding in-flight requests per upstream.
  • Response cache TTL — How long cacheable responses are served from the gateway before revalidating with the backend.
  • Request and response body size limits — Caps on payload size so a large upload cannot exhaust gateway memory.

Signals to watch

  • Gateway-added latency — p99 of the gateway hop measured against backend service time.
  • Rate-limit rejection rate — 429 responses per route or client — shows whether limits are biting.
  • Upstream error and timeout rate — 5xx and timeouts per route, isolating which backend is failing behind the single surface.
  • Gateway resource and connection count — CPU, memory, and open connections on the gateway tier.

Failure modes under load

  • Single point of failure — If the gateway tier is down or undersized, every route behind it is unreachable.
  • Head-of-line blocking — One slow upstream ties up the gateway's connection pool or worker threads, starving unrelated routes unless per-upstream timeouts and isolation are set.
  • Auth dependency outage — If the gateway calls an external auth service per request and it is down, every request fails at the edge.
  • Bad config rollout — A broken route or plugin config propagates and breaks live traffic until it is rolled back.

Readiness checklist

  • Gateway runs as a horizontally scaled, load-balanced tier, not a single instance.
  • Per-upstream timeouts and circuit breaking so one failing backend cannot cascade.
  • Rate limits and request-body size limits configured.
  • TLS termination and the auth path tested end to end.
  • Route and plugin config validated and rolled out gradually, with a rollback path.
