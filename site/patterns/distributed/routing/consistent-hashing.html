Consistent Hashing

Maps nodes and keys onto the same ring so that adding, removing, or resizing a node remaps only a thin slice of the keyspace — never all of it.

Distributed Routing & Scale

What it is

Consistent hashing maps both nodes and keys onto the same fixed circular space — hash each node's identity and each key onto a ring of, say, 0 to 232-1, and assign a key to the first node found walking clockwise from its position. Because nodes and keys share one hash space, a node's arrival or departure only touches the keys between it and its neighbor on the ring; everything else stays exactly where it was.

The problem it resolves is the fragility of naive hashing. A scheme like hash(key) % N ties every key's location to the total node count N. Add or remove a single node and N changes, so nearly every key remaps to a different node in one step — a million-entry cache goes cold on a routine deploy, or a store re-shuffles almost all its data just to add one machine. Consistent hashing decouples placement from N: only the fraction of keys near the changed node ever moves.

A single ring position per physical node would still own an uneven, unpredictable arc, and losing that node dumps its whole arc onto one neighbor. Real implementations give each physical node many points on the ring — virtual nodes — so load smooths out across all neighbors and no single failure or join swings capacity too far in either direction.

How it works

Variations

Virtual nodes (vnodes)
Each physical node is hashed to dozens or hundreds of ring points instead of one, spreading its load across many neighbors and bounding the blast radius of any single node's churn.
Bounded-load consistent hashing
Caps any node at roughly (1 + ε) times the average load; requests that would push it past the cap fall through to the next node on the ring, trading a pure single-hash lookup for a fairness guarantee.
Rendezvous (highest random weight) hashing
Skips the ring entirely: compute a weight per node per key and take the max. No sorted structure to maintain, and membership changes remap just as few keys.
Jump consistent hash
A tiny, allocation-free function that maps a key to a bucket index in O(log n) time with no stored ring — at the cost of only supporting removal from the end of the bucket list, not from the middle.

Trade-offs

Pros

  • Adding or removing a node remaps only about 1/N of keys, not the whole keyspace.
  • Lets a cluster scale horizontally without a coordinated full re-shard.
  • Underpins distributed caches, DHTs, and sharded databases at large scale.
  • With virtual nodes, load stays even even as membership keeps changing.

Cons

  • One ring point per node gives uneven load; fixing it with vnodes adds memory and lookup cost.
  • Ring lookup is O(log n) against a sorted structure, slower than O(1) modulo hashing.
  • Every node needs a consistent view of ring membership — a stale view means two nodes disagree about who owns a key.
  • Doesn't fix hot-key skew: one very popular key still overloads whichever single node owns it.

When to use it

Reach for it when

  • Nodes come and go — scaling, failures, rolling deploys — and you need placement to survive that.
  • Minimizing data movement or cache invalidation on membership change matters.
  • You're building a distributed cache, a DHT, a sharded store, or a load balancer that should stick clients to backends.

Avoid when

  • The node count is fixed and essentially never changes — a static mod-N hash is simpler and needs no ring bookkeeping.
  • Strict, real-time load balancing matters more than minimal movement — pair with a bounded-load variant or a dedicated load balancer instead.
  • A small, centralized directory of key-to-node assignments is simpler than propagating ring membership everywhere.

Code sketch

TypeScript — a ring with virtual nodes 
class HashRing {
  private ring = new Map<number, string>(); // ring position -> node id
  private positions: number[] = [];
  private readonly vnodes = 100;

  addNode(node: string): void {
    for (let i = 0; i < this.vnodes; i++) {
      this.ring.set(hash(`${node}#${i}`), node);
    }
    this.positions = [...this.ring.keys()].sort((a, b) => a - b);
  }

  removeNode(node: string): void {
    for (let i = 0; i < this.vnodes; i++) {
      this.ring.delete(hash(`${node}#${i}`));
    }
    this.positions = [...this.ring.keys()].sort((a, b) => a - b);
  }

  getNode(key: string): string {
    const h = hash(key);
    const idx = this.positions.findIndex((pos) => pos >= h);
    const pos = idx === -1 ? this.positions[0] : this.positions[idx]; // wrap
    return this.ring.get(pos)!;
  }
}

In the wild

Apache CassandraPlaces partitions on a token ring, with each node holding many virtual nodes so churn spreads across neighbours.
Amazon DynamoThe 2007 paper that put the ring plus virtual nodes into mainstream practice, and the ancestor of Cassandra and Riak.
