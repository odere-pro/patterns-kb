Valet Key

A short-lived, narrowly scoped key lets a client reach a resource directly — one file, one queue, one row — without ever holding your master credentials or making your servers proxy the traffic.

Distributed Routing & Scale

What it is

A valet key hands a client a token or URL that grants direct, temporary access to one specific resource — a blob in storage, a queue, a single row — without ever exposing the resource's real credentials. The client presents the key straight to the resource; the server that issued it isn't in the data path at all, only in the decision of who gets a key and what it's good for.

The force it resolves is a bottleneck at the trust boundary. Routing every upload, download, or message through your own servers turns them into a proxy for bytes they add no value to — burning bandwidth and connections on plain forwarding. But letting clients talk to the resource directly usually means handing them your master credentials, which is far more power than the moment calls for.

The pattern splits the two concerns. A trusted party authenticates the request and mints a key that's narrowly scoped — one resource, one action, one short expiry — and hands it over. The resource itself then validates the key on each use without ever calling back to your server. It's named for the valet who's handed a key that starts the car and pops the trunk, but can't touch the glovebox or drive it off the lot.

Once minted, a key is a bearer credential: whoever holds it can use it until it expires or its uses run out. That trade — convenience and scale against a window of exposure — is what every variation below tunes.

How it works

Variations

Pre-signed URLs
A cloud storage vendor signs a URL with an embedded expiry and permission set — S3 pre-signed URLs and Azure SAS tokens are the canonical form.
Signed cookies
The key rides in a cookie instead of the URL, so it covers a whole path prefix (a CDN's private content tree) rather than one object.
Single-use vs. multi-use keys
A single-use key is invalidated after one redemption, closing the replay window entirely; a multi-use key trades that safety for fewer round trips.
Capability URLs
The URL itself is the only credential — no separate login step — so possessing the link is equivalent to holding the permission.

Trade-offs

Pros

  • Removes the app server from the data path, freeing it for control-plane work only.
  • Scales bandwidth- or storage-heavy operations without provisioning servers to carry them.
  • Scoped and time-limited, so a leaked key exposes only one resource, and only briefly.
  • The client never holds standing credentials to the resource itself.

Cons

  • A valid key generally can't be revoked before it expires unless the resource supports it.
  • Getting scope or expiry wrong quietly grants broader or longer access than intended.
  • Requires the resource layer to support token-based validation — not every store does.
  • Clock skew or an overlong expiry undermines the whole "temporary" guarantee.

When to use it

Reach for it when

  • Clients need direct, high-volume access to a resource — file upload or download — and proxying it through your servers is wasted work.
  • You want to offload bandwidth to a CDN or blob store without ever issuing standing credentials.
  • The access need is transient and well-defined — one object, one action, one short window.

Avoid when

  • The operation needs per-request business logic or auditing that only your server can enforce.
  • You can't safely bound the scope or lifetime within the resource's own access model.
  • Revocation before expiry is a hard requirement and the resource has no way to honor it.

Code sketch

TypeScript — minting and verifying a scoped key 
interface ValetKey {
  resource: string;
  action: "read" | "write";
  expiresAt: number;
  signature: string;
}

function mintKey(resource: string, action: ValetKey["action"], ttlMs: number, secret: string): ValetKey {
  const expiresAt = Date.now() + ttlMs;
  const payload = `${resource}:${action}:${expiresAt}`;
  return { resource, action, expiresAt, signature: hmacSha256(payload, secret) };
}

// Run by the resource itself, never by the server that minted the key
function verifyKey(key: ValetKey, requestedResource: string, secret: string): boolean {
  if (Date.now() > key.expiresAt) return false;              // expired
  if (key.resource !== requestedResource) return false;       // wrong scope
  const payload = `${key.resource}:${key.action}:${key.expiresAt}`;
  return hmacSha256(payload, secret) === key.signature;        // tamper check
}

// App server mints a 5-minute write key, client uploads directly to storage
const key = mintKey("uploads/report.pdf", "write", 5 * 60_000, SECRET);
const url = `https://storage.example.com/uploads/report.pdf?exp=${key.expiresAt}&sig=${key.signature}`;

In the wild

Amazon S3 pre-signed URLsThe app signs a URL with the caller credentials, scoped to one object and one HTTP method, with a bounded expiry (up to 7 days for signature v4); the browser then uploads or downloads straight to S3 with no account key exposed.
Azure Shared Access SignaturesGrants a named permission set (read, write, list) on a blob, container, or queue for a start-to-expiry window, optionally restricted by IP range and HTTPS-only; a user-delegation SAS is signed with Microsoft Entra credentials so the account key never leaves the server.
CloudFront signed URLs and cookiesUses a CloudFront key pair to grant time-limited access to private edge content; a signed URL covers one file while signed cookies cover a path pattern, and a custom policy can additionally restrict by IP range and date window.

In production

Tuning knobs

  • Token expiry (TTL) — The lifetime of the minted key; as short as the operation allows, since an unexpired key generally cannot be pulled back once issued.
  • Scope — The resource and the actions the key grants — one object, one method (read or write) — bounding what a leaked key can reach.
  • Single-use vs multi-use — Whether a key is invalidated after one redemption (closing the replay window) or reusable until expiry (fewer round trips, wider exposure).
  • Clock-skew tolerance — The leeway allowed on expiry validation to absorb small clock differences between the minting server and the resource that checks the key.

Signals to watch

  • Invalid-key rejection rate — Counts of keys rejected as expired, wrong-scope, or bad-signature at the resource — a spike can signal clock drift, replay attempts, or a broken minting path.
  • Key issuance rate — Volume of keys minted per unit time at the trusted server, watched for abuse or a runaway client.
  • Direct-to-resource traffic share — Proportion of bytes moving straight between client and resource versus through the app server — the bandwidth the pattern is offloading.

Failure modes under load

  • Over-broad or over-long grant — Getting scope or expiry wrong silently hands out more access, or a longer window, than the moment needed — and a leaked key stays usable for all of it.
  • No revocation — A valid, unexpired key generally cannot be revoked before it expires, so a compromised key keeps working until its window closes.
  • Clock skew — Disagreement between the minter and the validating resource rejects still-valid keys or honors already-expired ones.
  • Key leakage — A bearer credential in a URL leaks via logs, referrer headers, or link sharing, and whoever holds it can use it.

Readiness checklist

  • Expiry is as short as the operation allows and scope is one resource plus one action
  • Minting is authenticated and authorized — only the trusted party can issue keys
  • The signing secret is stored securely and rotated, and never leaves the server for the client
  • Clocks are synchronized (NTP) so the minter and the resource agree on expiry
  • A blast-radius plan exists for a leaked key — short TTL, or resource-side revocation where supported
