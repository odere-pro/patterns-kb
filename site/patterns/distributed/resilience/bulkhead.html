Bulkhead

Partitions threads, connections, or other shared resources into isolated compartments per dependency or tenant — so one failing or saturated consumer drains only its own pool, never the rest of the system's.

Distributed Resilience

What it is

A bulkhead partitions a shared, finite resource — a thread pool, a connection pool, a semaphore's permits — into separate compartments, one per downstream dependency, tenant, or call type. Each compartment has its own bounded capacity, and calls that use it can only ever exhaust that compartment, never the whole system's supply.

The problem it resolves is resource exhaustion by a noisy neighbor. When every call shares one pool, a single slow or hanging dependency can occupy every thread or connection in it while it waits, and unrelated calls that happen to use the same pool queue up and time out behind it too. One bad dependency effectively takes down every other one it never even talked to.

The name comes from a ship's hull: watertight bulkheads divide it into sealed compartments, so a hole below the waterline floods one compartment and the ship still stays afloat. Applied to software, the compartments are resource pools instead of steel walls, but the effect is the same — containment, not prevention.

How it works

Variations

Thread Pool per dependency
Dedicate a fixed-size thread pool to each downstream call. Classic and easy to reason about, at the cost of reserving threads that may sit idle most of the time.
Semaphore isolation
Guard each dependency with a counting semaphore instead of a dedicated pool — callers still run on a shared executor, but only a bounded number can be in-flight per dependency at once. Cheaper than extra threads.
Process or container isolation
Run a workload in its own process, container, or node so a crash, memory leak, or CPU spike can't touch the rest of the fleet — the same idea enforced by the OS instead of application code.
Per-tenant / per-priority partitioning
Split pools by customer tier or request priority rather than by downstream service, so one abusive or high-volume tenant can't starve capacity meant for everyone else.

Trade-offs

Pros

  • Contains resource exhaustion to one compartment instead of letting it cascade system-wide.
  • A saturated pool is a precise, attributable signal — you know exactly which dependency is unhealthy.
  • Works even against slow degradation, not just outright failure, since it never depends on detecting an error.
  • Compartments can be tuned and paired with other resilience patterns independently, dependency by dependency.

Cons

  • More pools mean more configuration, and reserved capacity that often sits idle.
  • Static sizing is a guess; undersize a pool and you've recreated the starvation it was meant to prevent.
  • Extra threads, connections, or processes cost more overall than one generously-sized shared pool.
  • Isolates the damage but does nothing to fix the failing dependency itself.

When to use it

Reach for it when

  • Multiple call paths or downstream dependencies currently share one thread or connection pool.
  • A single slow or saturated dependency must not be allowed to starve calls that don't even use it.
  • You serve multiple tenants or priority classes from shared infrastructure and need to isolate them.

Avoid when

  • There's genuinely only one downstream dependency — there's nothing else to isolate it from.
  • The resource is cheap and effectively unbounded, so partitioning adds cost without a real benefit.
  • A single Rate Limiter already caps total concurrency well enough without dividing it up.

Code sketch

TypeScript — a semaphore-based bulkhead per dependency 
class Bulkhead {
  private inFlight = 0;

  constructor(private readonly maxConcurrent: number) {}

  async run<T>(fn: () => Promise<T>): Promise<T> {
    if (this.inFlight >= this.maxConcurrent) {
      throw new Error("bulkhead full — rejecting immediately");
    }
    this.inFlight++;
    try {
      return await fn();
    } finally {
      this.inFlight--; // always release the permit, even on error
    }
  }
}

// One compartment per downstream dependency.
const orders = new Bulkhead(10);
const inventory = new Bulkhead(10);

// A stall calling orders can't touch inventory's budget.
await orders.run(() => callOrdersService());
await inventory.run(() => callInventoryService());

In production

Tuning knobs

  • Compartment size (maxConcurrentCalls / pool size) — The ceiling of in-flight calls one dependency may hold. Size it from each dependency's measured peak concurrency, not a shared guess.
  • Wait budget (maxWaitDuration / queue capacity) — How long or how many callers block for a permit before being rejected. Zero means fail-fast; a bounded queue trades latency for fewer rejections under bursts.
  • Partition dimension — Whether compartments split by downstream service, tenant tier, or request priority — the axis along which one noisy consumer must not starve the rest.
  • Isolation mechanism — Semaphore (bounded permits on a shared executor, cheap) versus a dedicated thread pool per dependency (true thread isolation, higher overhead).

Signals to watch

  • Compartment saturation — Permits or threads in use versus the compartment ceiling; sustained near-full means the dial is too tight or the dependency is degrading.
  • Rejection rate per compartment — Calls turned away because the compartment was full — attributable directly to one dependency.
  • Permit acquisition wait / queue depth — How long callers block before entering a full compartment; rising wait precedes rejections.

Failure modes under load

  • Undersized compartment — Set too small, it rejects calls under normal load and recreates the very starvation it was meant to prevent.
  • Semaphore isolation still shares threads — With semaphore-based bulkheads callers run on a shared executor, so a slow dependency still ties up shared threads while holding its permits — only thread-pool isolation truly separates them.
  • Sum of compartments exceeds the host — Over-provisioning every compartment can total more threads or connections than the node can supply, defeating the bound.

Readiness checklist

  • Size each compartment from measured peak concurrency, not a round number.
  • Emit per-compartment saturation and rejection metrics before rollout.
  • Decide fail-fast versus bounded wait explicitly for each compartment.
  • Load-test with one dependency artificially stalled to confirm isolation actually holds.
