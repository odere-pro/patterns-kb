Network · Resilience

Circuit Breaker

Stops calling a service that's already failing — so callers fail fast instead of piling on, and the struggling dependency gets room to recover.

Distributed Resilience

What it is

A circuit breaker wraps a call to a remote dependency and watches it fail. While the dependency is healthy the breaker is closed and calls pass straight through. Once failures cross a threshold, the breaker trips open: further calls return immediately with an error (or a fallback) without even attempting the network. After a cooldown it goes half-open and lets a trial call through — success closes it again, failure re-opens it.

The problem it resolves is cascading failure. When a downstream service slows or dies, naive callers keep calling, pile up blocked threads and connections, exhaust their own resources, and take down healthy services with them. The breaker converts a slow, resource-eating failure into a fast, cheap one, and stops hammering a dependency that needs to recover.

It borrows its name and its three-state logic from an electrical circuit breaker: better to trip and cut the current than to let the whole house burn.

How it works

Variations

Count-based vs. time-window
Trip after N consecutive failures, or after a failure rate (e.g. >50% of calls in the last 10 s). Rate-based handles bursty traffic more fairly.
Half-open trial policies
Allow a single probe, or a limited number of concurrent trials, before deciding to close. Limiting trials avoids a thundering-herd the moment the cooldown ends.
Fallback / graceful degradation
An open breaker can return a cached value, a default, or a queued request instead of a bare error — pairs naturally with the Null Object idea.
Per-endpoint vs. per-dependency
One breaker per remote host, or a finer breaker per operation. Finer granularity isolates a single bad endpoint without cutting off a whole service.

Trade-offs

Pros

  • Stops cascading failures from spreading across services.
  • Fails fast — frees threads, sockets, and memory that would block.
  • Gives the failing dependency space to recover instead of hammering it.
  • Its state is a high-signal health metric for dashboards and alerts.

Cons

  • Another stateful component to tune — bad thresholds cause flapping or false trips.
  • An open breaker can mask a dependency that's only mildly degraded.
  • Per-instance breakers don't share state; a cluster may trip unevenly.
  • Needs sensible fallbacks, or "fail fast" just becomes "fail loudly."

When to use it

Reach for it when

  • You call a remote service or resource that can fail or slow down.
  • A slow dependency could exhaust your own threads or connections.
  • You want to shed load off a struggling downstream so it can recover.

Avoid when

  • The call is local, in-process, and can't fail in this way.
  • Failures are permanent, not transient — fix the call, don't trip around it.
  • A simple timeout plus a bounded retry already covers the risk.

Prevents the smell of a whole system frozen behind one dead dependency — a step toward the Big Ball of Mud failure mode where nothing is isolated.

Code sketch

TypeScript — a minimal breaker 
type State = "closed" | "open" | "half-open";

class CircuitBreaker {
  private state: State = "closed";
  private failures = 0;
  private openedAt = 0;

  constructor(
    private readonly threshold = 5,
    private readonly cooldownMs = 10_000,
  ) {}

  async call<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === "open") {
      if (Date.now() - this.openedAt < this.cooldownMs) {
        throw new Error("circuit open — failing fast");
      }
      this.state = "half-open"; // time to probe
    }
    try {
      const result = await fn();
      this.reset();              // success closes the breaker
      return result;
    } catch (err) {
      this.recordFailure();
      throw err;
    }
  }

  private recordFailure() {
    this.failures++;
    if (this.state === "half-open" || this.failures >= this.threshold) {
      this.state = "open";
      this.openedAt = Date.now();
    }
  }
  private reset() { this.state = "closed"; this.failures = 0; }
}

In the wild

Netflix HystrixPopularized the pattern in the JVM world: trips on an error percentage over a rolling window once a minimum request volume is met (errorThresholdPercentage, requestVolumeThreshold), then waits a sleep window before probing. Now in maintenance mode, with Resilience4j the usual successor.
EnvoyOutlier detection ejects an upstream host after consecutive 5xx or gateway failures, holds it out for a base ejection time that grows with repeat ejections, then probes it back in — with max_ejection_percent capping how much of the pool can be removed at once.
PollyThe .NET resilience library; its circuit-breaker strategy trips on a failure ratio measured over a sampling duration above a minimum throughput, and composes in a pipeline with retry and timeout.

In production

Tuning knobs

  • Failure threshold — The count of consecutive failures or the failure-rate percentage over a window that trips the breaker open (failureRateThreshold, errorThresholdPercentage).
  • Open-state duration (cooldown) — How long the breaker stays open before admitting a half-open probe (waitDurationInOpenState, sleepWindow).
  • Half-open trial count — How many probe calls are permitted while half-open before deciding to close or re-open — limiting this prevents a rush on a recovering dependency.
  • Rolling window and minimum volume — The count or time span over which failures are measured, plus a minimum call count before the rate applies (slidingWindowSize, minimumNumberOfCalls, requestVolumeThreshold) so a single failure on a quiet endpoint does not trip it.
  • What counts as a failure — Which exceptions and timeouts increment the counter, and whether slow calls above a latency threshold count as failures too (slowCallDurationThreshold).

Signals to watch

  • Breaker state — Closed / open / half-open per breaker — a first-class health signal; a stuck-open breaker names the unhealthy dependency directly.
  • Failure / error rate in the window — The measured failure or slow-call percentage the trip decision is based on.
  • Fast-failed call count — Calls rejected immediately while the breaker is open — the load shed off the downstream.

Failure modes under load

  • Threshold-induced flapping — Thresholds set too tight make the breaker oscillate open and closed under marginal load, adding failures instead of preventing them.
  • Per-instance state divergence — Breakers are usually per-process and share no state, so a cluster trips unevenly and the aggregate load on a downstream is still uncoordinated.
  • Thundering herd at half-open — If half-open trials are not limited, the instant the cooldown ends every caller rushes the still-fragile dependency at once.
  • Never-trips on low traffic — A minimum-volume window set too high means a low-traffic endpoint never accumulates enough calls to trip, and the breaker silently does nothing.

Readiness checklist

  • Set the failure window and minimum request volume so low-traffic endpoints do not trip on a single error.
  • Limit half-open trial calls to avoid a thundering herd when the cooldown ends.
  • Export breaker state and its transitions as metrics with alerts.
  • Define a fallback for the open state rather than returning a bare error.
  • Pair with a timeout so slow calls actually register as failures.
