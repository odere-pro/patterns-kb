Leader Election

Exactly one node in the cluster acts as coordinator at any moment — every peer runs the same protocol to agree on who that is, so writes, scheduling, and partition ownership never have to contend for the job.

Distributed Coordination & Data

What it is

Leader election is a protocol by which a set of otherwise-equal peer nodes agrees on exactly one of themselves to act as coordinator for some period. Candidates propose themselves, the group converges through votes, priority, or a lease granted by a coordination service, and the result is a single, mutually recognized leader — until it fails or steps down and the group runs the protocol again.

The force it resolves is the need for a single point of serialization without a single point of failure. Some jobs are only correct if exactly one node does them — issuing writes to a partition, running a scheduled task once, owning a lock — but hard-coding which node that is makes the whole system fragile: if that process dies, everything downstream waits for a human to promote a replacement. Leader election lets the survivors detect the absence and agree on a replacement themselves, with no operator in the loop.

The hazard it must guard against is split brain: a network partition or a slow, GC-paused node can let two nodes each believe they're leader at once, and both start writing. Correct implementations tie leadership to a monotonically increasing term or epoch and a fencing token, so a stale leader's writes are recognizable and rejected by anything downstream, even if it hasn't noticed it lost the job.

How it works

Variations

Bully algorithm
The node with the highest ID declares itself leader once it notices the current one is unreachable; lower-ID nodes yield. Simple, but chatty — a recovering high-ID node can trigger repeated re-elections.
Ring algorithm (Chang-Roberts)
An election message circulates a logical ring of nodes, each forwarding the higher of its own ID and whatever it received; the message returns to its originator carrying the winner.
Quorum & Consensus
Term-based voting, as in Raft or Paxos: a candidate needs a majority quorum before it may act as leader, tying the election itself to the same safety guarantee that protects writes.
Gossip Protocol
Liveness and leadership state propagate peer-to-peer instead of through direct heartbeats to every node — scales better across large, high-latency clusters at the cost of slower convergence.
Lease-based external election
Delegate the decision to a coordination service (ZooKeeper, etcd, Consul): a node holds leadership only while it can renew a time-bound lease, so a crashed leader's slot expires on its own.

Trade-offs

Pros

  • Serializes coordination through one node, so contending replicas never need to fight over the same write.
  • Automatic failover — the group detects a dead leader and elects a fresh one without an operator.
  • Once elected, the leader acts unilaterally, keeping consensus overhead off the hot path.
  • Well-studied algorithms (Raft, Paxos, ZAB, Bully) come with proofs, not just folklore.

Cons

  • The elected leader is a bottleneck and a single target for load on everything it owns.
  • Split brain is a real risk under partitions — needs terms, epochs, or fencing tokens to rule out.
  • Flaky networks or long pauses cause election churn, repeatedly disrupting an otherwise-healthy leader.
  • Getting the protocol right — timeouts, quorum sizes, fencing — is genuinely hard to do from scratch.

When to use it

Reach for it when

  • Several replicas exist, but exactly one must issue writes, run a scheduled job, or own a partition.
  • You need automatic failover with no human deciding who takes over next.
  • Serializing through a leader is cheaper than having every node coordinate with every other on each write.

Avoid when

  • A single always-on process is acceptable — a static, manually assigned owner needs no election.
  • Work can be sharded so nodes never compete for the same duty in the first place.
  • Idempotent, at-least-once processing is good enough — sidestep exclusive ownership entirely.

Code sketch

TypeScript — a lease-based elector 
interface LeaseStore {
  acquire(nodeId: string, ttlMs: number): Promise<boolean>;
  renew(nodeId: string, ttlMs: number): Promise<boolean>;
}

class LeaderElector {
  private isLeader = false;

  constructor(
    private readonly store: LeaseStore,
    private readonly nodeId: string,
    private readonly ttlMs = 5_000,
  ) {}

  async tryBecomeLeader(): Promise<boolean> {
    this.isLeader = await this.store.acquire(this.nodeId, this.ttlMs);
    return this.isLeader;
  }

  // Call on an interval well under ttlMs.
  async renew(): Promise<void> {
    if (!this.isLeader) return;
    const stillHeld = await this.store.renew(this.nodeId, this.ttlMs);
    if (!stillHeld) this.isLeader = false; // lease expired, step down
  }

  get leading(): boolean { return this.isLeader; }
}
