MapReduce

MapReduce processes data too big for one machine by running a map function in parallel over input splits, shuffling the outputs together by key, and folding each key's values into a result with a reduce function — while the framework handles distribution, parallelism, and re-running failed tasks.

Distributed Coordination & Data

What it is

MapReduce runs a computation over a dataset in three stages. First map: a function is applied in parallel to each of many input splits, emitting intermediate key/value pairs. Then shuffle: the framework groups all the pairs by key, so every value for a given key lands together. Finally reduce: a function folds each key's values into a result. Word count is the canonical example — map emits (word, 1) for every word, shuffle gathers all the 1s per word, reduce sums them.

The force it resolves is the single-machine ceiling. A dataset can be far too big to fit on one node, and a computation over billions of records can take hours on a single core no matter how tight the loop. MapReduce spreads both the data and the work across a cluster: add machines and the map and reduce stages parallelize onto them. The programmer writes only two functions; the framework handles distribution, parallelism, data movement, and fault tolerance — a task that dies is simply re-run somewhere else, so hardware failure at scale is routine rather than fatal.

The same map-then-reduce shape shows up well beyond classic batch analytics: long-document summarization (summarize each chunk in parallel, then combine the summaries), distributed data preprocessing, and large-scale embedding generation for model training and retrieval all follow it.

How it works

Variations

Classic batch vs. in-memory DAG
The original model (Hadoop) writes each stage to disk. In-memory DAG engines like Spark generalize map/shuffle/reduce into a graph of transformations kept in memory — far faster for multi-pass and iterative work, at the cost of holding data in RAM.
Combiner
A map-side pre-aggregation that partially reduces each map task's output before it is shuffled. When the reduce is associative (a sum, a count), a combiner shrinks the volume moved across the network dramatically.
Map-only jobs
Embarrassingly-parallel transforms — filtering, format conversion, enrichment — that need no grouping. Drop the reduce stage entirely and each map task writes its output directly.
Multi-stage pipelines
Real work rarely fits one map/reduce pass. Chain several jobs, or express the whole computation as a DAG, so the output of one stage feeds the next without a manual hand-off.

Trade-offs

Pros

  • Scales horizontally to enormous datasets — add machines and the work spreads onto them.
  • Parallelism and fault tolerance are automatic; a task that fails just gets re-run elsewhere.
  • A simple mental model — write two functions and let the framework handle the hard parts.
  • Moving computation to the data keeps most reads local and cuts network traffic.

Cons

  • High latency — it is a batch model, not interactive; jobs run in minutes to hours.
  • The shuffle is expensive: grouping by key moves data across the network and spills to disk.
  • Awkward for iterative algorithms — each pass re-reads and re-writes the whole dataset.
  • A skewed key sends most values to one reducer, making it the bottleneck for the whole job.

When to use it

Reach for it when

  • A batch computation runs over a very large, partitionable dataset.
  • You need aggregations, counts, or joins at a scale one machine cannot handle.
  • The work is a one- or few-pass transform where end-to-end latency does not matter.

Avoid when

  • You need low-latency or interactive query response.
  • The algorithm is heavily iterative — reach for an in-memory DAG engine instead.
  • The data fits comfortably on a single machine — the overhead is not worth it.
  • You need strong per-record transactions, not bulk fold-over-everything.

Code sketch

TypeScript — single-process word count, showing the shape 
type Pair<K, V> = [K, V];

// MAP: turn each input into zero or more key/value pairs.
function map(doc: string): Pair<string, number>[] {
  return doc
    .toLowerCase()
    .match(/[a-z]+/g)
    ?.map((word) => [word, 1] as Pair<string, number>) ?? [];
}

// REDUCE: fold all values for one key into a single result.
function reduce(word: string, counts: number[]): Pair<string, number> {
  return [word, counts.reduce((a, b) => a + b, 0)];
}

function mapReduce(docs: string[]): Map<string, number> {
  // map over every input (a real system runs this in parallel across nodes)
  const mapped = docs.flatMap(map);

  // shuffle: group every value by its key
  const grouped = new Map<string, number[]>();
  for (const [word, count] of mapped) {
    (grouped.get(word) ?? grouped.set(word, []).get(word)!).push(count);
  }

  // reduce each group
  const out = new Map<string, number>();
  for (const [word, counts] of grouped) {
    const [k, v] = reduce(word, counts);
    out.set(k, v);
  }
  return out;
}

mapReduce(["the cat sat", "the dog sat"]); // { the: 2, sat: 2, cat: 1, dog: 1 }

In the wild

Google MapReduceThe 2004 system and paper that popularized the model, used internally to build the search index across thousands of machines.
Apache Hadoop MapReduceThe open-source implementation: map and reduce tasks over HDFS, with automatic re-execution of failed tasks.
Apache SparkGeneralizes MapReduce into an in-memory DAG of transformations — much faster for multi-pass and iterative jobs while keeping the map/shuffle/reduce model.
LangChain map-reduce summarizationSummarizes a document longer than the context window by summarizing each chunk in parallel (map) and combining those summaries (reduce).

In production

Tuning knobs

  • number of partitions / splits — how finely the input is divided across map tasks
  • number of reducers — parallelism of the reduce stage
  • combiner — map-side pre-aggregation, on or off, to cut shuffle volume

Signals to watch

  • task skew — slowest reducer vs the median — a hot key shows up here
  • shuffle bytes — data moved between map and reduce; the usual bottleneck
  • failed / retried task count — how much work is being re-run
  • job wall-clock — end-to-end time, dragged by stragglers

Failure modes under load

  • data skew — one hot key sends most values to a single reducer that then dominates the job
  • shuffle saturation — the all-to-all data movement overwhelms network or disk
  • stragglers — a few slow tasks hold up the whole job
  • out-of-memory on a hot key — a reducer can't hold all values for one key

Readiness checklist

  • partition to balance keys and avoid skew
  • use a combiner to shrink the shuffle where the reduce is associative
  • size the reducer count to the data volume
  • keep tasks idempotent — the framework re-runs them
