Count-Min Sketch

A compact grid of counters that answers "about how many times has this key appeared?" over a stream — using memory fixed in advance, independent of how many distinct keys flow through it, at the cost of a small over-estimate it can never turn into an undercount.

Distributed

What it is

A Count-Min Sketch is a two-dimensional array of counters — d rows by w columns, all zero to start — with one independent hash function per row. To record an occurrence of a key, hash it once per row and increment the single counter it lands on in that row, so each occurrence bumps d counters. To estimate how often a key has been seen, hash it the same d ways, read those counters, and take the minimum. The minimum, because every counter is shared by all the keys that hash to it, so a collision can only push a count up, never down — the smallest of a key's cells is the one least polluted by other keys, and the number it holds is an upper bound, never an undercount.

The force it resolves is counting per-item frequency over a stream when the keys are far too many to hold an exact counter for each. An exact tally needs one counter per distinct key — trivial for thousands, hopeless for billions of URLs, IP addresses or search terms. The sketch fixes its size up front at w×d counters, a footprint that does not grow with the number of distinct keys no matter how many pour through it. It buys that fixed memory with a bounded over-estimate: width w governs how much error each estimate can pick up from collisions, and depth d governs the confidence that the true error stays within that bound.

Cormode and Muthukrishnan described the structure in 2005. Like a Bloom filter it stores no keys — only collided counts — so it can tell you roughly how often a key you name has appeared, but it can neither list the keys it has seen nor recover them. That is why it is almost always paired with something that already holds the candidate keys: a small heap of hot keys turns it into an approximate top-K, and a running set of heavy hitters turns it into a trending list. On its own it is a frequency oracle you have to know what to ask.

How it works

Variations

Conservative Update
On insert, raise only the counters that are currently at the key's minimum rather than all d of them. This shaves the over-estimate substantially on skewed streams, at the price of no longer supporting decrements or deletions.
Count-Mean-Min Sketch
Estimates the collision noise in each row from the total volume, subtracts it from each counter, and takes the median instead of the minimum. This corrects much of the upward bias — useful when the long tail matters, not only the heavy hitters.
Count Sketch
Adds a second ±1 hash per row and takes the median of the signed counters, giving an unbiased estimate that can err in either direction rather than a one-sided upper bound. Better when rare items and frequent ones both need estimating.
Decayed / sliding-window sketch
Periodically scales every counter down, or keeps per-window sketches and ages the oldest out, so recent occurrences outweigh old ones — keeping the estimate tuned to what is frequent now rather than all-time totals.

Trade-offs

Pros

  • Fixed memory, independent of the number of distinct keys — w×d counters whether you track thousands of keys or billions.
  • Insert and query are O(d) — a handful of hashes and increments, constant no matter how long the stream or how many keys.
  • Never undercounts: the estimate is a guaranteed upper bound, so a key's true frequency is always at or below what the sketch reports.
  • Two sketches of the same dimensions add cell-by-cell — combine per-shard sketches into a global one, so counting parallelizes across a cluster.

Cons

  • Over-estimates by design, and the inflation grows with total volume and skew — heavy hitters spill onto everything that collides with them.
  • Cannot enumerate or recover keys — it stores counts against hashed cells, not the keys, so it needs a companion structure that already knows the candidates.
  • Accurate for the heavy hitters, noisy for the long tail — a rare key's small true count is easily swamped by a frequent key sharing one of its cells.
  • Width and depth are sized upfront for an assumed volume and error target; a stream far larger or more skewed than planned drifts past the bound.

When to use it

Reach for it when

  • You need approximate per-item frequencies — or the heavy hitters and top-K — over a stream with far too many distinct keys to keep an exact counter for each.
  • A small over-count is acceptable and you care most about the frequent items: trending topics, hot keys, most-viewed content.
  • You want fixed, predictable memory and mergeable per-shard counters, so the counting can spread across a cluster.

Avoid when

  • You need exact counts, or downstream logic breaks on an over-estimate — reach for exact counters or a database aggregate.
  • The number of distinct keys is small enough that a plain hash map of counters fits comfortably in memory.
  • The question is "how many distinct items" rather than "how often each one" — that is a cardinality estimator like HyperLogLog; "is this item present at all" is a Bloom filter.

Code sketch

TypeScript — a minimal sketch 
class CountMinSketch {
  private readonly rows: Uint32Array[];

  constructor(private readonly width = 1 << 16, private readonly depth = 5) {
    this.rows = Array.from({ length: depth }, () => new Uint32Array(width));
  }

  // one independent hash per row — seed each with the row index
  private column(row: number, key: string): number {
    return fnv1a(key + ":" + row) % this.width;
  }

  add(key: string, count = 1): void {
    for (let r = 0; r < this.depth; r++) this.rows[r][this.column(r, key)] += count;
  }

  estimate(key: string): number {
    let min = Infinity;
    for (let r = 0; r < this.depth; r++) {
      min = Math.min(min, this.rows[r][this.column(r, key)]);
    }
    return min; // an upper bound — collisions only inflate, never deflate
  }
}

In the wild

Redis Stack Count-Min SketchA first-class sketch type: CMS.INITBYDIM or CMS.INITBYPROB to size it, CMS.INCRBY to count occurrences, CMS.QUERY to estimate, and CMS.MERGE to fold several sketches into one.
Apache SparkDataFrame.stat.countMinSketch(col, eps, confidence, seed) builds a sketch over a column, giving approximate frequencies without a full group-by shuffle over the whole dataset.
Twitter AlgebirdProvides Count-Min Sketch as a mergeable monoid for streaming aggregation; its top-N variant keeps the heaviest keys alongside the sketch — the classic sketch-plus-heap route to approximate top-K.
CaffeineIts Window-TinyLFU eviction keeps a count-min frequency sketch — 4-bit counters, periodically halved to age out old counts — to judge which keys are hot enough to admit, an LFU-style signal.
