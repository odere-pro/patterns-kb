Federated Identity

One identity provider authenticates a user once, and every participating service accepts that provider's signed word for who they are instead of checking a password of its own.

Distributed Coordination & Data

What it is

Federated identity lets one identity provider (IdP) authenticate a user once and then vouch for that identity to any number of independent relying services, which never see a password themselves. The IdP issues a signed, time-bound proof — a SAML assertion or an OpenID Connect ID token — and each relying service checks its signature against a trust anchor the two sides agreed on beforehand. Trusting the signature is enough; there's no callback to the IdP on every subsequent request.

The forces it resolves are password proliferation and duplicated authentication logic. Left alone, every service ends up with its own credential store, its own reset flow, its own MFA implementation, and its own breach surface — while users reuse the same weak password across all of them anyway. Centralizing authentication in one IdP means MFA, password policy, and session lifetime are enforced once, and disabling a user in one place cuts off access everywhere at once.

Establishing that trust is a deliberate, out-of-band step: exchanging metadata — issuer URL, signing certificate or JWKS endpoint, supported bindings — so each side can verify the other cryptographically before any user ever logs in. That's what separates federation from a shared login screen: the IdP and the relying party can be owned by entirely different organizations, and neither has to trust the other's infrastructure, only its signed claims.

That same centralization is the pattern's sharpest edge. Every relying service inherits the IdP's trust wholesale, so a compromised IdP or a forged assertion compromises all of them simultaneously — federation trades many small doors for one much larger one that has to be defended accordingly.

How it works

Variations

SAML 2.0
XML assertions signed by the IdP, delivered via browser redirect or POST bindings — still the default for enterprise and government SSO.
OpenID Connect (OIDC)
OAuth 2.0 plus a signed ID token in JWT form — lighter weight than SAML, and the default for modern web and mobile sign-in.
SP-initiated vs. IdP-initiated
The flow can start when a user hits the service and gets redirected out, or from a link on the IdP's own portal. SP-initiated preserves a state parameter and resists injected-assertion attacks better.
Identity broker
A middle party sits between many identity providers and many relying services so each side integrates once instead of pairwise — the shape behind Okta, Auth0, and Azure AD B2C.
Just-in-time provisioning
The relying service creates or updates the local user record automatically on first successful assertion, so no one has to pre-register accounts by hand.

Trade-offs

Pros

  • One login serves every participating service — users manage a single credential.
  • MFA, password policy, and session lifetime are enforced once, at the IdP.
  • Disabling a user at the IdP revokes their access everywhere at once.
  • Relying services can drop their own credential store, shrinking a major breach surface.
  • Enables partner and B2B sign-in without provisioning a fresh account per user per system.

Cons

  • The IdP becomes a single point of trust and failure — compromise it and every service is compromised.
  • Trust setup is real operational overhead: metadata exchange, certificate rotation, clock skew.
  • Global logout is unreliable — killing the IdP session doesn't always end every SP-side session.
  • Debugging spans two systems and a redirect chain; an audience or clock mismatch fails cryptically.
  • Every relying service is now hard-dependent on the IdP's uptime to let anyone in at all.

When to use it

Reach for it when

  • Users need one identity across many services, internal or partner-owned.
  • You want to centralize authentication policy — MFA, password rules, revocation — in one place.
  • An enterprise customer requires SSO through their own identity provider to onboard at all.

Avoid when

  • You run a single, standalone service with nothing else to federate with — a local login is simpler.
  • The relationship is a one-off, machine-to-machine call — a scoped API key or client credential suffices.
  • You can't tolerate the IdP as a hard external dependency for every login, as in offline or air-gapped environments.

Code sketch

TypeScript — verifying an OIDC assertion from a trusted IdP 
import { createRemoteJWKSet, jwtVerify } from "jose";

const idp = {
  issuer: "https://idp.example.com",
  jwks: createRemoteJWKSet(new URL("https://idp.example.com/.well-known/jwks.json")),
};

async function federatedLogin(idToken: string): Promise<LocalSession> {
  const { payload } = await jwtVerify(idToken, idp.jwks, {
    issuer: idp.issuer,
    audience: "our-service-client-id", // reject tokens minted for someone else
  });

  // We trust the IdP's claims outright — we never saw a password.
  const sub = payload.sub as string;
  const email = payload.email as string;

  const user = await findOrProvisionUser(sub, email); // just-in-time provisioning
  return startLocalSession(user); // our own session, scoped to our service only
}
