Quorum & Consensus

A majority of nodes must acknowledge a write before it counts as durable — so no single replica, however fast or however wrong, gets to decide the outcome alone.

Distributed Coordination & Data

What it is

In a system with N replicas of some data, a quorum is the minimum number of nodes that must take part in an operation for it to be trusted. A write commits once W replicas acknowledge it; a read is considered authoritative once R replicas agree. Choose W and R so that W + R > N, and every possible read quorum is guaranteed to overlap with every possible write quorum on at least one replica — so a read can never miss the latest committed write.

The force it resolves is that no single node can be fully trusted alone. Machines crash, disks corrupt, networks partition, and any one replica can be stale or simply unreachable at the exact moment you need it. Waiting for every replica to agree makes the system unavailable the instant one node is slow. Trusting any single replica makes it wrong the instant that one lags or lies. Quorum sits between the two extremes: it tolerates a bounded number of failed or unreachable nodes without stalling, and a bounded number of stale nodes without being fooled, because a majority can't be wrong at the same time — so long as failures aren't correlated.

This same intersection math underlies two related but distinct problems. Dynamo-style quorum systems use only the overlap guarantee to keep reads and writes consistent, with no notion of global ordering. Consensus protocols like Paxos and Raft build further on the same majority arithmetic, adding a leader and a replicated log so that nodes agree not just that a write happened, but on the exact order every write happened in.

How it works

Variations

Read/write quorum tuning (W + R > N)
Pick W and R so they sum past N — e.g. N=3, W=2, R=2 — guaranteeing every read touches at least one replica that saw the latest write.
Strict vs. sloppy quorum
A strict quorum only counts the N designated replicas. A sloppy quorum accepts acks from any reachable node and hands the write off later (hinted handoff), trading strict consistency for availability during a partition.
Majority vs. weighted quorum
Classic majority gives every replica one vote. A weighted quorum assigns more votes to reliable or well-connected nodes, so the outcome doesn't hinge on a single flaky one.
Flexible / asymmetric quorums
Decouple the quorum used to elect a leader from the quorum used to replicate entries, as long as the two still intersect — shrink one and the other can grow, cutting latency on the hot path.
Leader-based consensus (Paxos, Raft)
Layer a stable leader and an ordered log on top of the same majority arithmetic, so replicas agree not just that a write happened but in what order — quorum becomes full consensus.

Trade-offs

Pros

  • Tolerates minority failures without stopping — up to floor((N-1)/2) nodes can be down or partitioned.
  • No single node is a single point of truth; a stale or bad replica gets outvoted.
  • Consistency and availability are tunable via W and R, not baked in as an all-or-nothing choice.
  • Correctness rests on simple, well-understood intersection math, not on trusting any one participant.

Cons

  • Majority requires reaching most of the cluster — a bad enough partition can leave every side short of quorum.
  • Adds write latency: the request isn't done until W replicas answer, not one.
  • Overlap alone gives no ordering — concurrent conflicting writes still need versioning or a real consensus protocol.
  • Changing cluster membership mid-flight is delicate; quorum math assumes a known, stable N.

When to use it

Reach for it when

  • Data is replicated across multiple nodes and no single copy can be trusted alone.
  • You need to survive node failure or a network partition without halting every write.
  • You can afford the latency of waiting on more than one acknowledgment per operation.

Avoid when

  • A single authoritative source of truth already exists — there's nothing for a quorum to arbitrate.
  • You need strict, linearizable ordering and haven't paired quorum with a real consensus protocol.
  • The added round trips of touching multiple nodes per write blow your latency budget.

Code sketch

TypeScript — a minimal quorum write 
async function quorumWrite<T>(
  replicas: ReplicaClient[],
  value: T,
  w: number,
): Promise<void> {
  const n = replicas.length;
  if (w > n) throw new Error("quorum exceeds replica count");

  return new Promise((resolve, reject) => {
    let acked = 0;
    let failed = 0;
    for (const replica of replicas) {
      replica.write(value).then(
        () => {
          if (++acked >= w) resolve(); // quorum reached, stop waiting
        },
        () => {
          if (++failed > n - w) reject(new Error("quorum unreachable"));
        },
      );
    }
  });
}

// N=3, W=2: tolerate one slow or failed replica per write
await quorumWrite(replicas, { key: "x", val: 42 }, 2);
