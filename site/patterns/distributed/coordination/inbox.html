Inbox

Persists a received message and applies its effects in the same local transaction, then safely acknowledges to the broker — so a message is never lost, never duplicated, even if the service crashes mid-processing, and the receipt and the write can never drift apart.

Distributed Coordination & Data

What it is

The inbox pattern adds a table to a service's own database that holds messages received from a broker, waiting to be processed. When the service receives a message, it writes both the message row and the business row that applies the message's effects — inside the same local ACID transaction. Only after commit does it acknowledge the message back to the broker. A separate process then polls the inbox for unacknowledged messages and retries them, ensuring no message is lost.

It resolves the dual-write problem from the receiving side. A service's database and its message broker are two different systems with no shared transaction between them. Receive and process the message, then acknowledge, and a crash between the two leaves the broker believing the message was lost — it redelivers, but the service has no record that it already processed it, leading to duplication. Acknowledge and then process, and a crash or a processing failure leaves the service with orphaned messages that no one knows about. There's no way to make "read from the broker" and "write to the database" atomic across two resource managers without something like distributed two-phase commit, which most message brokers don't support and most teams don't want to run.

Inbox sidesteps the problem instead of solving it head-on: it turns the receipt into a second local write, which the database transaction already makes atomic with the first. The handler must be idempotent — because the broker's "at-least-once" guarantee means the same message can arrive multiple times — but idempotency is a small, local concern, not a distributed coordination nightmare. The genuinely hard part — not losing or duplicating messages — is pushed onto the pattern and the handler's idempotency contract.

How it works

Variations

Synchronous processing
The handler processes the message immediately, inside the same request. The message and result commit together. Fast and simple, but if processing is slow it blocks the connection and handler timeouts become hazardous.
Deferred processing
The handler writes the message to the inbox and returns immediately, delegating processing to a background job. The inbox row tracks progress; the job polls for unprocessed rows and applies them. Decouples latency and reduces timeout risk.
Per-aggregate inbox
Each aggregate (domain object) gets its own inbox table instead of a shared one. Ordering guarantees stay scoped to one aggregate's queue, and hot tables don't block each other.
CDC (Change Data Capture)
A tool like Debezium tails the database's write-ahead log and turns each inbox insert into a message to a processing topic or trigger, eliminating polling and visibility window timeouts.

Trade-offs

Pros

  • Solves the dual-write problem from the receiving side with a single local ACID transaction — no distributed transaction coordinator needed.
  • Guarantees at-least-once reception: a message is never lost because the service crashed between receipt and acknowledgement.
  • Handler can be offline during broker visibility window; ack process retries indefinitely, so window pressure is eliminated.
  • Works with any relational database and any broker; it's a pattern, not a product.

Cons

  • Adds a table, an ack process, and one more moving part to deploy and monitor.
  • Delivers at-least-once, not exactly-once — handlers MUST be idempotent, not optional.
  • Polling ack processes add latency and load; CDC infrastructure (if used) adds operational complexity.
  • The inbox table grows unbounded without its own archiving or cleanup job.

When to use it

Reach for it when

  • A service must receive and reliably process messages from a broker or event stream.
  • You need to guarantee that a message is never lost or silently duplicated — and you can afford idempotent processing.
  • The database and the message broker are separate systems with no shared transaction between them.

Avoid when

  • Your handler's effects are already idempotent (e.g., reads, cache invalidations, queries with deduplication built in) and losing a message is acceptable.
  • Broker connectivity is always reliable and an occasional missed message is acceptable.
  • The extra table and ack process complexity outweigh the reliability benefit for your domain and scale.

Code sketch

TypeScript — receiving, storing, and processing with idempotency 
// Handler receives message and stores it atomically
async function handleMessage(tx: Transaction, message: Message) {
  // Same local transaction: inbox row + business row
  const idempotencyKey = message.id;
  const existing = await tx.query(
    "select id from inbox where idempotency_key = $1",
    [idempotencyKey],
  );
  if (existing.rows.length > 0) {
    // Already processed (broker retransmitted); return idempotently
    return;
  }

  await tx.query(
    `insert into inbox (id, idempotency_key, message, processed)
     values ($1, $2, $3, false)`,
    [randomUUID(), idempotencyKey, JSON.stringify(message)],
  );

  // Apply business logic
  await tx.query(
    "insert into orders (id, status) values ($1, $2)",
    [message.orderId, "received"],
  );
  // Both rows commit together, or neither does.
}

// Ack process: polls inbox and safely acknowledges
async function ackOnce(db: Db, broker: Broker) {
  const rows = await db.query(
    "select * from inbox where processed = false order by id limit 100",
  );
  for (const row of rows) {
    try {
      // Re-ack is idempotent on the broker's side
      await broker.ack(row.idempotency_key);
      await db.query(
        "update inbox set processed = true where id = $1",
        [row.id],
      );
    } catch (e) {
      // Broker rejected; leave unprocessed and retry later
      console.error(`Failed to ack ${row.id}:`, e);
    }
  }
}

In production

Tuning knobs

  • Handler timeout — For synchronous processing, how long the broker waits before timing out and redelivering. Longer reduces redelivery pressure but risks losing the connection; shorter is safer but may cause false timeouts on slow processing. Deferred processing sidesteps this entirely.
  • Broker visibility window — How long the broker hides a received message before auto-redelivering if not acknowledged. Must be longer than handler latency + ack latency. Shorter windows create thrashing; longer windows tolerate brief outages.
  • Ack process poll interval — For a polling ack process, how often it scans for unacknowledged rows. Shorter cuts ack latency but raises steady read load; CDC sidesteps polling entirely.
  • Cleanup / retention cadence — How aggressively a job deletes or archives acknowledged rows. Without it the table grows unbounded and the poll query slows as it scans dead rows.

Signals to watch

  • Inbox backlog depth — Count of rows with processed = false. Steady growth means the ack process is not keeping up or handlers are failing.
  • Processing latency — Time between message arrival and processed = true. The end-to-end latency; dominated by handler latency and ack process poll interval.
  • Handler error rate — Failed processing attempts per interval. Rising errors stall the inbox and leave messages unacknowledged.
  • Broker redelivery rate — Messages the broker redelivered due to unacknowledged or visibility window timeout. High redelivery creates duplicate processing pressure on idempotent handlers.

Failure modes under load

  • Handler timeout cascade — Slow processing → visibility window expires → broker redelivers → duplicate processing pressure → handler slower → more timeouts. Deferred processing breaks this cycle.
  • Inbox backlog starvation — Handler failures or a rejecting downstream service stall the ack process; unacknowledged messages pile up and eventually cause broker memory pressure.
  • Idempotency key collision — Two different messages with the same id (broker bug, clock skew, key generation flaw) → second message silently dropped. Verify key uniqueness at generation time.
  • Table bloat stalls polling — Cleanup lags and the poll query scans millions of acknowledged rows, driving its own latency up and worsening the backlog it was meant to drain.

Readiness checklist

  • Handlers are demonstrably idempotent — same message id produces same result, no matter how many times delivered
  • A partial index on processed = false keeps the ack poll query fast as the table grows
  • A cleanup or archiving job removes acknowledged rows on a schedule
  • Visibility window is longer than handler latency + ack poll interval + network jitter
  • Inbox backlog depth and processing latency are monitored with alerts on sustained growth or degradation
