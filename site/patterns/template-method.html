Template Method

Locks the shape of an algorithm in a base class — the steps and their order never change — while subclasses supply the parts that vary.

GoF Behavioral

What it is

Template Method defines the skeleton of an algorithm in one method of a base class and defers the varying parts to subclasses. That method — the template — calls a fixed sequence of operations: some are concrete and shared, some are abstract and left blank, some are hooks with a default. Subclasses override the blanks and hooks to fill in behavior without ever touching the sequence.

The forces it resolves are duplication and drift. You have several procedures that share the same overall shape but differ in a step or two. Copy the whole procedure once per variant and you copy the invariant scaffolding with it — and those copies quietly diverge over time. Template Method factors the invariant part into a single place and isolates the variant steps behind overridable operations.

Control is inverted: the base class owns the flow and calls down into the subclass at the right moments. A subclass cannot reorder, skip, or wrap the steps — it only supplies them. This is the Hollywood Principle, "don't call us, we'll call you," expressed through inheritance.

How it works

Variations

Abstract primitive operations
Steps with no default that subclasses must implement — the compiler enforces that every variant fills them in.
Hook methods
Steps with a default (often empty) body; overriding is optional, so subclasses opt into an extension point only when they need it.
Non-Virtual Interface (NVI)
Keep the template method itself non-overridable and expose only the steps as virtual/protected — callers get one stable entry point.
Factory Method
When a varying step's whole job is to create an object, that step becomes a factory method the subclass supplies.

Trade-offs

Pros

  • Removes duplication by holding the invariant algorithm in one place.
  • The base class owns the sequence, so no variant can get the order wrong.
  • A new variant implements only the steps that actually differ.
  • Hooks give clean, named extension points without editing the skeleton.

Cons

  • Built on inheritance — each variant is bound to one base class and its lifecycle.
  • The set of overridable steps becomes a contract that is painful to change later.
  • Behavior is split between base and subclass, so the real flow can be hard to trace.
  • Deep hierarchies and many hooks make the skeleton rigid and brittle.

When to use it

Reach for it when

  • Several procedures share one structure but differ in a couple of steps.
  • You want to enforce a fixed sequence of steps across all variants.
  • The varying steps are known up front and stable in number.

Avoid when

  • The varying parts must swap independently at runtime — reach for Strategy.
  • Composition would serve better than binding behavior into a class hierarchy.
  • There is only one variant today — the abstraction is premature.

Code sketch

TypeScript — a skeleton with overridable steps 
abstract class ReportBuilder {
  // the template method — the fixed skeleton
  build(): string {
    const rows = this.loadRows();
    const body = rows.map((r) => this.formatRow(r)).join("\n");
    return this.header() + body + this.footer();
  }

  // mandatory steps — every variant must supply these
  protected abstract loadRows(): string[];
  protected abstract formatRow(row: string): string;

  // hooks — sensible defaults, override only if needed
  protected header(): string { return ""; }
  protected footer(): string { return ""; }
}

class CsvReport extends ReportBuilder {
  protected loadRows() { return ["ada,1", "bob,2"]; }
  protected formatRow(r: string) { return r; }
  protected header() { return "name,value\n"; }
}
