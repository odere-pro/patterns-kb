Anti-Corruption Layer

Sits at the boundary between your bounded context and an external system or legacy model, translating every call in both directions — so foreign concepts, names, and quirks never leak into your own domain model.

DDD Lens

What it is

An anti-corruption layer (ACL) is a boundary layer that sits between your bounded context and a model you don't control — a legacy system, a third-party service, or simply another bounded context with its own ubiquitous language. It translates every call and every response crossing the boundary, so your domain code only ever sees its own types, names, and invariants.

The force it resolves is model corruption. Integrate directly with an external system's API or schema and its vocabulary, quirks, and historical mistakes creep into your domain call by call — a status enum built from someone else's error codes, a record shaped by their normalization decisions, a workflow bent to fit their state machine. Left unchecked, your model gradually mirrors theirs, and you lose the freedom to model your own domain the way it actually works.

Eric Evans coined the term for exactly this situation: a new, carefully modeled bounded context has to talk to an old system that predates it, or was built under different concerns. The ACL keeps the new model clean while the legacy system carries on unchanged underneath. The same shape applies more broadly to any boundary between bounded contexts whose models diverge, legacy or not.

In practice it's rarely a single class. A gateway owns the call to the external system, and a translator maps its data shapes and vocabulary onto your domain types. The anti-corruption layer is that combination, not either piece alone.

How it works

Variations

Gateway
The wire-level call is isolated in a gateway; a translator maps its data and vocabulary onto your domain types — the ACL is that combination, not one class.
Strangler Fig
A temporary ACL raised in front of a legacy system during a strangler migration, torn down once the new implementation fully replaces it.
One-way vs. two-way translation
Translate only inbound reads from the foreign system, or convert both directions — outbound calls into its terms and responses back into yours.
Per-context vs. shared integration layer
One ACL per external system it talks to, or a single shared translation layer serving several contexts — sharing risks becoming its own leaky, overloaded abstraction.

Trade-offs

Pros

  • Keeps your domain model pure — the ubiquitous language never absorbs a legacy or third-party vocabulary.
  • Confines the blast radius of change: an upstream schema or API shift touches the ACL, not domain code.
  • Makes the integration an explicit, testable seam you can mock or contract-test in isolation.
  • Lets a legacy system be migrated off, or a new one migrated onto, gradually — without every consumer feeling the seam.

Cons

  • Extra code and a translation step on every call across the boundary — real latency and maintenance cost.
  • Easy to under-build: a thin or partial ACL still lets foreign concepts leak through in edge cases.
  • Two models to hold in your head at once — yours, and the one you're being protected from.
  • A shared ACL serving many contexts can calcify into its own bottleneck and leaky abstraction.

When to use it

Reach for it when

  • You integrate with a legacy system, external service, or another bounded context whose model doesn't match yours.
  • You don't control the other side's model and refuse to let its concepts leak into your domain.
  • You're migrating off an old system gradually and need new code shielded from its shape in the meantime.

Avoid when

  • Both sides already share the same model and ubiquitous language — translation would be pure ceremony.
  • You control both systems and can align their models directly instead of maintaining a permanent seam.
  • The integration is small and one-off — a single Adapter at the call site is enough.

Code sketch

TypeScript — translating a legacy customer record 
// External system's shape — not ours to change
interface LegacyCustomerRecord {
  cust_id: string;
  full_nm: string;
  acct_status: "A" | "I" | "S";
}

// Our own domain model, our own vocabulary
interface Customer {
  id: string;
  name: string;
  status: "active" | "inactive" | "suspended";
}

const STATUS: Record<LegacyCustomerRecord["acct_status"], Customer["status"]> = {
  A: "active", I: "inactive", S: "suspended",
};

class CustomerAcl {
  constructor(private readonly legacy: LegacyClient) {}
  async getCustomer(id: string): Promise<Customer> {
    const r = await this.legacy.fetchRecord(id);
    return { id: r.cust_id, name: r.full_nm, status: STATUS[r.acct_status] };
  }
}
