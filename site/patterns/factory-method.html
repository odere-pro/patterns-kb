Factory Method

Declares a creation method on a base class but hands the choice of concrete type down to its subclasses — so the code that uses an object never names the class it actually gets.

GoF Creational

What it is

The Factory Method defines an interface for creating an object, but defers the actual instantiation to subclasses. A base class — the creator — contains the business logic and calls an overridable method to obtain the product it needs; each subclass answers that call with a different concrete class.

The force it resolves is a class that must create objects it cannot name. The moment you write new ConcreteThing(), the surrounding code is welded to that one type. Factory Method replaces the hard-coded constructor with a single overridable seam, so the decision of which class to build lives in exactly one place instead of scattered across the caller.

Because the creator works only against the abstract product type, it stays open to new products: add a subclass, override one method, and the existing logic runs unchanged against the new type. The knowledge of concrete classes is pushed to the edges of the hierarchy where it belongs.

How it works

Variations

Parameterized factory method
A single method takes a key or type token and switches on it to return one of several products — fewer subclasses, but a growing switch to maintain.
Default implementation
The creator's factory method returns a sensible default product; subclasses override only when they need something different, rather than always.
Static factory method
A named static method (of, valueOf, from) stands in for a constructor — an idiom that adds naming and caching, though it can't be overridden.
Registry-based factory
Concrete creators register themselves in a lookup table keyed by name, so new products plug in at runtime with no changes to the calling code.

Trade-offs

Pros

  • Decouples the caller from concrete product classes — it depends only on the interface.
  • Centralizes construction in one overridable method, honoring single responsibility.
  • New products arrive by adding a subclass, without editing the existing creator.
  • Gives subclasses a clean hook to customize exactly what gets built.

Cons

  • Can force a parallel Creator hierarchy just to vary one product — class explosion.
  • Subclassing to change a created type is heavyweight when composition would do.
  • The extra indirection obscures the flow when there's really only one product.
  • The base creator must still define and depend on the abstract product type.

When to use it

Reach for it when

  • A class can't anticipate the concrete type of the objects it must create.
  • You want subclasses to specify the objects that base-class logic operates on.
  • The knowledge of which concrete class to build belongs in one overridable spot.

Avoid when

  • There's a single concrete product and no foreseeable variation — just call the constructor.
  • Passing in the product or a factory (composition, injection) is simpler than subclassing.
  • The added class hierarchy costs more than the decoupling it buys.

Code sketch

TypeScript — subclasses pick the product 
interface Button {
  render(): string;
}

class HtmlButton implements Button {
  render() { return "<button>OK</button>"; }
}
class WindowsButton implements Button {
  render() { return "[ OK ]"; }
}

abstract class Dialog {
  // the factory method — no concrete type named here
  protected abstract createButton(): Button;

  render(): string {
    const button = this.createButton();   // defer to subclass
    return `dialog: ${button.render()}`;
  }
}

class WebDialog extends Dialog {
  protected createButton(): Button { return new HtmlButton(); }
}
class DesktopDialog extends Dialog {
  protected createButton(): Button { return new WindowsButton(); }
}
