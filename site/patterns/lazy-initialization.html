Lazy Initialization

Holds off building a value until something first asks for it — then creates it once, caches it, and hands back the same instance ever after.

GoF Also Essential

What it is

Lazy initialization defers the construction or computation of a value until the first moment it's actually needed. Instead of building everything up front, an accessor checks whether the value already exists; if not, it creates it, stores it, and returns it. Every access after that returns the cached instance.

The force it resolves is wasted, badly-timed work. Eager construction pays for objects that may never be used, drags out startup, and can fail before the object is even wanted. Some things are genuinely expensive — a connection pool, a parsed configuration, a large in-memory index — and some can't be built at construction time because their inputs aren't ready yet.

Lazy initialization trades that up-front cost for a small guard on each read: a one-time creation, deferred to the point of first use, then amortized across the object's lifetime. The cost doesn't vanish — it moves.

How it works

Variations

Lazy field / lazy getter
A nullable backing field with a check-then-create accessor — the simplest form, inlined right where the value is read.
Memoized holder
A reusable Lazy<T> wrapper that takes a factory, runs it at most once, and caches the outcome behind a clean get().
Thread-safe lazy (double-checked locking)
Guards the first-access race so two threads don't both build the value — a fast unlocked check followed by a locked one.
Virtual Proxy
A stand-in with the real object's interface that loads the heavy subject only when a method is actually called.
Lazy Singleton
The single shared instance is created on the first getInstance() rather than at class-load time.

Trade-offs

Pros

  • Faster startup — nothing is built for features a given run never touches.
  • Spreads expensive work across runtime instead of one up-front spike.
  • Lets a value wait until its dependencies actually exist.
  • A natural fit for costly or rarely-used resources.

Cons

  • Every read pays a small "is it built yet?" check.
  • First access carries a latency spike that eager init would have hidden.
  • Concurrent first access races unless you add locking.
  • Hidden state and side effects at read time make lifecycles and errors surface late.

When to use it

Reach for it when

  • Building the value is expensive and it may not be needed on every run.
  • Its inputs aren't available at construction time.
  • You want a fast startup and can accept the cost on first use.

Avoid when

  • The value is always needed — eager init is simpler and more predictable.
  • Construction is cheap; the guard costs more than it ever saves.
  • You'd rather fail fast at startup than discover the failure on first access.

Code sketch

TypeScript — a reusable lazy holder 
class Lazy<T> {
  private value: T | undefined;
  private computed = false;

  constructor(private readonly factory: () => T) {}

  get(): T {
    if (!this.computed) {
      this.value = this.factory();  // run the factory once
      this.computed = true;         // and cache the result
    }
    return this.value as T;
  }
}

// The pool is never built if get() is never called.
const pool = new Lazy(() => new ConnectionPool({ size: 20 }));

function query(sql: string) {
  return pool.get().run(sql);  // created on first query, reused after
}
