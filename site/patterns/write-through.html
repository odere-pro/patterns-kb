Lens · Caching

Write-Through

Every write updates the cache and the backing store together, in the same synchronous call, so a read that follows a write can never see a stale value.

Caching Lens

What it is

A write-through cache sits directly in the write path. When the application writes a value, the cache writes it to the backing store first, waits for that write to be durably accepted, and only then updates its own entry and reports success to the caller. From the caller's point of view, a write isn't done until the cache and the store agree.

The force it resolves is divergence. In a cache-aside setup, the application writes to the store and separately pokes the cache — invalidate it, or update it — as two distinct steps. A crash or race between those steps leaves the cache holding a value the store has already moved past. Write-through collapses the two into one call, so cache and store can never disagree in the first place.

The cost is paid on every write: the caller now waits for the store's full round trip plus the cache update, and a key that's written once and never read again still occupies cache space for no benefit. That tension — consistency now vs. write latency — is exactly what its alternative, Write-Behind, trades away in the other direction.

How it works

Variations

Store-then-cache vs. cache-then-store ordering
Persist to the store first and only then update the cache, or update the cache first and write through behind it. Store-first never lets a value that failed to persist sit in the cache.
Paired with Read-Through
Misses on the read side fault through the same cache into the store, so both directions treat the cache as the single authoritative front door.
Selective write-through (write-around for cold keys)
Skip caching keys that are unlikely to be read again and let a later read fault them in — avoids spending the cache-write cost on data that's never reused.
Replicated / distributed write-through
In a clustered cache, the node taking the write must also propagate it (or an invalidation) to peer nodes before acking — more write latency in exchange for cluster-wide consistency.

Trade-offs

Pros

  • Cache and store never diverge — a read right after a write always sees the fresh value.
  • Removes the cold-read penalty: data is warm in the cache the instant it's written.
  • Simplifies the read path — reads can trust the cache with no invalidate-then-fetch races.
  • A failed persist is caught synchronously, at the write call, not discovered later.

Cons

  • Every write pays the store's full latency plus the cache update — never faster than the slower of the two.
  • Write-heavy, rarely-read keys still get cached, wasting space on data nobody reads back.
  • Cache and store must both succeed, or the write needs careful rollback or retry handling.
  • Does nothing for write throughput — for that, look at Write-Behind instead.

When to use it

Reach for it when

  • Reads vastly outnumber writes and stale data right after a write is unacceptable.
  • A cache miss is expensive — a slow recompute or a cold store hit — so keeping the cache warm on write matters.
  • You need read-your-writes consistency without bolting on separate invalidation logic.

Avoid when

  • Writes are frequent or bursty and write latency has to stay low — see Write-Behind.
  • Most written keys are rarely or never read back — caching them on write is wasted work.
  • The backing store already serves fast, consistent reads on its own — caching adds no value.

Code sketch

TypeScript — a minimal write-through cache 
interface Store<K, V> {
  read(key: K): Promise<V | undefined>;
  write(key: K, value: V): Promise<void>;
}

class WriteThroughCache<K, V> {
  private cache = new Map<K, V>();

  constructor(private readonly store: Store<K, V>) {}

  async get(key: K): Promise<V | undefined> {
    if (this.cache.has(key)) return this.cache.get(key);
    const value = await this.store.read(key);
    if (value !== undefined) this.cache.set(key, value);
    return value;
  }

  async set(key: K, value: V): Promise<void> {
    await this.store.write(key, value); // durable first
    this.cache.set(key, value);         // only cache once persisted
  }
}
