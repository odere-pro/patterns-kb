Lens · Domain-Driven Design

Aggregate

A cluster of associated entities and value objects, bound behind a single root and treated as one unit whenever data changes, so invariants that span the cluster are never left half-enforced.

DDD Lens

What it is

An aggregate is a cluster of associated domain objects — one or more entities and value objects — that the model treats as a single unit whenever its data changes. Exactly one entity inside the cluster is the aggregate root: it's the only member that code outside the boundary is allowed to hold a reference to, and every command that touches the cluster arrives through it.

The force it resolves is an invariant that spans more than one object. An order's total must equal the sum of its line items; an account's balance can never go negative once a withdrawal lands. If any code can reach into the line items or the ledger entries directly, no single piece of code owns keeping those rules true, and some path that touches the parts without touching the whole will eventually violate them. The aggregate closes that gap by hiding everything except the root — internal entities and value objects are only ever reached and mutated through methods the root exposes, and the root refuses any change that would break its own invariants.

The boundary doubles as the unit of transaction and concurrency. Loading, saving, and locking happen one aggregate at a time: a repository fetches the whole cluster, the root's methods apply a change and check invariants in memory, and the cluster is written back atomically, usually guarded by an optimistic version on the root. Consistency between separate aggregates is deliberately left eventual, closed later by domain events or a process manager, because holding several aggregates' transactions open at once to keep them in lockstep doesn't scale.

The design fails in either direction. Draw the boundary too small and invariants leak back out into services that reach across several roots by hand; draw it too large and unrelated data gets dragged into every transaction, turning each write into a lock that competes with everything else touching that root.

How it works

Variations

Reference by identity
An aggregate holds other aggregates by ID, never by object reference, so each cluster stays small and independently transactional.
Domain events on commit
The root records events internally as it mutates; they're published only after the transaction that saved it has actually succeeded.
Event-sourced aggregate
State is derived by folding a stored stream of past events rather than loaded as a snapshot; a command only ever produces new events, never a direct field write.
Aggregate factory
Multi-step construction — assembling an Order from a cart, say — is delegated to a dedicated factory, so the root's constructor stays a simple, valid-by-construction call.

Trade-offs

Pros

  • Invariants spanning multiple objects are enforced in one place, not scattered across services.
  • A clear transaction boundary — one aggregate is one atomic load and save, no partial updates.
  • Encapsulation: internal entities are hidden, so their rules can't be bypassed from outside.
  • Concurrency control stays tractable — version or lock the root, not a web of related tables.

Cons

  • An aggregate that grows too large drags unrelated data into every transaction and lock.
  • Cross-aggregate consistency becomes eventual, which needs real machinery — events, sagas — to close.
  • Choosing the right boundary is genuinely hard, and a wrong one is expensive to redraw later.
  • Loading a large aggregate can pull in far more data than a single command actually needs.

When to use it

Reach for it when

  • Several objects share an invariant that must never be violated, like a total matching its parts.
  • You need a clear transaction boundary for saving and loading a related cluster of data.
  • The domain has a natural grouping that only makes sense together, like an order and its line items.

Avoid when

  • The objects share no real invariant — plain Entities or Value Objects are enough on their own.
  • You're tempted to widen the boundary "just in case" — that's exactly how aggregates balloon and lock contention appears.
  • The true consistency requirement is eventual, not immediate — reference the other side by ID and let a Domain Event synchronize it later.

Prevents the smell of an Anemic Domain Model, where entities are reduced to data bags and the invariants meant to live on the aggregate root leak out into service-layer code instead.

Code sketch

TypeScript — an Order aggregate root 
class Order { // aggregate root
  private readonly items: LineItem[] = [];
  private status: "draft" | "placed" = "draft";

  addItem(sku: string, quantity: number, unitPrice: Money): void {
    if (this.status !== "draft") throw new Error("order already placed");
    if (quantity <= 0) throw new Error("quantity must be positive");
    this.items.push(new LineItem(sku, quantity, unitPrice));
  }

  place(): void {
    if (this.items.length === 0) throw new Error("cannot place an empty order");
    this.status = "placed";
  }

  get total(): Money {
    return this.items.reduce((sum, i) => sum.add(i.subtotal), Money.zero());
  }
}

// LineItem is reachable only through Order — never constructed
// or mutated directly by code outside the aggregate.
class LineItem {
  constructor(
    readonly sku: string,
    readonly quantity: number,
    readonly unitPrice: Money,
  ) {}
  get subtotal(): Money { return this.unitPrice.times(this.quantity); }
}
