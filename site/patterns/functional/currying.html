Currying

Turns a function of several arguments into a chain of one-argument functions, so pre-filling the early ones hands you back a function that's just waiting for the rest.

Functional Lens

What it is

Currying transforms a function of several arguments into a sequence of functions that each take exactly one, so f(a, b, c) becomes f(a)(b)(c). Calling it with fewer than all its arguments doesn't fail or block — it returns a new function closed over whatever was supplied so far, waiting for the rest.

The force it resolves is argument order versus argument availability. Some arguments — a config, a connection, a conversion rate — are known early and rarely change; others — the record being processed, the event just received — only arrive later, one at a time, inside a loop or a callback. Currying lets you supply the early, stable arguments once and hand the resulting function to whatever eventually produces the rest, instead of re-passing the stable ones on every call or writing an ad-hoc closure by hand each time.

It differs from partial application in mechanism, not intent. Partial application binds some arguments of an n-ary function and returns a function over the remainder in one step; a curried function is unary all the way down, and partial application falls out of it for free — you just stop calling before the last argument. Neither is built into JavaScript or TypeScript; both are conveniences layered on ordinary closures.

Because every curried function takes one value and returns one value, curried functions compose like any unary function with no arity to reconcile — exactly the shape a composition pipeline needs.

How it works

Variations

Manual currying
Write the nested closures by hand: a => b => c => …. No dependency, full clarity, but tedious past two or three arguments.
Auto-currying
A helper like curry(fn) inspects fn.length and returns a function that keeps collecting arguments until it has enough, then invokes the original in one shot.
Curry with placeholders
Accept a placeholder token (Ramda's R.__) so a caller can fix a later argument first and leave an earlier one open, instead of currying strictly left to right.
Data-last argument order
Functional libraries put the data being transformed last, so map(fn) curries the transformer and leaves a unary function ready to receive the collection inside a pipeline.
Uncurrying
The inverse transform: collapse a chain of unary functions back into one call taking all the arguments at once, useful when interoperating with ordinary n-ary APIs.

Trade-offs

Pros

  • Specializes a general function into a reusable one by fixing arguments once, not on every call.
  • Every curried function is unary, so it composes directly with other unary functions.
  • Supports point-free style — the data argument stays implicit until the final call.
  • Plays well with lazy pipelines: intermediate calls just build closures, nothing runs until fully applied.

Cons

  • Argument order becomes load-bearing — reordering a curried function's parameters breaks every partial application of it.
  • Deep curry chains are harder to read and step through than one call with named arguments.
  • Type inference across a long curried chain can be awkward in TypeScript without explicit overloads or a curry-aware helper type.
  • A stack of returned closures is less legible in a debugger than a single stack frame.

When to use it

Reach for it when

  • You repeatedly call the same function with the same first few arguments and vary only the last one.
  • You're building a composition pipeline and need every step to be a unary function.
  • You want to derive specialized functions — a logger bound to one prefix, a validator bound to one schema — from one general one.

Avoid when

  • The function only has one call site with all its arguments at hand — currying adds indirection with no reuse to show for it.
  • Argument order in your domain is unstable, or arguments are optional and named — a single object parameter serves better.
  • Curried style would slow down code review more than the reuse is worth for a team unfamiliar with it.

Code sketch

TypeScript — a small curry helper 
function curry<A, B, C, R>(
  fn: (a: A, b: B, c: C) => R,
): (a: A) => (b: B) => (c: C) => R {
  return (a: A) => (b: B) => (c: C) => fn(a, b, c);
}

function add(a: number, b: number, c: number): number {
  return a + b + c;
}

const curriedAdd = curry(add);
const addFive = curriedAdd(5);       // fix the first argument
const addFiveAndTwo = addFive(2);    // fix the second
addFiveAndTwo(3);                    // 10 — runs on the last call

// A pipeline step: bind the config, leave the record for later
const withTax = (rate: number) => (price: number) => price * (1 + rate);
const addVat = withTax(0.2);
[10, 20, 30].map(addVat);            // [12, 24, 36]

In the wild

HaskellEvery function is curried by default: the type a -> b -> c is really a -> (b -> c), so applying fewer than all arguments yields a partially applied function with no helper, and operator sections like (+1) fall out of the same rule.
RamdaEvery function ships auto-curried and data-last, so calling with fewer arguments returns a specialized function ready for a pipeline; the R.__ placeholder lets you fix a later argument and leave an earlier one open.
Lodash curry_.curry(fn) reads fn.length to decide when it has collected enough arguments across calls before invoking; the _ placeholder holds a position open, and _.curryRight applies arguments from the right instead of the left.
