Application

DTO

Carries data across a process or network boundary as a flat bag of fields — no behavior, no domain rules baked in, just the shape the far side agreed to.

Enterprise

What it is

A DTO (Data Transfer Object) is a plain, flat object whose only job is to carry data across a boundary — a network call, a process boundary, a serialization format, a UI tier. It carries at most getters and setters, never business rules, validation, or identity. Its shape is dictated entirely by what the boundary needs to send or receive, not by how the domain models the same concept internally.

The force it resolves is the cost of crossing that boundary. Remote calls are expensive per round trip, so a chatty interface that fetches an object's fields one getter at a time drowns in latency. A DTO batches everything a call needs into one coarse-grained object sent in a single trip. It also resolves a coupling problem: domain objects carry lazy-loading proxies, circular references, private fields, and behavior that has no business crossing a wire or reaching a client — the DTO exposes only what the contract promises and nothing else.

Fowler coined the term for exactly this in Patterns of Enterprise Application Architecture, aimed at genuinely remote calls between tiers. The idea outlived that narrow reading: today any API request or response body, any message payload, any shape crossing a serialization boundary is a DTO in the same sense, whether or not the call is literally remote.

How it works

Variations

Request DTO / Response DTO pair
Separate shapes for what a call accepts and what it returns, instead of one shared type — keeps write-only fields out of the read shape and vice versa.
Assembler (or Mapper)
A dedicated function or class that converts between domain object and DTO in one place, so mapping logic doesn't spread across controllers or leak into either shape.
Aggregate / coarse-grained DTO
Bundles several related entities — an order plus its line items and customer — into one shape, answering a whole screen or call in a single round trip.
Versioned DTO
The wire shape is versioned independently of the domain model (UserDtoV2), so an API keeps old clients working while the internal model moves on.
Immutable / record DTO
Fields are readonly and set once at construction — a plain record with no setters, matching how most serialization formats treat the data anyway.

Trade-offs

Pros

  • Decouples the wire or API shape from the internal domain model — either can change without breaking the other.
  • Batches data into one coarse-grained call, cutting chatty round trips over the network.
  • Plain data with no behavior serializes trivially and is easy to version, cache, and log.
  • Gives a boundary an explicit, reviewable contract instead of leaking internal objects across it.

Cons

  • Duplicates fields already on the domain model, and the mapping code is often repetitive boilerplate.
  • Two shapes to keep in sync — every model change risks becoming a silent DTO mismatch.
  • Business logic or validation that creeps into a DTO turns it into an anemic clone of the domain model.
  • Used across a call with no real boundary, it's pure mapping ceremony with nothing to justify it.

When to use it

Reach for it when

  • Data crosses a real boundary — process, network, service, or UI tier — and the two sides shouldn't share one internal type.
  • You're crossing a boundary drawn by a Service Layer and need an explicit shape for what goes in and out.
  • A single call should return exactly what the caller needs, not a partial or over-fetched domain graph.

Avoid when

  • Everything runs in one process with no boundary to protect — pass the domain object directly.
  • You need a small domain concept with real equality and behavior, not just a shipping container — that's a Value Object.
  • The "DTO" would carry business rules or validation — that behavior belongs on the domain side, not the transfer shape.

Code sketch

TypeScript — an assembler mapping domain to DTO 
// Domain model — has behavior, holds internal-only fields
class User {
  constructor(
    public readonly id: string,
    private passwordHash: string,
    public email: string,
    public roles: string[],
  ) {}
  hasRole(role: string): boolean { return this.roles.includes(role); }
}

// DTO — flat, no behavior, only what the API contract promises
interface UserDTO {
  id: string;
  email: string;
  roles: string[];
}

// Assembler: the only place that knows both shapes
function toUserDTO(user: User): UserDTO {
  return { id: user.id, email: user.email, roles: user.roles };
}

// Service boundary returns the DTO, never the domain object
app.get("/users/:id", async (req, res) => {
  const user = await users.findById(req.params.id);
  res.json(toUserDTO(user)); // passwordHash never leaves the process
});
