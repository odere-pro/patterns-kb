Lens · Caching

Read-Through

On a cache miss, the cache reaches into the backing store itself, loads the value, populates the entry, and hands it back — the application only ever calls get() and never sees the gap underneath.

Caching Lens

What it is

A read-through cache sits directly in front of a backing store and takes on the job of loading data, not just holding it. The application calls a single get(key); if the entry is present the cache returns it immediately, and if it's absent the cache itself invokes a loader bound to the backing store, populates the entry, and returns the result. The caller never branches on hit or miss — from where it stands, the cache simply always has the value.

The force it resolves is duplicated, drifting miss-handling logic. Against a plain cache, every caller that wants a value runs its own dance: check the cache, fall through to the database on a miss, write the result back, then return it — repeated at every call site, and prone to drifting apart across them (different TTLs, someone forgets to populate, two callers race and load twice). Read-through centralizes that logic once, inside the cache itself, so every caller gets identical, correct miss handling for free.

Because the cache owns the load path, it can also protect the backing store: concurrent misses for the same key can collapse into a single in-flight load instead of stampeding the database, and a key known to be absent can be cached negatively so repeated lookups stop hitting the store. The trade is coupling — the cache has to know how to load, wired to a loader function or a co-located data source, which is a tighter arrangement than a cache that's merely a dumb key-value store.

How it works

Variations

Synchronous load
The cache blocks the caller until the loader returns. Simplest to reason about, but every concurrent caller on a miss waits for the same round trip.
Request coalescing / single-flight
Concurrent misses for the same key share one in-flight load instead of each triggering its own call — the first caller loads, the rest await that same promise.
Negative caching
Cache a short-lived "not found" marker for missing keys, so a flood of lookups for absent data doesn't repeatedly round-trip the store.
Refresh-Ahead
Instead of waiting for a miss, proactively reload a hot key shortly before its entry expires, so the read-through load path stays cold for busy keys.
Multi-level read-through
A local in-process cache is itself read-through to a shared remote cache, which is read-through to the store — each tier only loads from the one behind it.

Trade-offs

Pros

  • Application code just calls get() — no miss-handling branch scattered across callers.
  • Loading and population logic lives in one place, so TTLs, serialization, and coalescing stay consistent for everyone.
  • Can collapse concurrent misses into a single backing call, shielding the store from thundering herds.
  • Callers never talk to the store directly, so the store's shape can change behind the loader.

Cons

  • Needs a cache that supports a bound loader function — harder to bolt onto an existing dumb cache.
  • A cold cache or an evicted hot key still pays a synchronous load penalty on the next read.
  • Ties the cache to one backing store's shape, which is awkward when different callers want different fallback logic.
  • Solves reads only — writes still need a paired strategy or the cache goes stale silently.

When to use it

Reach for it when

  • Multiple services or call sites currently duplicate their own check-cache-else-load-else-populate logic.
  • Callers should be shielded entirely from cache misses and the loading mechanics behind them.
  • The backing store benefits from load coalescing under bursts of concurrent reads for the same key.

Avoid when

  • Different callers need genuinely different loading or fallback logic for the same key — Cache-Aside keeps that control in application code.
  • Writes are frequent and must be reflected immediately — pair with Write-Through rather than leaving reads stale.
  • You don't control the cache client closely enough to wire in a loader function.

Code sketch

TypeScript — a coalescing read-through cache 
class ReadThroughCache<K, V> {
  private entries = new Map<K, V>();
  private inFlight = new Map<K, Promise<V>>();

  constructor(
    private readonly loader: (key: K) => Promise<V>,
    private readonly ttlMs = 30_000,
  ) {}

  async get(key: K): Promise<V> {
    const hit = this.entries.get(key);
    if (hit !== undefined) return hit;          // no store call

    const pending = this.inFlight.get(key);
    if (pending) return pending;                 // coalesce concurrent misses

    const load = this.loader(key).then((value) => {
      this.entries.set(key, value);
      this.inFlight.delete(key);
      setTimeout(() => this.entries.delete(key), this.ttlMs);
      return value;
    });
    this.inFlight.set(key, load);
    return load;
  }
}

// Caller never branches on hit vs. miss
const users = new ReadThroughCache((id: string) => db.findUser(id));
const u = await users.get("42");
