Replication

Keeps copies of the same data on multiple machines, so losing one node never means losing the data, and reads can be served from whichever copy is closest or least busy.

Distributed Coordination & Data

What it is

Replication keeps more than one copy of the same data on different nodes — different disks, hosts, racks, or regions — and keeps every copy up to date as writes happen. In the common single-leader form, one node accepts all writes, appends them to an ordered log, and ships that log to one or more followers, which apply the entries in the same order and stand ready to serve reads or step up if the leader dies.

The first force it resolves is durability: a single copy on a single node is a single point of failure — a bad disk, a dead host, or an outage in one datacenter loses the data outright, or at least loses access to it. Spreading identical copies across independent failure domains means one of those failures no longer means data loss.

The second is read scale and locality: one machine's disk and CPU cap how many reads it can serve and how fast. Replicas let reads fan out across many machines, and let you place a copy close to each population of readers so requests don't cross a continent to get an answer.

The catch is that copies living on separate machines can't update atomically across a network. The moment there's more than one copy, something has to decide how strongly — and how fast — they must agree, and that's where replication runs straight into the CAP trade-off: wait for every replica and you lose availability under a partition; don't wait, and a reader can see stale or conflicting data.

How it works

Variations

Single-leader (primary-replica)
One node accepts every write and ships an ordered log to followers. Simple to reason about; followers can lag behind, and a leader failure needs a handover.
Multi-leader
Several nodes accept writes independently and replicate to each other — useful across datacenters, but concurrent writes to the same key can conflict and need reconciling.
Leaderless (quorum-based)
Any replica accepts reads and writes; overlapping read and write quorums are what force the copies to converge instead of a single leader ordering them.
Synchronous vs. asynchronous
Synchronous replication waits for a replica's acknowledgment before confirming a write — safer, slower, less available under a partition. Asynchronous confirms immediately and risks losing the most recent writes on failover.

Trade-offs

Pros

  • Survives node, disk, or availability-zone failure without losing data or access to it.
  • Scales read traffic horizontally by fanning reads out across replicas.
  • Lets you place a copy near each population of readers, cutting latency.
  • A live standby can take over immediately if the leader fails.

Cons

  • Replicas can lag, so a read may return data that's already stale.
  • More copies mean more storage and steady network cost to keep them in sync.
  • Failover has to elect a new leader and reconcile any writes that never fully replicated.
  • Doesn't help write throughput — writes still funnel through however many leaders you allow.

When to use it

Reach for it when

  • Losing a single node must never mean losing the data.
  • Read traffic outstrips what one node's disk and CPU can serve.
  • Readers are geographically spread and latency to a single copy matters.

Avoid when

  • The dataset itself exceeds what any single node can hold — that's a job for Sharding, not more copies.
  • Every reader must see the absolute latest write and can't tolerate any lag — commit to synchronous replication or a stronger consistency pattern deliberately, not by default.
  • A single instance with regular backups already meets your durability bar — replication adds operational cost you don't need yet.

Code sketch

TypeScript — a primary shipping a log to replicas 
interface LogEntry {
  seq: number;
  key: string;
  value: string;
}

class Primary {
  private log: LogEntry[] = [];
  private replicas: Replica[] = [];

  write(key: string, value: string): void {
    const entry: LogEntry = { seq: this.log.length, key, value };
    this.log.push(entry);
    for (const r of this.replicas) r.apply(entry); // ship in commit order
  }

  attach(replica: Replica): void {
    for (const entry of this.log) replica.apply(entry); // catch up first
    this.replicas.push(replica);
  }
}

class Replica {
  private store = new Map<string, string>();
  private lastSeq = -1;

  apply(entry: LogEntry): void {
    if (entry.seq !== this.lastSeq + 1) throw new Error("gap in log");
    this.store.set(entry.key, entry.value);
    this.lastSeq = entry.seq;
  }

  read(key: string): string | undefined {
    return this.store.get(key); // may lag the primary slightly
  }
}
