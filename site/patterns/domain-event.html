Domain Event

Captures something that indisputably happened in the domain — named in the past tense, immutable the moment it's raised, and broadcast to whichever aggregates, services, or bounded contexts care that it occurred.

DDD Lens

What it is

A domain event is a fact, not a request: something that already happened in the business, named in the past tense with the words domain experts actually use — OrderShipped, PaymentDeclined, SeatReserved. Once raised it's immutable; you don't edit history, you only append to it. It carries whatever data makes the occurrence meaningful on its own, not just a bare signal that something changed.

The force it resolves is a coupling problem inside the Aggregate that owns the change. The aggregate enforces the invariant and knows precisely when something worth telling the rest of the system just happened — but if it calls the logger, the cache, and every other interested context directly, its own transaction boundary swells to include all of them, and adding one more consumer means editing the aggregate again. A domain event lets the aggregate simply record what happened and hand the list to infrastructure, which dispatches it once the transaction commits. The aggregate never learns who, if anyone, is listening.

Scope matters: inside its own bounded context a domain event can be nothing more than a plain in-process value object. It only earns the heavier name integration event, with a stable, versioned public shape, once it crosses into another team's context.

How it works

Variations

Notification event (thin)
Carries only an id and an event type; subscribers call back to fetch whatever details they need. Small and cheap to version, at the cost of an extra round trip.
Event-carried state transfer (fat)
Carries the full data a subscriber needs to act, so it never has to call back the source at all — decoupled at runtime, but the payload duplicates state and now has its own schema to maintain.
Event Sourcing
Stop treating the event as a side notification and make it the system of record — the aggregate's current state is only ever a fold over its own event history.
Domain event vs. integration event
The same fact wears two hats: a lightweight in-process object while it stays inside one bounded context, a stable and explicitly versioned contract once another team's context needs to consume it.

Trade-offs

Pros

  • Names the fact that occurred, so behavior is modeled around what happened, not requests to make it happen.
  • Decouples the aggregate that raises the event from every consumer that reacts to it.
  • New reactions plug in as new subscribers — the code that raised the event never changes.
  • A persisted stream of them doubles as an audit trail of what happened and when.

Cons

  • Easy to leak persistence-model shape into the event, coupling subscribers to internals.
  • Delivery and ordering guarantees become the caller's problem the moment events leave the process.
  • Consumers see the fact after the fact — real eventual consistency, never the original transaction.
  • A published event's shape is a contract forever; schema evolution has to be versioned, not edited.

When to use it

Reach for it when

  • An aggregate's state change should trigger side effects that aggregate shouldn't know about.
  • Multiple parts of the system, or multiple bounded contexts, need to react to the same occurrence.
  • You want a durable, replayable record of what happened, not just the current state.

Avoid when

  • The reaction is purely local and synchronous — a direct method call is simpler and easier to trace.
  • The side effect must stay inside the same transaction as the change — model it there instead.
  • The "event" is really just an internal implementation detail nobody outside needs to know about.

Code sketch

TypeScript — an aggregate recording an event 
interface DomainEvent {
  readonly type: string;
  readonly occurredAt: Date;
}

class OrderShipped implements DomainEvent {
  readonly type = "OrderShipped";
  readonly occurredAt = new Date();
  constructor(
    readonly orderId: string,
    readonly trackingCode: string,
  ) {}
}

class Order {
  private events: DomainEvent[] = [];
  private status: "pending" | "shipped" = "pending";

  constructor(private readonly id: string) {}

  ship(trackingCode: string): void {
    if (this.status === "shipped") return; // no double-shipping
    this.status = "shipped";
    this.events.push(new OrderShipped(this.id, trackingCode));
  }

  // Infrastructure calls this after the commit succeeds, never before.
  pullEvents(): DomainEvent[] {
    const pending = this.events;
    this.events = [];
    return pending;
  }
}
