In-Process Cache

Keeps cached data inside the application process's own memory, so a hit needs no network hop at all — the fastest cache there is — at the cost of each instance holding its own, potentially divergent, copy.

Caching

What it is

An in-process cache keeps cached values in the application's own memory — a map, or a small library-managed cache, living inside the same process that serves requests. A hit is a plain memory lookup: no network call, no serialization, just a reference already sitting in the same address space. It's the fastest kind of cache by a wide margin, because it removes the one cost even a distributed cache can't avoid — the round trip to reach it.

The force it resolves is latency on the hottest, smallest data. Modern app servers run on machines with plenty of memory, and some data — configuration, feature flags, a small lookup table — is read on nearly every request and changes rarely. Paying a network hop to a shared cache for that on every request is pure overhead. The catch is the flip side of "in the process": each instance has its own copy, so caches on different instances can diverge, and the same data is duplicated across the fleet. That makes it a targeted optimization, not the default — and a natural front layer (an L1) in front of a shared cache for the very hottest keys.

How it works

Variations

Bounded with eviction
A size cap plus an eviction policy (usually LRU) keeps the cache from growing without bound and consuming the process's heap — the common form for anything but a tiny fixed table.
TTL-based expiry
Each entry expires after a set time so the local copy can't drift too far from the source; the shorter the TTL, the narrower the cross-instance inconsistency window.
Near cache (L1 in front of L2)
The in-process cache fronts a shared distributed cache: it absorbs the hottest reads locally and defers everything else to the shared tier, cutting network hops for the keys that matter most.
Invalidation by broadcast
To fight divergence, a write publishes an invalidation (e.g. over a pub/sub channel) that every instance listens for and applies to its local copy — buying tighter consistency at the cost of more moving parts.

Trade-offs

Pros

  • The fastest cache possible: a hit is an in-memory lookup with no network hop and no serialization.
  • No extra infrastructure — it's a library or a plain map, with nothing new to deploy or operate.
  • No shared-cache dependency in the read path, so it keeps working even if the shared tier is down.
  • An effective L1 in front of a shared cache, soaking up the hottest keys before they leave the process.

Cons

  • Each instance has its own copy, so instances can serve different values for the same key.
  • The same data is duplicated across every instance, wasting memory at scale.
  • It starts cold on every restart or newly-scaled instance, so a deploy re-warms from scratch.
  • Invalidating a change across all instances is genuinely hard — there's no single copy to update.

When to use it

Reach for it when

  • Small, hot, read-mostly data — config, feature flags, a lookup table — is needed on nearly every request.
  • You need the lowest latency possible and can't spare even a shared-cache network hop.
  • You want an L1 in front of a distributed cache to shield the hottest keys from the network.

Avoid when

  • The data changes often and must read consistently across all instances — divergence will bite.
  • The working set is large; duplicating it in every instance wastes memory a shared cache would pool.
  • You run many instances and the memory duplication outweighs the latency you'd save.

Code sketch

TypeScript — a bounded-by-TTL cache living in the process's own memory 
// A plain Map in the app's own memory — no network hop on a hit.
// Each instance holds its own copy, so entries can diverge across the fleet.
type Entry<V> = { value: V; expiresAt: number };
const store = new Map<string, Entry<unknown>>();

function get<V>(key: string): V | undefined {
  const e = store.get(key);
  if (!e) return undefined;
  if (Date.now() > e.expiresAt) { store.delete(key); return undefined; }  // TTL expiry
  return e.value as V;
}

function set<V>(key: string, value: V, ttlMs: number): void {
  store.set(key, { value, expiresAt: Date.now() + ttlMs });
}
