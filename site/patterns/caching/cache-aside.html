Lens · Caching

Cache-Aside

On every read the application checks the cache first — a hit returns straight away, and only a miss sends it to the source of truth, whose value it then writes back into the cache for next time.

Caching Lens

What it is

Cache-Aside — also called lazy loading — puts the application in charge of both reading the cache and filling it. On a read, the app asks the cache for the key. A hit returns the value immediately, never touching the source of truth. A miss sends the app to the database (or service, or file) to fetch the value, which it then writes into the cache before handing it back to the caller. On a write, the app updates the source of truth directly and either deletes the now-stale cache entry or overwrites it — the cache is never written to on its own initiative.

The force it resolves is a mismatch between what the application knows and what a generic cache knows. A key-value store like Redis or Memcached has no idea what a "user" or an "order" is, how to load one, or when its copy has gone stale — it only knows keys and values. Cache-Aside keeps that ignorance intentional: the cache stays a dumb, general-purpose store, and all the domain logic — what to cache, how to load it, when to invalidate it — lives in application code, where it's visible and can differ per call site.

That division also makes the cache optional in the failure sense. If it's down or cold, the app falls straight through to the source and keeps serving, just slower — there's no hard dependency on the cache being correct or even present. And because entries are only created on demand, the cache holds exactly the working set that's actually being requested, rather than being pre-warmed with data no one reads.

How it works

Variations

Read-Through
The cache library itself loads on a miss instead of the app doing it — same effect, opposite owner of the fetch logic.
Write-invalidate on update
A write deletes the cache key rather than updating it, so the next read repopulates from the source of truth instead of risking a bad overwrite.
TTL / lazy expiration
Entries expire after a fixed time-to-live in addition to (or instead of) explicit invalidation, bounding staleness cheaply when writes are hard to track.
Negative caching
Cache a "not found" marker too, so repeated lookups for a key that doesn't exist don't hit the database on every single request.

Trade-offs

Pros

  • The cache holds only data that's actually been requested, so it stays small and relevant.
  • A generic key-value cache needs zero knowledge of the backing store or domain model.
  • A cache outage degrades gracefully — reads just fall through to the source, slower but correct.
  • The read path is simple to reason about: check, miss, load, populate, return.

Cons

  • Every call site that reads must repeat the check-miss-load-populate logic, or share a helper that does.
  • The first reader after an eviction pays the full latency of the source — the "cache penalty."
  • There's a stale window between a write to the source and the cache entry being invalidated.
  • No built-in protection against a hot key expiring under load and a herd of misses hitting the source at once.

When to use it

Reach for it when

  • Reads vastly outnumber writes and the same keys get requested repeatedly.
  • Access is sparse or unpredictable, so pre-warming or caching everything would waste memory.
  • You're using a generic cache that has no knowledge of your data model or loading logic.

Avoid when

  • Data changes so often the hit rate would stay low anyway — caching adds complexity for little gain.
  • Every read needs a guarantee of the freshest value — the cache can be stale between a write and its invalidation.
  • The loading and invalidation logic belongs with the storage layer itself — reach for Read-Through instead.

Code sketch

TypeScript — a cache-aside read and write path 
async function getUser(id: string): Promise<User> {
  const cached = await cache.get<User>(`user:${id}`);
  if (cached !== null) return cached; // hit — database untouched

  const user = await db.query<User>(
    "SELECT * FROM users WHERE id = $1", [id],
  );
  if (user === null) {
    await cache.set(`user:${id}`, NOT_FOUND, { ttlSeconds: 30 });
    throw new NotFoundError(id);
  }

  await cache.set(`user:${id}`, user, { ttlSeconds: 300 });
  return user;
}

async function updateUser(id: string, patch: Partial<User>): Promise<void> {
  await db.update("users", id, patch);
  await cache.delete(`user:${id}`); // invalidate, next read repopulates
}

In the wild

MemcachedA deliberately dumb key-value store with no loader concept at all, so the application must check it, fall through to the source, and populate it by hand.
RedisThe default read-cache in front of a relational database, where application code owns the GET, the fallback query, the SET, and the DEL on write.
