Lens · Caching

Write-Behind

Writes hit the cache and return immediately, while the matching update to the backing store is queued and flushed asynchronously — the store catches up on its own schedule, not the caller's.

Caching Lens

What it is

A write-behind cache (also called write-back) accepts a write, updates its own copy, and returns to the caller right away. The matching update to the backing store isn't made inline — it's queued and pushed later, on a timer, once a batch fills up, or when the entry is evicted. Until that flush happens, the cache — not the database — holds the most current copy of the data.

The force it resolves is write latency versus durability. A write-through cache is safe: every write lands in the store before it's acknowledged, so nothing is ever lost. But every write also pays the store's full latency, and under heavy write volume the store becomes the bottleneck long before the cache does. Write-behind decouples the two — the caller is freed the instant the cache is updated, and the store catches up out of band.

That deferral pays for itself when writes are bursty or repetitive: a flood of writes smooths into a steady trickle to the store, and several writes to the same key before a flush collapse into one write of the final value. The trade is a window of vulnerability — data that exists only in the cache until flushed is data a crash can erase, so write-behind only earns its keep when that loss is tolerable, or the cache itself is made durable: persisted to disk, replicated, or backed by a write-ahead log.

How it works

Variations

Time-based flush
Drain the queue every N milliseconds regardless of size. Simple, and it bounds the staleness window to a known maximum.
Size- or count-based flush
Flush once the dirty set reaches a threshold, so the store sees fewer, larger, more efficient writes under sustained load.
Write coalescing
Multiple writes to the same key before a flush collapse into one — the store only ever sees the final value, never the intermediates.
Durable write queue
Back the pending-writes queue with disk or replication so a cache crash doesn't silently drop unflushed data — trades some latency back for safety.

Trade-offs

Pros

  • Write latency is cache-speed — the caller never waits on the backing store.
  • Absorbs write bursts, smoothing spiky load into a steady trickle to the store.
  • Coalescing collapses repeated writes to a hot key into a single store write.
  • Frees the store to batch writes, which is usually far cheaper than many small ones.

Cons

  • A crash between ack and flush loses data, unless the queue itself is durable.
  • Readers that bypass the cache and hit the store directly see stale values until flush.
  • The flush pipeline needs its own retry, ordering, and failure handling.
  • The system of record briefly disagrees with itself — harder to reason about and debug.

When to use it

Reach for it when

  • Write volume is high and the store's latency or throughput is the bottleneck.
  • Losing the very latest writes on a crash is tolerable, or the cache is itself durable.
  • Writes to the same keys repeat often, so coalescing before flush actually pays off.

Avoid when

  • Every write must be durable the instant it's acknowledged — ledgers, audited transactions.
  • Readers elsewhere in the system need the store itself to be immediately consistent.
  • The write volume doesn't justify the risk — a plain Write-Through cache is safer and simpler.

Code sketch

TypeScript — a minimal write-behind cache 
interface Store<K, V> { save(key: K, value: V): Promise<void>; }

class WriteBehindCache<K, V> {
  private data = new Map<K, V>();
  private dirty = new Set<K>();

  constructor(private readonly store: Store<K, V>, flushMs = 2_000) {
    setInterval(() => this.flush(), flushMs);
  }

  write(key: K, value: V): void {
    this.data.set(key, value);
    this.dirty.add(key);        // acknowledged now, not yet in the store
  }

  read(key: K): V | undefined {
    return this.data.get(key);  // cache is the record until flush runs
  }

  private async flush(): Promise<void> {
    const keys = [...this.dirty];
    this.dirty.clear();
    await Promise.all(keys.map((k) => this.store.save(k, this.data.get(k)!)));
  }
}
