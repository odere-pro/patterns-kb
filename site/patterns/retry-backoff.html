Network · Resilience

Retry with Backoff

Waits progressively longer between attempts — usually with a dash of randomness — so a fleeting failure gets a chance to clear instead of being hammered by an instant retry.

Distributed Resilience

What it is

Retry with Backoff wraps a call that can fail transiently and, on failure, waits before trying it again — each wait longer than the last, typically growing exponentially (base × 2^attempt) up to some ceiling. Control returns to the caller only on success, once attempts are exhausted, or the moment an error is recognized as non-transient and not worth retrying at all.

The force it resolves is a mismatch between failure and response speed. A dropped packet, a momentary overload, a lock held a beat too long — these clear on their own, and an immediate retry often succeeds. But immediate, synchronized retries from many clients at once do the opposite of helping: they pile onto a dependency at the exact moment it's already struggling, turning a brief blip into a self-inflicted retry storm. Backoff spreads that load out over time; adding jitter — randomizing the delay rather than following a fixed curve — desynchronizes clients further so they don't all retry in lockstep.

Backoff alone isn't sufficient. Each attempt still needs its own Timeout / Deadline, or a hung attempt stalls the whole retry loop instead of failing into it. And because retrying re-executes the call, safety depends on Idempotency — a retried write that isn't idempotent can duplicate its effect the moment a response is lost after the operation actually succeeded.

How it works

Variations

Fixed delay
Retry after the same interval every time. Simplest to reason about, but it doesn't ease off under sustained failure and can still synchronize a flood of retries.
Exponential backoff
Each retry's wait multiplies the last, usually doubling, capped at a maximum delay so the curve doesn't grow unbounded on a long outage.
Jittered backoff
Randomize the delay around the backoff curve — full jitter, equal jitter, or decorrelated jitter — so clients that failed at the same instant don't all retry at the same instant too.
Retry budgets
Cap total retry volume as a share of overall traffic, independent of any single caller's attempt count, so retries collectively can never outgrow the load they're meant to smooth over.

Trade-offs

Pros

  • Resolves transient blips automatically, without ever surfacing an error to the caller.
  • Spaces out load over time, giving a struggling dependency room to recover instead of adding to the pile-up.
  • Jitter desynchronizes clients so a shared outage doesn't turn into a synchronized retry storm.
  • Drops into any call site with no changes required on the callee's side.

Cons

  • Only helps with transient failures — retrying a permanent error just delays it and wastes time.
  • Naive or unbounded retries under a real outage amplify load exactly when the dependency needs it least.
  • Requires idempotent operations, or a retried write can duplicate its side effects.
  • Adds latency variance — the failure path gets slower, and less predictably so, with every attempt.

When to use it

Reach for it when

  • The failure is plausibly transient — network blips, brief overload, lock contention, rate limiting.
  • The operation is idempotent, or safe to retry via a dedup key.
  • You control the client and can afford some extra latency on the failure path.

Avoid when

  • The failure is permanent — a bad request, an auth failure, a missing resource — retrying just delays the error.
  • The write isn't idempotent and there's no dedup mechanism to make a repeat safe.
  • The dependency is visibly down for a while — pair with a Circuit Breaker so retries stop adding to the pile-up.

Code sketch

TypeScript — exponential backoff with full jitter 
type RetryOptions = {
  maxAttempts?: number;
  baseMs?: number;
  maxMs?: number;
};

async function withRetry<T>(
  fn: () => Promise<T>,
  { maxAttempts = 5, baseMs = 200, maxMs = 8_000 }: RetryOptions = {},
): Promise<T> {
  for (let attempt = 1; ; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (attempt >= maxAttempts || !isTransient(err)) throw err;
      const backoff = Math.min(maxMs, baseMs * 2 ** attempt);
      const jitteredMs = Math.random() * backoff; // full jitter
      await sleep(jitteredMs);
    }
  }
}

const sleep = (ms: number) =>
  new Promise<void>((resolve) => setTimeout(resolve, ms));

function isTransient(err: unknown): boolean {
  return err instanceof NetworkError || err instanceof RateLimitError;
}
