Abstract Factory

Hands the client one interface that produces a whole family of matching objects — so switching from one family to another, one theme, one platform, one backend, is a single change of factory.

GoF Creational

What it is

An abstract factory is an interface for creating a family of related objects without naming their concrete classes. Instead of the client writing new WinButton() and new WinCheckbox(), it calls factory.createButton() and factory.createCheckbox() on an object that promises to hand back a matching set. Each concrete factory implements that interface for one family — one platform, one theme, one database dialect.

The problem it resolves is consistency across a set of products that must be used together. A dialog built from a Windows button but a macOS checkbox is a bug waiting to happen; a query built from a Postgres connection but SQLite-flavoured SQL is worse. By funnelling every member of the family through one factory, the pattern makes a mismatched combination unrepresentable.

The forces it balances are decoupling versus flexibility. The client depends only on abstract product interfaces and the factory interface, so concrete classes stay swappable and testable. The cost is that the shape of a family — which products it contains — is fixed by the interface, and changing that shape ripples through every implementation.

How it works

Variations

Factory-Method internals
Each create* operation is itself a Factory Method that concrete factories override — the classic way the two patterns compose.
Prototype-backed factory
The factory stores prototypical instances and clones them per request, so a single factory class can serve many families without a subclass each.
Singleton factory
Because one factory per family usually suffices, concrete factories are frequently exposed as singletons or injected once at composition time.
Registry / parameterized factory
A single factory keyed by a family identifier looks products up in a map instead of dedicating a subclass to every family.

Trade-offs

Pros

  • Guarantees the products a client uses together come from one compatible family.
  • Isolates concrete classes — callers depend only on abstract interfaces.
  • Swapping an entire family is a single change: which factory you instantiate.
  • Concentrates construction knowledge in one place, easing tests and wiring.

Cons

  • Adding a new product kind changes the interface and every concrete factory.
  • More classes and indirection than a plain constructor call.
  • Overkill when there is only one family or products never vary together.
  • The family's shape is fixed early; late-breaking product types fit awkwardly.

When to use it

Reach for it when

  • Your code must work with several families of related products and stay independent of which one.
  • You need to guarantee that objects created together are mutually compatible.
  • You want to select and configure a whole product family in one place, such as at startup or in a DI container.

Avoid when

  • There is only one family, or the products don't need to vary as a set.
  • New product kinds appear often — each one forces edits across every factory.
  • A single Factory Method or a plain constructor already covers the need.

Code sketch

TypeScript — a cross-platform widget factory 
interface Button { render(): void }
interface Checkbox { render(): void }

interface GUIFactory {
  createButton(): Button
  createCheckbox(): Checkbox
}

class MacFactory implements GUIFactory {
  createButton() { return new MacButton() }
  createCheckbox() { return new MacCheckbox() }
}

class WinFactory implements GUIFactory {
  createButton() { return new WinButton() }
  createCheckbox() { return new WinCheckbox() }
}

// Client knows only the interfaces — never Mac* or Win*.
function buildUI(factory: GUIFactory) {
  factory.createButton().render();
  factory.createCheckbox().render();
}

const factory = os === "mac" ? new MacFactory() : new WinFactory();
buildUI(factory);
