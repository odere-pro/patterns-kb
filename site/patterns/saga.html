Saga

Breaks a business transaction that spans several services into a chain of local transactions, each paired with a compensating action that undoes it if a later step fails.

Distributed Coordination & Data

What it is

A saga replaces one big cross-service transaction with a sequence of small local ones. Each participating service commits its own local transaction, touching only data it owns, then triggers the next step — either by publishing an event or by responding to a coordinator's command. If any step fails, the saga runs compensating transactions for every step already committed, undoing their effects in reverse order.

The force it resolves is that a business transaction now spans data no single database can lock. In a monolith, wrapping several tables in one ACID transaction is free. Split that same operation across microservices and there's no shared transaction manager: a two-phase commit would need every participant to hold locks until the slowest one votes, coupling everyone's availability to the least reliable service and colliding badly with real-world timeouts and partitions.

A saga refuses that coupling. Each local transaction commits and is durable the instant it runs — no service blocks waiting on another's yes. What you give up is isolation and immediate consistency: a saga in flight leaves partially-applied state visible to the rest of the system, and correctness arrives only once every step, or every compensation, has actually run.

Coordination happens one of two ways: choreographed, where each service listens for the previous step's event and decides its own next action, or orchestrated, where a dedicated coordinator issues each step as a command and tracks the saga's progress explicitly, as a state machine.

How it works

Variations

Orchestration
A saga execution coordinator issues each step as a command and persists the saga's progress as a state machine — the transaction logic lives in one visible place instead of scattered across services.
Choreography
Each service publishes a Domain Event when its step completes, and downstream services subscribe and react on their own. No central coordinator, but the flow is implicit across every participant's code.
Compensating Transaction
Not every failure needs undoing — idempotent retries absorb transient errors, while a Compensating Transaction only runs for a step that fully committed and must be semantically reversed.
Semantic lock
Mark an entity provisional or pending while its saga is in flight, so concurrent readers and writers can detect the in-progress state instead of treating it as final.

Trade-offs

Pros

  • Avoids distributed locks and two-phase commit — each service stays autonomous and available.
  • Scales horizontally, since no participant blocks waiting on the others' votes.
  • The sequence of steps and compensations documents the business process explicitly.
  • Works across heterogeneous datastores; each service keeps its own transaction technology.

Cons

  • No isolation: partial saga state is visible to other transactions mid-flight, unless masked by a semantic lock.
  • Every step needs a compensation designed for it, including ones that feel irreversible.
  • A failure now spans several services — tracing and debugging needs distributed context.
  • Compensations must themselves be idempotent and retry-safe, and their own failure needs a plan.

When to use it

Reach for it when

  • A business transaction spans multiple services or databases with no shared transaction manager.
  • Two-phase commit is off the table, or coupling every participant's availability is unacceptable.
  • Steps are naturally reversible, or an approximate rollback — refund, cancel, release — is good enough.

Avoid when

  • The whole operation stays inside one service, where a local transaction already gives you ACID.
  • A step is truly irreversible and no compensation makes business sense.
  • You need strict isolation or a single consistent view mid-transaction — sagas only give you eventual consistency.

Code sketch

TypeScript — a minimal orchestrated saga 
interface Step {
  action: () => Promise<void>;
  compensate: () => Promise<void>;
}

class Saga {
  private completed: Step[] = [];
  constructor(private readonly steps: Step[]) {}

  async run(): Promise<void> {
    for (const step of this.steps) {
      try {
        await step.action();
        this.completed.push(step);      // remember for compensation
      } catch (err) {
        await this.compensate();
        throw err;
      }
    }
  }

  private async compensate(): Promise<void> {
    for (const step of this.completed.reverse()) {
      await step.compensate();          // undo in reverse order
    }
  }
}

await new Saga([
  { action: () => orders.create(), compensate: () => orders.cancel() },
  { action: () => payments.charge(), compensate: () => payments.refund() },
  { action: () => inventory.reserve(), compensate: () => inventory.release() },
]).run();
