Evaluation

Measuring whether an ML system actually works — a layered stack from business objective down through product and ML metrics to methodology, where offline metrics earn their keep only by predicting online results.

ML

What it is

Evaluation is the discipline of deciding whether an ML system is doing its job, and by how much. It is best structured as a stack: start from the business objective, derive user-facing product metrics that signal success, choose ML metrics that align with those and can be measured from the model, then pick an evaluation methodology — offline and online — and finally address the challenges that make measurement hard.

The force it resolves is that a metric divorced from the business is worthless. A classifier at 99% accuracy is useless when the positive class is 1% of the data. An offline gain that does not move an A/B test is noise. Working at every layer of the stack — and demonstrating that the offline metric you optimize actually predicts the online outcome you care about — is what shows you can ship, not just train, a model.

Each system family needs its own strategy. Classification leans on precision, recall, and PR-AUC under imbalance. Recommenders care about ranking quality (NDCG, MRR, Hit@K) and diversity over long sessions. Search adds rank-aware metrics at a cutoff and debiasing of click logs. Generative systems, where "correct" is subjective, need portfolios of automated and human metrics.

How it works

Variations

Classification
Precision and recall at an operating threshold, summarized by the precision-recall curve; PR-AUC is far more informative than ROC-AUC under heavy imbalance, where ROC-AUC can flatter a useless model. Fix the threshold near human-level precision to keep the system useful.
Recommenders
Ranking metrics — NDCG, MRR, Hit@K — plus catalogue coverage and calibration, measured longitudinally over sessions rather than single impressions. Interleaving compares two rankers on the same user with roughly 10–20× less traffic than an unpaired A/B test.
Search & information retrieval
Rank-aware metrics at a cutoff k (NDCG@k, MAP, MRR) over a graded-relevance test set, with query diversity across head, torso, and tail. Click logs carry presentation bias, so labels need debiasing via inverse-propensity weighting or interleaving.
Generative
No single scalar: combine cheap-but-brittle overlap scores (BLEU, ROUGE), semantic similarity (BERTScore), task-specific fact-checkers, safety and toxicity classifiers, and periodic human preference ratings — tracking how well the automated proxies correlate with human judgement.

Trade-offs

Pros

  • The stack forces every metric to trace back to the business objective, catching the classic trap of optimizing a number users don't care about.
  • Choosing the right metric for the data — PR-AUC over ROC-AUC under imbalance, rank-aware metrics for ranking — prevents a useless model from looking good.
  • Offline evaluation gives a fast iteration loop, and shadow mode lets a model be validated before it ever affects a user.
  • Interleaving and importance sampling make rigorous online measurement affordable even for small lifts or rare classes.

Cons

  • Offline metrics only approximate a moving target; they must be continually validated against online results, and divergence between a short-term metric (CTR) and a long-term one (retention) signals a trap.
  • High-quality labels are expensive and slow, and under class imbalance random sampling wastes budget and inflates variance.
  • Feedback loops let a model train on its own echo chamber, so evaluation needs exploration traffic and a golden set unaffected by the model's own decisions.
  • Generative quality resists any single number, so evaluation stays partly subjective and human-dependent.

When to use it

Reach for it when

  • You need to prove a new model is actually better than the current one, not just different on an offline number.
  • The positive class is rare and accuracy is misleading, so you need PR-AUC, recall at a precision threshold, or impact-weighted metrics.
  • You are designing how a recommender, search, or generative system will be judged, each of which needs its own metric family.
  • You must connect an offline metric to a business outcome and show the two move together before trusting the offline loop.

Avoid when

  • You would report a single headline metric with no methodology, no challenge analysis, and no tie back to the business objective.
  • You would run an expensive online experiment before a cheap offline proxy and shadow-mode check have ruled out the obvious regressions.

Code sketch

TypeScript — precision, recall, and F1 from a confusion count, and why accuracy misleads 
interface Counts { tp: number; fp: number; fn: number; tn: number; }

const precision = (c: Counts) => c.tp / (c.tp + c.fp || 1); // of what we flagged, how much was right
const recall    = (c: Counts) => c.tp / (c.tp + c.fn || 1); // of all positives, how much we caught
const f1 = (c: Counts) => {
  const p = precision(c), r = recall(c);
  return (2 * p * r) / (p + r || 1);
};
const accuracy = (c: Counts) => (c.tp + c.tn) / (c.tp + c.fp + c.fn + c.tn || 1);

// A "detector" that flags nothing, on data with 1% positives:
const lazy: Counts = { tp: 0, fp: 0, fn: 100, tn: 9900 };
console.log("accuracy", accuracy(lazy).toFixed(3)); // 0.990 — looks great
console.log("recall",   recall(lazy).toFixed(3));   // 0.000 — catches nothing
// Accuracy is fooled by imbalance; precision/recall (and PR-AUC across thresholds) are not.
