Message Queue

A point-to-point channel that holds each message until exactly one consumer pulls and processes it — decoupling producer from consumer in both time and pace.

Messaging Lens

What it is

A message queue is an ordered, durable channel sitting between a producer and one or more consumers. A producer enqueues a message and moves on; the queue holds it until a consumer dequeues and processes it. The defining trait is point-to-point delivery: even with several consumers attached to the same queue, each message is delivered to and consumed by exactly one of them — never broadcast to all.

The force it resolves is coupling to the consumer's uptime and pace. A producer that calls a consumer directly is at the mercy of that consumer being reachable, fast, and available right now — if it's down, slow, or already overloaded, the producer blocks or the work is simply lost. A queue breaks that dependency: the producer enqueues and returns immediately, the consumer drains at whatever rate it can sustain, and neither side needs the other online at the same instant.

It also absorbs bursts. Real traffic rarely arrives at a steady rate; a queue lets producers write as fast as they like while consumers pull at a sustainable pace, turning a spike that would overwhelm a consumer into a backlog that drains over the following seconds or minutes instead of failing outright.

Durability is what separates a queue from an in-process buffer. Messages are typically persisted to disk or replicated across broker nodes, so a crash between enqueue and processing doesn't silently drop the work — the message is still sitting there when a consumer, or its replacement, comes back to look.

How it works

Variations

Competing Consumers
Run a pool of consumer instances against the same queue to scale throughput horizontally — the queue still guarantees any one message reaches only one of them.
Visibility timeout / lease-based redelivery
A dequeued message is hidden rather than deleted until the consumer acknowledges it; if the lease expires without an ack, another consumer picks it up — the basis of at-least-once delivery.
Dead Letter Channel
A message that fails processing past a retry limit is diverted to a separate channel instead of blocking the queue or being redelivered forever.
FIFO vs. standard queues
FIFO queues guarantee strict order and exactly-once processing within a group, at the cost of throughput; standard queues favor throughput and tolerate occasional reordering or duplicates.
Claim Check
Store a large payload externally and put only a reference in the message, so broker size limits and throughput aren't spent moving bulk data around.

Trade-offs

Pros

  • Decouples producer and consumer lifecycles — either can be down without losing submitted work.
  • Absorbs bursts, turning spiky arrival rates into a steady, sustainable consumer load.
  • Consumers scale horizontally against one queue without any change to producers.
  • Durable persistence protects in-flight work across crashes and restarts.

Cons

  • Introduces a broker as new infrastructure to run, monitor, and keep highly available.
  • Adds end-to-end latency versus a direct call — a message waits until something polls for it.
  • At-least-once delivery pushes idempotency onto every consumer, or duplicates cause real damage.
  • Flow becomes indirect: tracing a message's path is harder than following a synchronous call stack.

When to use it

Reach for it when

  • Producer and consumer run at different speeds, or shouldn't need to be online at the same time.
  • You need to absorb bursts and let consumers process at a controlled, sustainable rate.
  • Submitted work must survive a crash between being accepted and being processed.

Avoid when

  • The caller needs an immediate, synchronous response — a queue adds latency instead of removing it.
  • Every interested party must see every message — that's broadcast, reach for Publish-Subscribe instead.
  • Producer and consumer are in the same process — a plain function call or in-memory channel is simpler.

Code sketch

TypeScript — a minimal queue with visibility timeout 
interface Message<T> {
  id: string;
  body: T;
  visibleAt: number; // 0 means visible now
}

class MessageQueue<T> {
  private messages: Message<T>[] = [];
  private inFlight = new Map<string, Message<T>>();

  enqueue(body: T): void {
    this.messages.push({ id: crypto.randomUUID(), body, visibleAt: 0 });
  }

  dequeue(visibilityMs = 30_000): Message<T> | undefined {
    const now = Date.now();
    const msg = this.messages.find((m) => m.visibleAt <= now);
    if (!msg) return undefined;
    this.messages = this.messages.filter((m) => m !== msg);
    msg.visibleAt = now + visibilityMs; // hidden until ack or lease expiry
    this.inFlight.set(msg.id, msg);
    return msg;
  }

  ack(id: string): void {
    this.inFlight.delete(id); // processed successfully, gone for good
  }

  nack(id: string): void {
    const msg = this.inFlight.get(id);
    if (msg) {
      this.messages.push(msg); // back on the queue for redelivery
      this.inFlight.delete(id);
    }
  }
}

In the wild

RabbitMQHolds each message until the consumer sends basic.ack; basic.qos sets per-consumer prefetch, quorum queues replicate the queue across nodes, and a dead-letter exchange catches rejected or expired messages.
Amazon SQSManaged queue with a configurable VisibilityTimeout and MessageRetentionPeriod up to 14 days; a redrive policy dead-letters a message after maxReceiveCount attempts, and FIFO queues add ordering plus dedup within a message group.
Apache ActiveMQA JMS broker whose queues give classic point-to-point delivery; a redelivery policy retries with backoff before routing to a dead-letter queue, and the KahaDB store persists messages across restarts.

In production

Tuning knobs

  • Visibility timeout (message lease) — How long a dequeued message stays hidden before redelivery; set above worst-case processing time or extend it with a heartbeat.
  • Consumer prefetch / concurrency — How many unacknowledged messages one consumer holds in flight; caps memory and spreads load across the pool.
  • Max receive count before dead-lettering — Number of failed delivery attempts before a message is diverted to a dead-letter channel.
  • Message retention / TTL — How long an unconsumed message survives before the broker discards it.
  • Consumer pool size — Number of competing consumers draining one queue.

Signals to watch

  • Queue depth (backlog) — Messages waiting to be consumed; the primary saturation signal.
  • Age of oldest unacknowledged message — How long the front of the queue has waited to be picked up.
  • Dead-letter and redelivery rate — Messages diverted or redelivered per interval; a rise flags poison messages or failing consumers.
  • Enqueue vs. dequeue rate — Whether consumers are keeping pace with producers.

Failure modes under load

  • Poison message — A message that always fails is redelivered each cycle, burning a consumer slot until it hits the dead-letter threshold.
  • Lease shorter than processing time — The visibility timeout expires mid-work, a second consumer takes the same message, and it runs twice.
  • Backlog runaway — Sustained enqueue above consumer capacity grows depth without bound; latency climbs until retention drops the oldest.
  • Duplicate delivery — At-least-once redelivery double-applies non-idempotent side effects.

Readiness checklist

  • Consumers are idempotent — dedupe on message id or a business key
  • A dead-letter channel is configured with a bounded retry count
  • Visibility timeout exceeds worst-case processing, or the consumer extends the lease with a heartbeat
  • Queue depth and oldest-message age are alerted, not just graphed
  • Load test confirms consumers drain faster than sustained peak enqueue rate
