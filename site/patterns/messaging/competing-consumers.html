Competing Consumers

A pool of interchangeable consumers pulls from one shared channel, so a single message goes to exactly one worker and the pool's combined throughput — not any one consumer's — sets the pace.

Messaging Lens

What it is

Competing consumers put several identical consumer instances on the same channel — a queue, a topic partition, a work stream — and let a broker hand each incoming message to whichever consumer is free. The consumers don't coordinate with each other and don't know how many siblings they have; they just pull, process, and ack. The broker's delivery guarantee is what keeps two consumers from processing the same message twice.

The force it resolves is a single-consumer throughput ceiling. One consumer reading a channel serially can only go as fast as one process, one CPU, one network connection allows — and if the producer outpaces it, the backlog grows without bound. Rewriting the producer or the message format to fix this is usually not on the table; the channel and its contract are fixed.

The pattern turns "add capacity" into "add a consumer instance." Because consumers are stateless and interchangeable, scaling out is a deployment decision, not a code change: start more workers pointed at the same channel and the broker starts spreading messages across all of them. Scaling back down is just as cheap.

This is a lens, not a single fixed mechanism — the same idea shows up as JMS queue listeners, Kafka consumer groups, RabbitMQ competing consumers on one queue, and SQS workers polling the same queue. What they share is one channel, many pullers, and exactly-once-per-message delivery to the group.

How it works

Variations

Static pool
A fixed number of consumer instances, sized for expected peak load. Simple to operate, wasteful when load is bursty.
Partitioned / keyed consumption
Messages are hashed to a partition or shard by key, and each partition is owned by exactly one consumer at a time — preserves per-key order while still spreading unrelated keys across the pool (Kafka partitions, Kinesis shards).
Publish-Subscribe
Layer competing consumers under pub/sub: each subscriber group receives every message once, but the consumers inside a group compete for it — fan-out across groups, load-sharing within one.
Exclusive delivery via lease / visibility timeout
The broker hides a delivered message from other consumers for a lease window; a consumer that crashes before acking lets it reappear for someone else to pick up.

Trade-offs

Pros

  • Scales throughput horizontally by adding consumer instances, with no producer changes.
  • Consumers are stateless and interchangeable — any one can crash and restart without special handling.
  • The channel itself absorbs load spikes as a buffer while the pool catches up.
  • Failed processing just means the message goes back for another consumer to try.

Cons

  • No ordering guarantee across the pool — two related messages can be processed out of sequence by different consumers.
  • Most brokers give at-least-once delivery, so a redelivered message must be handled safely, not processed twice.
  • A poison message that always fails can bounce between consumers until it's dead-lettered.
  • Uneven message cost means one consumer can be starved while another sits idle.

When to use it

Reach for it when

  • A single consumer can't keep up with the rate messages arrive on a channel.
  • The work is naturally parallel — messages don't depend on each other's outcome.
  • You want to scale processing capacity by deployment, without touching the producer.

Avoid when

  • Messages must be processed in strict global order — a single consumer, or per-key partitioning, fits better.
  • Every consumer needs to see every message — that's fan-out, not sharing; reach for Publish-Subscribe instead.
  • Processing isn't safely repeatable — redelivery under at-least-once delivery will corrupt state.

Code sketch

TypeScript — a worker pulling from a shared queue 
interface Queue<T> {
  receive(visibilityMs: number): Promise<{ id: string; body: T } | null>;
  ack(id: string): Promise<void>;
  nack(id: string): Promise<void>; // returns the message for another consumer
}

async function runConsumer<T>(
  queue: Queue<T>,
  handle: (msg: T) => Promise<void>,
) {
  while (true) {
    const msg = await queue.receive(30_000); // hidden from others for 30s
    if (!msg) continue;                      // nothing waiting, poll again
    try {
      await handle(msg.body);
      await queue.ack(msg.id);               // done — remove from queue
    } catch {
      await queue.nack(msg.id);               // let a sibling retry it
    }
  }
}

// N interchangeable workers pulling from the same queue
for (let i = 0; i < workerCount; i++) {
  runConsumer(queue, processOrder);
}

In the wild

Kafka consumer groupsPartitions are divided among the consumers in a group so exactly one member reads each partition; a rebalance reassigns partitions when a member joins or leaves. The partition count is the hard ceiling on parallelism — consumers beyond it sit idle with nothing assigned.
CeleryPython workers compete for tasks on a shared RabbitMQ or Redis broker, scaling by adding worker processes or raising --concurrency. worker_prefetch_multiplier controls how many tasks each worker reserves ahead of time, the knob that trades throughput against even distribution.
SidekiqRuby workers compete for jobs on shared Redis queues, each process running a pool of threads (concurrency) that pull with BRPOP. Because plain BRPOP removes the job before it is done, an in-flight job is lost if the process is killed; reliable fetch that re-queues interrupted jobs is a Pro/Enterprise feature.
Amazon SQSMany workers poll one queue and each message is handled once: a visibility timeout hides an in-flight message from sibling workers while it is processed, a Lambda event source mapping polls the queue and invokes functions in batches, and a redrive policy moves a message to a dead-letter queue after maxReceiveCount failed attempts.

In production

Tuning knobs

  • Consumer count / concurrency — The number of consumer instances or threads pulling from the channel — the primary throughput dial; scaling out is a deployment change, not a code change.
  • Prefetch / batch size — How many unacked messages one consumer holds at a time (RabbitMQ basic.qos prefetch_count, Kafka max.poll.records) — too high and one consumer hoards the backlog while siblings idle.
  • Visibility timeout / lease — How long a delivered message is hidden from other consumers before it reappears (SQS VisibilityTimeout, Kafka max.poll.interval.ms) — set below processing time and the message is redelivered mid-flight.
  • Max delivery attempts — The redelivery count after which a repeatedly-failing message is diverted to a dead-letter path (SQS maxReceiveCount) instead of bouncing through the pool forever.
  • Partition / shard count — In a keyed model, the number of partitions caps how many consumers in a group can work in parallel (Kafka partitions, Kinesis shards).

Signals to watch

  • Queue depth / consumer lag — Backlog waiting versus drain rate (Kafka consumer-group lag, SQS ApproximateNumberOfMessages) — the headline signal of whether the pool is keeping up.
  • Per-consumer throughput spread — Distribution of work across instances — a wide spread means one consumer is starved or hot while others idle.
  • In-flight / unacked count — Messages leased but not yet acked — high and stuck points at slow handlers or leaks.
  • Redelivery rate — How often messages are re-delivered after nack or lease expiry — a proxy for handler failures and duplicate work.

Failure modes under load

  • Lease shorter than processing — A visibility timeout below the real processing time redelivers the message while the first consumer is still working it, so two consumers do the same work.
  • Prefetch starvation — A prefetch set too high lets one consumer pull the whole backlog into its buffer while its siblings sit idle, defeating the load sharing.
  • Idle consumers past partition count — In a keyed model, consumers beyond the partition count get no partition assigned and sit idle no matter how much backlog exists.
  • Poison message churn — A message that always fails bounces between consumers, burning capacity, until a max-attempts cap dead-letters it.

Readiness checklist

  • Make handlers idempotent — at-least-once delivery means a redelivered message must be safe to process twice.
  • Set the visibility / lease timeout comfortably above p99 processing time.
  • Tune prefetch low enough that work spreads evenly across the pool.
  • Configure a dead-letter path with a max-attempts cap so poison messages leave the pool.
  • Ensure partition or shard count is at least the target consumer count in a keyed model.
