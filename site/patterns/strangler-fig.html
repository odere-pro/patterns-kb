Strangler Fig

Replaces a legacy system one slice at a time — routing each capability through a facade to a new implementation until the old system has nothing left to do and can be switched off.

Distributed Coordination & Data

What it is

The strangler fig pattern replaces a legacy system by growing a new one around it, one capability at a time, until the old system has nothing left to do and can be switched off. It takes its name from the strangler fig vine, which germinates in the canopy of a host tree, sends roots down around its trunk, and slowly out-competes it — by the time the host dies and rots away, the fig stands on its own, having never needed the tree to fall first.

The force it resolves is the risk of a big-bang rewrite. Replacing a large legacy system outright means freezing feature work for months or years, betting the business on a single cutover, and discovering integration gaps only after the old system is already switched off with no way back. Martin Fowler named the pattern for exactly this reason: it lets a team replace a system's capabilities incrementally, verify each one in production, and always have a working system — old, new, or a mix of both — at every point in the migration.

Mechanically, a routing facade sits in front of the legacy system and intercepts every call. Each capability is rebuilt behind that facade, and once the new version is trusted, the facade redirects that traffic to it instead of the legacy path. The legacy system keeps serving whatever hasn't been migrated yet, shrinking release by release, until it handles nothing at all and can be decommissioned outright.

How it works

Variations

Branch by abstraction
An in-process variant: put an interface in front of the code to be replaced and swap implementations behind it, migrating without a network hop or a separate deployable.
API Gateway routing
An edge router dispatches each request to old or new backend by path or feature flag — the most common way to build the facade at the network boundary.
Database strangling
Migrate the data layer slice by slice too, using dual writes or change-data-capture to keep old and new stores consistent during the transition.
Anti-Corruption Layer facade
The routing seam also translates models, so the new system is built against a clean domain and never coupled to the legacy schema.

Trade-offs

Pros

  • The business keeps shipping — no long feature freeze while a rewrite catches up.
  • Risk is spread across many small cutovers instead of one irreversible big-bang release.
  • Each slice can be rolled back independently by pointing the router back at legacy.
  • The legacy system remains a working safety net until the new one has proven itself.

Cons

  • The routing facade becomes a new, mission-critical piece of infrastructure to get right.
  • Running two systems — and often two data stores — in parallel costs double the operational overhead, sometimes for years.
  • Keeping shared data consistent across old and new is usually harder than the code migration itself.
  • A migration with no firm end date can stall halfway, leaving a permanent two-headed system.

When to use it

Reach for it when

  • The legacy system is business-critical and can't be taken offline for a rewrite.
  • You want to de-risk a large migration by shipping and validating small increments.
  • Different capabilities can be migrated and verified independently, without one atomic cutover.

Avoid when

  • The system is small enough that a straightforward rewrite is faster than building and running a routing seam.
  • There's no clean seam to route through — a tangled monolith may need decomposition work before slices can be carved out at all.
  • The team can't commit to finishing the migration — a stalled strangler just leaves two systems running forever.

Prevents the smell of a stalled or ever-growing rewrite calcifying into a Big Ball of Mud — routing incrementally keeps the system shippable at every step instead of frozen mid-migration.

Code sketch

TypeScript — a minimal routing facade 
type Handler = (req: Request) => Promise<Response>;

// Capabilities that have been rebuilt and verified on the new system.
const migrated = new Set<string>(["/api/users", "/api/orders"]);

class StranglerFacade {
  constructor(
    private readonly legacy: Handler,
    private readonly modern: Handler,
  ) {}

  async handle(req: Request): Promise<Response> {
    const route = new URL(req.url).pathname;
    // Route by capability, not by request content — the seam is explicit.
    return migrated.has(route)
      ? this.modern(req)
      : this.legacy(req);
  }
}

// As each capability is rebuilt, add its route here.
// The legacy handler never sees that route again.
migrated.add("/api/invoices");
