Builder

Assembles a complex object through a sequence of small, readable steps — so construction logic lives apart from the object's final shape, and the same steps can yield different representations.

GoF Creational

What it is

A builder separates the construction of a complex object from its final representation. Instead of one constructor that takes every field at once, you drive a builder through a series of small steps — set this, add that, configure the other — and then ask it for the finished product.

The problem it resolves is the telescoping constructor and the giant parameter list. When an object has many fields, several of them optional, construction degrades into a wall of positional arguments no one can read, and each new option forces another overload. A builder replaces those arguments with named, order-independent method calls.

In the classic Gang-of-Four form a separate Director scripts the build sequence while the Builder supplies the steps, so the same process can produce different representations. In everyday code the Director often disappears and the caller drives a fluent builder directly.

How it works

Variations

Fluent (chained) builder
Each step returns this, so calls chain into one readable expression ending in build(). The most common modern form.
Director-driven vs. builder-only
Keep a Director to encapsulate a canonical build recipe, or drop it and let the client sequence the steps itself. Most code today skips the Director.
Staged (step) builder
Return a different interface from each step so the type system enforces required fields and ordering — you can't call build() until the mandatory steps are done.
Test Data Builder
A builder seeded with sensible defaults, where a test overrides only the one field it cares about — fixtures stay terse and intention-revealing.

Trade-offs

Pros

  • Builds complex objects step by step, keeping construction readable.
  • Separates construction from representation — same steps, different products.
  • Retires telescoping constructors and long positional argument lists.
  • Can validate invariants and required fields before producing the object.

Cons

  • More code: a builder class per product adds real boilerplate.
  • Overkill for simple objects with only a few fields.
  • A half-configured builder can emit an incomplete object if unguarded.
  • Builder state is mutable; reusing one instance across builds can leak values.

When to use it

Reach for it when

  • An object has many parameters, several of them optional.
  • Construction is multi-step, order-sensitive, or needs validation along the way.
  • You want the same build process to yield different representations.

Avoid when

  • The object is simple — a plain constructor or a factory reads better.
  • All fields are required and few; positional arguments are clear enough.
  • Immutable records plus named arguments already express intent.

Code sketch

TypeScript — a fluent builder 
// A fluent builder: each step returns `this` for chaining.
class QueryBuilder {
  private parts: string[] = [];

  select(...cols: string[]): this {
    this.parts.push(`SELECT ${cols.join(", ")}`);
    return this;
  }
  from(table: string): this {
    this.parts.push(`FROM ${table}`);
    return this;
  }
  where(cond: string): this {
    this.parts.push(`WHERE ${cond}`);
    return this;
  }
  build(): string {
    return this.parts.join(" ");   // the finished product
  }
}

const sql = new QueryBuilder()
  .select("id", "name")
  .from("users")
  .where("active = true")
  .build();
