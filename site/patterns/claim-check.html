Claim Check

Stores a bulky payload in durable storage and lets only a lightweight reference — the claim check — travel through the message channel, so the channel stays fast and the data moves only when a consumer actually redeems it.

Messaging Lens

What it is

A claim check takes a large piece of data — a file, an image, a video, a bulk export — writes it to durable, shared storage, and puts only a small reference to it on the message channel. The message carries an identifier or URL instead of the payload; whichever consumer actually needs the data trades that reference for the real thing by fetching it from the store.

The force it resolves is the mismatch between what a message channel is good at and what a large payload demands. Brokers are tuned for many small, fast messages — Kafka defaults to a 1 MB message cap, SQS to 256 KB — and pushing megabytes through them chews up broker memory, network bandwidth, and replay time, even for the consumers that never look at the body. Splitting storage from notification lets the channel stay quick and the data sit wherever it's cheapest to hold.

The name comes from the same idea as a coat check or an airport baggage tag: you hand over the bulky item once, get back a small ticket, and only redeem the ticket for the item when you actually need it back.

How it works

Variations

Store-first hand-off
The producer writes the payload to the store and waits for confirmation before publishing the claim check, so the message never references data that isn't there yet.
Reference as pre-signed URL
The claim check is a time-boxed, directly fetchable URL rather than an opaque key, so any consumer can retrieve the payload without a shared client library or extra credentials.
Content-addressed claim check
The reference is a hash of the payload itself, so identical bodies collapse to one stored object and a consumer can verify what it fetched matches what was sent.
Claim check with TTL / cleanup
Stored objects expire or get swept after a retention window, so an unredeemed or already-consumed claim check doesn't leave storage growing forever.

Trade-offs

Pros

  • Keeps the message channel fast by capping what actually crosses it to a small reference.
  • Consumers that don't care about the payload never pay to move, deserialize, or buffer it.
  • Storage and messaging scale independently — cheap object storage for bulk data, a lean broker.
  • The payload can be inspected, versioned, or reprocessed in the store without replaying the message.

Cons

  • Adds a network round trip and a second system — the store — that must be up for the message to be useful.
  • Two writes, store then queue, aren't atomic — a crash between them leaves an orphaned object or a phantom reference.
  • Someone has to garbage-collect stored payloads once every claim check is redeemed or expires.
  • Debugging is harder: the message body only tells you where to look, not what happened.

When to use it

Reach for it when

  • A payload routinely exceeds the broker's practical or hard message-size limit.
  • Most consumers only need to route or filter on metadata, not the full body.
  • Storage and message-channel throughput need to scale independently.

Avoid when

  • Payloads are consistently small enough to fit the channel without strain — the extra store round trip only adds latency.
  • The store write and the message publish must be atomic — pair with an outbox instead of send-and-hope.
  • Only one consumer ever exists and it always needs the full payload immediately — just send it inline.

Code sketch

TypeScript — producer and consumer around a claim check 
interface ClaimCheckStore {
  put(payload: Buffer): Promise<string>;   // returns a claim check id
  get(id: string): Promise<Buffer>;
}

interface Queue {
  send(message: { claimCheckId: string }): Promise<void>;
}

async function publishLargePayload(
  payload: Buffer,
  store: ClaimCheckStore,
  queue: Queue,
): Promise<void> {
  const claimCheckId = await store.put(payload); // payload lands in the store first
  await queue.send({ claimCheckId });             // only the reference crosses the channel
}

async function consumeLargePayload(
  message: { claimCheckId: string },
  store: ClaimCheckStore,
): Promise<Buffer> {
  return store.get(message.claimCheckId);         // redeem the ticket for the real data
}
