Outbox

Commits the state change and the event that announces it in a single local transaction, then hands the event to a relay that publishes it at least once — so the write and its notification can never drift apart, even if the broker is unreachable at the moment of commit.

Distributed Coordination & Data

What it is

The outbox pattern adds a table to a service's own database that holds messages waiting to be published. When the service changes state, it writes both the business row and an outbox row — the event describing that change — inside the same local ACID transaction. A separate process then reads the outbox and delivers each message to a broker, marking it published once it's safely on its way.

It resolves the dual-write problem. A service's database and its message broker are two different systems with no shared transaction between them. Publish before committing and a rolled-back write leaves the world believing something happened that never did. Commit and then publish, and a crash or a broker outage between the two silently drops the event, with the database looking perfectly consistent and no error anywhere. There's no way to make "write to the database" and "send to the broker" atomic across two resource managers without something like distributed two-phase commit, which most message brokers don't support and most teams don't want to run.

Outbox sidesteps the problem instead of solving it head-on: it turns the publish into a second local write, which the database transaction already makes atomic with the first. The genuinely hard part — getting the message across the network exactly when it should go — is pushed onto a relay that can retry indefinitely against a table, rather than a request handler that gets one shot before the client hangs up.

How it works

Variations

Polling relay
A process periodically queries the outbox for unpublished rows and publishes them. Simple to build and reason about, at the cost of polling latency and steady read load on the table.
Transaction log tailing (CDC)
A connector like Debezium tails the database's write-ahead log or binlog directly and turns each outbox insert into a broker message with no polling and near-zero added latency.
Event Sourcing
When every write is already an appended event, the event store doubles as the outbox — there's no separate table, because the write is the thing to publish.
Shared vs. per-aggregate outbox table
One outbox table serves the whole service, or each aggregate gets its own — the latter isolates hot tables and lets ordering guarantees stay scoped to one aggregate's stream.

Trade-offs

Pros

  • Solves the dual-write problem with a single local ACID transaction — no distributed transaction coordinator needed.
  • Guarantees at-least-once delivery: an event is never lost because the broker happened to be down at commit time.
  • Recovery is just retrying unpublished rows — no special failure-handling logic in the request path.
  • Works with any relational database and any broker; it's a pattern, not a product.

Cons

  • Adds a table, a relay process, and one more moving part to deploy and monitor.
  • Delivers at-least-once, not exactly-once — consumers must dedupe or tolerate replays.
  • Polling relays add publish latency and load; log-tailing needs CDC infrastructure to run.
  • The outbox table grows unbounded without its own archiving or cleanup job.

When to use it

Reach for it when

  • A service must update its own data and reliably announce that change to other services or a broker.
  • You need at-least-once delivery that survives a crash or a broker outage right after commit.
  • The database and the message broker are separate systems with no shared transaction between them.

Avoid when

  • Nothing outside the service's own transaction needs to know about the change.
  • An occasional missed notification is acceptable and the extra table and relay aren't worth running.
  • Delivery volume and stakes are low enough that a direct, synchronous call is simpler and fast enough.

Code sketch

TypeScript — a write, an outbox row, and a relay 
// Same local transaction: state change + outbox row
async function placeOrder(tx: Transaction, order: Order) {
  await tx.query(
    "insert into orders (id, status) values ($1, $2)",
    [order.id, "placed"],
  );
  await tx.query(
    `insert into outbox (id, topic, payload, published)
     values ($1, $2, $3, false)`,
    [randomUUID(), "order.placed", JSON.stringify(order)],
  );
  // Both rows commit together, or neither does.
}

// Relay: runs independently, drains unpublished rows
async function relayOnce(db: Db, broker: Broker) {
  const rows = await db.query(
    "select * from outbox where published = false order by id limit 100",
  );
  for (const row of rows) {
    await broker.publish(row.topic, row.payload);
    await db.query(
      "update outbox set published = true where id = $1",
      [row.id],
    );
  }
}
