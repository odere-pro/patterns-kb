Backpressure

A slow consumer signals its own saturation back upstream, so the producer throttles, buffers within bounds, or sheds load — instead of blindly burying it.

Concurrency Lens

What it is

Backpressure is a feedback signal that travels against the direction of data flow. When a consumer — or any stage in a pipeline — can't keep up, it tells whoever is feeding it, and that upstream stage responds by slowing production, holding items in a bounded buffer, or dropping some of them, rather than pushing data at a rate the receiver can't absorb.

The force it resolves is a speed mismatch between stages that are otherwise wired together. A producer and a consumer rarely run at exactly the same rate, and the gap between them has to go somewhere. Without backpressure it goes into an ever-growing queue: memory climbs, latency for anything still in the queue balloons, and eventually the process runs out of memory or the whole pipeline falls over — often long after the mismatch actually started, which makes it hard to trace back to its source.

It's a lens rather than one fixed shape: the same idea recurs at every layer of a system — TCP's receive window, HTTP/2 and gRPC flow control, Reactive Streams' request(n) protocol, a bounded channel between goroutines, consumer lag driving a producer's send rate. Whatever the mechanism, the scheme only works end to end — every hop between producer and consumer has to honor the signal, or the one stage that ignores it becomes the new place the queue grows unbounded.

How it works

Variations

Producer-Consumer with a bounded queue
The simplest form: the producer blocks (or waits) once the queue hits capacity, and draining it is itself the signal to resume.
Credit / window-based flow control
The receiver advertises how much it can accept — TCP's window, HTTP/2 and gRPC's flow-control credits — and the sender never exceeds that budget.
Reactive pull (request(n))
The consumer explicitly asks for a number of items before the producer emits them, inverting push into a demand-driven pull, as in Reactive Streams.
Load shedding as fallback
When there's no time to wait for the signal to propagate, drop the oldest or newest items instead of blocking — trades completeness for staying alive.

Trade-offs

Pros

  • Keeps memory bounded — the producer slows instead of a queue growing without limit.
  • Surfaces the pipeline's real throughput instead of hiding it behind a swelling buffer.
  • Degrades gracefully under sustained load rather than failing catastrophically later.
  • Matches the whole pipeline's pace to its slowest stage instead of its fastest.

Cons

  • Only works if every stage between producer and consumer honors the signal.
  • Pushing a slowdown upstream can turn one stage's local hiccup into a system-wide one.
  • Adds latency — producers now wait or throttle instead of firing and forgetting.
  • Badly tuned feedback loops can oscillate, stall, or deadlock across stages.

When to use it

Reach for it when

  • Pipeline stages run at inherently different, variable speeds and the fast one can outrun the slow one.
  • You need bounded memory and predictable behavior under sustained load, not just short bursts.
  • You control (or can influence) the producer, so it can actually act on a slow-down signal.

Avoid when

  • The producer is outside your control, like external clients — reject over-quota traffic with a Rate Limiter instead of asking it to politely slow down.
  • The load is a short, bursty spike you'd rather absorb — a Queue-Based Load Leveling buffer can be simpler.
  • The consumer itself is the real bottleneck — fix that first instead of engineering flow control around it.

Code sketch

TypeScript — a bounded channel that pushes back 
class BoundedChannel<T> {
  private readonly buffer: T[] = [];
  private readonly consumers: Array<(item: T) => void> = [];
  private readonly producerWaiters: Array<() => void> = [];

  constructor(private readonly capacity: number) {}

  // Producer blocks once the buffer is full — this IS the backpressure.
  async send(item: T): Promise<void> {
    if (this.buffer.length >= this.capacity) {
      await new Promise<void>((resolve) => this.producerWaiters.push(resolve));
    }
    this.buffer.push(item);
    this.drain();
  }

  // Consumer pulls when ready, never faster than it can process.
  receive(): Promise<T> {
    return new Promise<T>((resolve) => {
      this.consumers.push(resolve);
      this.drain();
    });
  }

  private drain(): void {
    while (this.buffer.length > 0 && this.consumers.length > 0) {
      this.consumers.shift()!(this.buffer.shift()!);
      this.producerWaiters.shift()?.(); // freed a slot, unblock a producer
    }
  }
}
