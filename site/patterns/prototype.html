Prototype

Builds a new object by copying a fully-formed example — cloning a configured instance instead of constructing one from scratch.

GoF Creational

What it is

The Prototype pattern creates new objects by copying an existing instance — the prototype — rather than instantiating a class directly. Each object knows how to clone itself, so the client asks an example for a copy and gets a ready-made object of the right concrete type without naming that type at all.

It resolves two forces. First, construction can be expensive: an object that took a database read, a parse, or heavy computation to assemble is far cheaper to duplicate than to rebuild. Second, the concrete class is often unknown at compile time — the client holds a prototype it was handed and wants another like it, without a growing tangle of if/switch on type.

The usual alternative — a factory subclass per product variant — creates a parallel class hierarchy that shadows the products themselves. Prototype collapses that: register a few pre-configured example instances and clone the one you need, adding or removing variants at runtime by changing the set of prototypes.

How it works

Variations

Shallow vs. deep clone
A shallow copy shares nested references; a deep copy duplicates the whole object graph. The right choice depends on which state must be independent.
Prototype registry
A named manager holds ready-made prototypes so clients fetch and clone by key — variants become data you can register at runtime.
Copy constructor
Instead of a clone() method, a constructor that takes an instance of the same class and copies its fields — idiomatic in C++ and often in TypeScript.
Serialize-and-rebuild
Round-trip through a serialized form (JSON, binary) to produce a fully detached deep copy without hand-writing per-field logic.

Trade-offs

Pros

  • Add and remove product variants at runtime by registering prototypes.
  • Skips costly initialization when copying an already-built instance.
  • Produces complex, pre-configured objects without knowing their concrete class.
  • Avoids the parallel factory-subclass hierarchy that shadows the products.

Cons

  • Deep-copying graphs with cycles or shared resources is error-prone.
  • Every concrete type must implement clone correctly, or copies come out half-built.
  • Cloning can silently duplicate references you meant to share — or share ones you meant to copy.
  • Mutable internal state carried into a clone couples it to its origin in subtle ways.

When to use it

Reach for it when

  • Building an object is expensive and you already hold a suitable instance to copy.
  • The concrete class to instantiate is chosen at runtime, not compile time.
  • You need many objects that differ only slightly from a configured template.

Avoid when

  • Objects are cheap to construct directly and carry no heavy setup.
  • Instances hold non-copyable resources — open sockets, file handles — that make cloning ambiguous.
  • A plain factory or builder expresses the intent more clearly than "copy this one."

Code sketch

TypeScript — clone with a prototype registry 
interface Prototype<T> {
  clone(): T;
}

class Document implements Prototype<Document> {
  constructor(
    public title: string,
    public tags: string[] = [],
  ) {}

  clone(): Document {
    // copy the mutable array, not just its reference
    return new Document(this.title, [...this.tags]);
  }
}

// a registry of ready-made prototypes
const registry = new Map<string, Document>();
registry.set("report", new Document("Untitled Report", ["draft"]));

const doc = registry.get("report")!.clone();
doc.title = "Q3 Report";
doc.tags.push("final");        // the stored prototype is untouched
