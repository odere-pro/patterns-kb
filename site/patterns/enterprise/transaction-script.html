Application

Transaction Script

Handles each business transaction as a single straight-line procedure — pull in the data it needs, apply the rules inline, write the results back out — with no domain model standing between the request and the database.

Enterprise

What it is

A transaction script is one procedure that handles one request from the presentation layer to the database and back — placeOrder, cancelSubscription, applyDiscount. It reads the rows it needs, runs whatever validation and calculation the business rule requires, writes the rows back, and returns. There's no domain object standing in between deciding how to behave; the logic lives in the procedure itself.

The force it resolves is upfront modeling cost. Building a real domain model — objects with behavior, invariants enforced in one place, a mapping layer to persistence — pays off when business logic is rich and reused across many operations. Most transactions aren't that: they're a handful of checks and one write. A transaction script matches the shape of the problem exactly, is easy to write top-to-bottom, and is trivial to trace because nothing is indirected through a class hierarchy.

Martin Fowler catalogued it in Patterns of Enterprise Application Architecture as the natural starting point for application logic, and the natural thing to graduate away from once transactions stop being simple. It scales in the number of procedures, not in the sophistication of any one of them — which is exactly where it starts to strain.

How it works

Variations

Scripts as free functions
Each transaction is a standalone function grouped by module or file — no class at all, just procedures and the data access calls they make.
Scripts as class methods
Related transactions are grouped as methods on a class named for the subject area (e.g. OrderTransactions), purely for organization — the class carries no state or behavior of its own.
Scripts as stored procedures
The procedure is written and executed inside the database itself, pushing the whole transaction — logic and data access both — into the DB tier.
Shared subroutine extraction
Common steps — parsing input, formatting a response, a repeated validation — are pulled into helper functions called by several scripts, trimming duplication without introducing a domain model.

Trade-offs

Pros

  • Simple and direct — the whole transaction reads top to bottom in one place.
  • No upfront design cost: no domain model, no object-relational mapping to build.
  • Matches procedural and SQL-heavy teams' existing mental model.
  • Cheap to add a new transaction — write one more procedure, nothing else to touch.

Cons

  • Business rules shared across transactions get copy-pasted instead of reused.
  • No object enforces invariants, so the same rule can be checked inconsistently — or forgotten — in different scripts.
  • Business logic is welded to data access, making it hard to unit-test without a database.
  • Scripts grow long and tangled as more rules and edge cases pile onto one procedure.

When to use it

Reach for it when

  • The logic per transaction is a handful of validations and one write — genuinely simple.
  • The team and codebase already think procedurally, and a domain model wouldn't be rewarded with reuse.
  • You need to ship a transaction now, without first designing a domain model.

Avoid when

  • Business rules are numerous, interrelated, and start duplicating across scripts — a Service Layer over a real domain model earns its cost.
  • You need to unit-test business rules independently of the database — a script's logic is welded to its data access.
  • Several transactions should share and enforce the same invariants, not just similar data access.

Piling ever more logic into procedures while the domain classes stay pure data bags is exactly how a codebase ends up with an Anemic Domain Model.

Code sketch

TypeScript — one procedure, one transaction 
// One function does the whole transaction: read, decide, write, return.
async function placeOrder(db: Database, input: PlaceOrderInput): Promise<OrderResult> {
  const customer = await db.queryOne(
    "SELECT id, credit_limit FROM customers WHERE id = $1", [input.customerId],
  );
  if (!customer) throw new Error("unknown customer");

  const total = input.items.reduce((sum, i) => sum + i.price * i.qty, 0);
  if (total > customer.credit_limit) {
    throw new Error("order exceeds credit limit");
  }

  const orderId = await db.insert("orders", {
    customer_id: input.customerId,
    total,
    status: "placed",
  });
  for (const item of input.items) {
    await db.insert("order_items", { order_id: orderId, ...item });
  }

  await db.commit();
  return { orderId, total };
}
