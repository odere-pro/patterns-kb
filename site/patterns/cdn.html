Network · Routing & Scale

CDN

Pushes static content out to a distributed fleet of edge servers positioned near users, so a request is answered from a nearby cache instead of crossing continents to a single origin.

Distributed Routing & Scale

What it is

A content delivery network is a fleet of cache servers — points of presence (PoPs) — distributed across regions, sitting between clients and an origin server. DNS or anycast routing sends each request to the nearest PoP rather than the origin. If that PoP already holds a fresh copy of the asset, it answers directly: no round trip to the origin, no origin load at all.

The force it resolves is distance. Round-trip latency is bounded by the speed of light long before any origin-side optimization runs out of headroom — a request crossing half the globe pays that tax no matter how fast the origin responds. A CDN doesn't make the origin faster; it makes the trip shorter by answering from somewhere close to the request.

It resolves a second force too: concentrated load. A single origin serving every client on every request is one target for every spike, every hotlink, every crawler. By absorbing the repeat, cacheable traffic at the edge, the origin only ever sees the cold path — first requests and cache misses — and a global surge in demand never has to reach it directly.

How it works

Variations

Cache-Aside pull model
The default: an edge node checks its own cache, and on a miss fetches from origin and stores the result — origin traffic tracks only cold, actually-requested paths.
Push CDN
The origin proactively uploads or replicates content to edges ahead of demand, instead of waiting for a first request. Common for large media libraries with a known release schedule.
Consistent Hashing for edge selection
Within a PoP or across a shard, requests for the same key route to the same cache node by hash, so duplicate copies of one object don't scatter across the fleet.
Multi-CDN
Route across two or more CDN vendors, by region or health, to avoid a single provider's outage or peering problem taking down delivery entirely.
Edge compute
Run request-time logic — auth checks, redirects, A/B routing, image transforms — at the PoP itself, so even dynamic decisions skip the trip to origin.

Trade-offs

Pros

  • Cuts latency by answering from a PoP physically close to the user, not the origin.
  • Absorbs traffic spikes and bandwidth cost at the edge, shielding the origin.
  • Improves availability — cached content can keep serving through an origin outage.
  • Scales to a global audience without provisioning the origin for worldwide peak load.

Cons

  • Cache invalidation is the hard part — stale content lingers until TTL expiry or a purge propagates everywhere.
  • Personalized, highly dynamic, or per-user content doesn't cache well and gains little.
  • Another layer of infrastructure, config, and vendor to operate and debug.
  • Doesn't remove the origin dependency — cold paths and misses still hit it directly.

When to use it

Reach for it when

  • You serve static assets — images, scripts, video, downloads — to a geographically spread audience.
  • Traffic is spiky or global, and the origin shouldn't have to absorb it directly.
  • Latency to a distant origin region is the bottleneck users actually feel.

Avoid when

  • Content is deeply personalized per request with no cacheable overlap between users.
  • Your whole audience is co-located with the origin — there's no distance to remove.
  • Reads must be strictly consistent and a stale cached response is unacceptable.

Code sketch

TypeScript — a minimal pull-mode edge cache 
interface Origin {
  fetch(path: string): Promise<{ body: string; ttlSeconds: number }>;
}

interface CacheEntry {
  body: string;
  expiresAt: number;
}

class EdgeNode {
  private cache = new Map<string, CacheEntry>();

  constructor(private readonly origin: Origin) {}

  async serve(path: string): Promise<string> {
    const hit = this.cache.get(path);
    if (hit && hit.expiresAt > Date.now()) {
      return hit.body; // cache hit — no origin request
    }
    const { body, ttlSeconds } = await this.origin.fetch(path); // cache miss
    this.cache.set(path, { body, expiresAt: Date.now() + ttlSeconds * 1000 });
    return body;
  }

  purge(path: string): void {
    this.cache.delete(path); // explicit invalidation ahead of TTL
  }
}
