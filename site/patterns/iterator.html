Iterator

Hands the client one element at a time through a uniform cursor — hasNext, next — so a list, a tree, or a streamed result set all traverse the same way while their internals stay sealed.

GoF Behavioral

What it is

An iterator is an object that encapsulates how to walk through a collection — the aggregate — one element at a time. It exposes a tiny protocol (classically hasNext() then next(), or a single next() that also signals done) and holds the traversal state — the current position — itself, so the collection never has to.

The force it resolves is coupling to representation. When callers loop by indexing into an array or chasing node pointers, they bind themselves to the concrete storage. Swap that array for a hash set, a tree, or a paged remote query and every loop breaks. An iterator decouples how you traverse from how the data is stored: the same client code runs over anything that can produce one.

Because each iterator carries its own cursor, several traversals can run over the same collection at once without stepping on each other. And a single structure can offer many orders — forward, reverse, in-order, filtered — simply by handing back different iterators.

It is so fundamental that most languages bake it into syntax: for…of and Symbol.iterator, IEnumerable, Iterable. The pattern is often invisible precisely because it succeeded.

How it works

Variations

External (active) iterator
The client drives the walk, calling next() when it's ready. Maximum control — you can pause, interleave, or stop early.
Internal (passive) iterator
The collection drives the walk and calls back into client code, like forEach. Less flexible, but the traversal logic lives in one place.
Lazy / generator iterator
Elements are produced on demand rather than materialized up front, which makes infinite and streamed sequences finite in memory.
Bidirectional & random-access
A richer cursor adds prev() or index jumps, trading the minimal contract for the ability to move both ways.
Composite traversal
One iterator walks a tree of nodes uniformly, hiding whether each node is a leaf or a branch.

Trade-offs

Pros

  • Callers traverse any collection through one interface, ignorant of its storage.
  • The container stays encapsulated — internals never leak to the client.
  • Multiple independent cursors can run over the same collection at once.
  • New traversal orders are just new iterators, not new client code.

Cons

  • An extra object and indirection for what a plain index loop would do.
  • Mutating the collection mid-traversal can invalidate the cursor.
  • External iterators leak subtle contract details — call order, exhaustion, reuse.
  • Over a tiny fixed array it's ceremony with little payoff.

When to use it

Reach for it when

  • Clients must traverse a collection without knowing how it stores elements.
  • The same code should walk several different structures interchangeably.
  • You want more than one active traversal, or several traversal orders.

Avoid when

  • The structure is a small, fixed array a direct loop reads plainly.
  • The language already gives you iteration you'd only be wrapping.
  • Callers genuinely need the concrete type's own richer operations.

Code sketch

TypeScript — an external iterator over a sealed collection 
interface Iterator<T> {
  hasNext(): boolean;
  next(): T;
}

class ArrayIterator<T> implements Iterator<T> {
  private i = 0;
  constructor(private readonly items: readonly T[]) {}

  hasNext(): boolean {
    return this.i < this.items.length;
  }
  next(): T {
    if (!this.hasNext()) throw new Error("iterator exhausted");
    return this.items[this.i++];
  }
}

// The collection stays sealed behind iterator()
class Bag<T> {
  private items: T[] = [];
  add(x: T) { this.items.push(x); }
  iterator(): Iterator<T> { return new ArrayIterator(this.items); }
}

const bag = new Bag<number>();
bag.add(1); bag.add(2);
const it = bag.iterator();
while (it.hasNext()) console.log(it.next()); // 1, then 2
