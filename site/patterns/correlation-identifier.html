Correlation Identifier

Stamps every request with a unique token that each downstream reply, retry, or log line copies verbatim — so a flood of concurrent, asynchronous messages on a shared channel can still be traced back to the one conversation that produced them.

Messaging Lens

What it is

A correlation identifier is a token — a GUID, a monotonic sequence number, or a business key already known to be unique — that a sender stamps into a message's header when it fires off a request. Every participant that produces a downstream message tied to that request copies the same token into whatever it emits, unchanged. Its shape means nothing to the messaging infrastructure; it exists purely so a listener can ask "which conversation does this belong to?" and get one unambiguous answer.

The forces it resolves show up wherever request and reply aren't connected by an open call stack or a synchronous socket. A client fires many requests at a shared queue or topic before any answers return; replies land in whatever order the network and the receivers happen to produce, interleaved with traffic from other clients and other requests. Without something in the message declaring its lineage, a listener has no reliable way to say which pending request a given reply belongs to — and a match that resolves incorrectly under load is worse than one that fails loudly.

The pattern generalizes past a single request/reply pair. Thread the same identifier through every hop of a longer chain — a message that fans out to several services, a saga's sequence of compensating steps, a request that crosses five microservices before it produces a response — and every log line, span, and metric emitted along the way can be filtered down to the one logical operation, even though the systems that emitted them share nothing else in common.

It's deliberately distinct from a message ID. A message ID names this one message and dies with it; a correlation ID names the conversation or operation the message is part of, and many messages across many systems can legitimately share it.

How it works

Variations

Message-header correlation ID
The classic form — a GUID placed in a header field (JMSCorrelationID, AMQP's correlation_id) that the responder copies straight from the request into the reply.
Business key as correlation ID
Reuse a domain identifier that's already unique — an order id, a session id — instead of minting a synthetic token. Cheaper, but only safe when that key genuinely can't repeat across concurrent conversations.
Scatter-Gather aggregation key
One correlation id ties every parallel reply back to a single fan-out request, so the aggregator knows when it has collected them all.
Distributed trace ID
The same idea propagated across every hop of a call chain (W3C traceparent, X-Request-Id) so logs and spans from otherwise unrelated services join into one trace.
Saga correlation
One id threads every step of a long-running saga, so its completion, compensation, and timeout handlers all know which instance they belong to.

Trade-offs

Pros

  • Lets one channel or queue carry many concurrent conversations without cross-talk between them.
  • Replies can arrive in any order, or interleaved with unrelated traffic, and still land on the right pending request.
  • Costs almost nothing to add — one extra header field, copied verbatim at every hop.
  • Extends directly into distributed tracing: the same mechanism stitches a whole multi-service call into one story.

Cons

  • Every hop must copy the id correctly — one component that drops or regenerates it silently breaks the chain.
  • The requestor must track pending correlation ids itself, with timeouts, or a reply that never arrives leaks memory forever.
  • A collision or an accidentally reused id attributes a reply to the wrong requestor.
  • It only marks messages as related — it provides no ordering and no guarantee that every expected reply actually arrived.

When to use it

Reach for it when

  • Multiple requests are in flight at once over the same asynchronous channel or queue.
  • Replies can arrive out of order, or interleaved with unrelated messages, and still must land on the right caller.
  • One logical operation spans several services or processes and needs to be traced as a single unit.

Avoid when

  • The exchange is a synchronous request/reply call — the open connection already correlates request and response for free.
  • Only one request is ever outstanding at a time, so there's nothing to disambiguate.
  • You need the workflow around the correlated messages managed, not just the matching — reach for Scatter-Gather or Saga.

Code sketch

TypeScript — matching replies to pending requests 
interface Reply { correlationId: string; payload: unknown; }

class CorrelatingClient {
  private pending = new Map<string, (reply: Reply) => void>();

  async request(payload: unknown): Promise<Reply> {
    const correlationId = crypto.randomUUID();
    return new Promise<Reply>((resolve) => {
      this.pending.set(correlationId, resolve);
      channel.send({ correlationId, payload });
    });
  }

  // Called whenever a message arrives on the reply channel.
  onMessage(msg: Reply) {
    const resolve = this.pending.get(msg.correlationId);
    if (!resolve) return; // unknown, late, or already-handled reply
    this.pending.delete(msg.correlationId);
    resolve(msg);
  }
}
