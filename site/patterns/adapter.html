Adapter

Wraps a class you can't change in the interface a client already speaks — so two components with mismatched APIs snap together without either side knowing about the other.

GoF Structural

What it is

An adapter is a class that implements the interface a client expects — the target — and forwards each call to an object with a different, incompatible interface — the adaptee. The client talks only to the target; the adapter quietly translates method names, argument shapes, and return values behind it.

The force it resolves is a fixed interface mismatch. You have a class that does exactly what you need — a third-party SDK, a legacy module, a library whose signatures don't line up with your code — but you can't reshape it. The client is written against a stable interface you don't want to churn, and the adaptee is owned by someone else. Neither side can move toward the other, so a piece in the middle does the reconciling.

The common form is the object adapter: it holds a reference to the adaptee and delegates through composition, which lets one adapter wrap any subtype. A class adapter instead inherits from both, but that needs multiple inheritance and couples adapter to a concrete adaptee. Composition is the pragmatic default.

How it works

Variations

Object adapter
Holds the adaptee by reference and delegates through composition — flexible, and the same adapter works for any subtype of the adaptee.
Class adapter
Inherits from both the target and the adaptee at once. Needs multiple inheritance and binds tightly to one concrete adaptee, so it's rarer in practice.
Two-way adapter
Implements both interfaces, so the object is usable as either type — handy when two subsystems each expect the other's shape.
Default (interface) adapter
Supplies empty implementations of a wide interface so subclasses override only the few methods they care about — the shape behind Java's MouseAdapter.

Trade-offs

Pros

  • Lets existing, incompatible classes collaborate without touching either one's source.
  • Confines conversion logic to one place, keeping client code free of vendor quirks.
  • Makes legacy and third-party code reusable behind a familiar interface.
  • Swappable — a new adaptee needs only a new adapter, and the client is untouched.

Cons

  • Adds a layer of indirection and one more class to keep in sync with two interfaces.
  • Deep or lossy gaps force awkward, leaky translations that hide behind a clean signature.
  • Overuse papers over real design mismatches that should be fixed at the source.
  • A thick adapter tends to accrete business logic that doesn't belong in a translator.

When to use it

Reach for it when

  • You must integrate a third-party or legacy class whose interface you can't change.
  • A client expects one interface but the implementation you want offers another.
  • You want to keep vendor-specific APIs from leaking into your domain code.

Avoid when

  • You control both sides — just align the interfaces directly instead of bridging them.
  • The problem is a whole subsystem's complexity, not one interface — reach for a Facade.
  • You need abstraction and implementation to vary independently — that's a Bridge.

Code sketch

TypeScript — an adapter around a third-party SDK 
// What our checkout code already speaks
interface PaymentProcessor {
  pay(amountCents: number): Promise<void>;
}

// A third-party SDK we cannot modify
class VendorSdk {
  async charge(dollars: string, currency: string): Promise<{ ok: boolean }> {
    /* ...remote call... */
    return { ok: true };
  }
}

// Adapter: exposes PaymentProcessor, delegates to VendorSdk
class VendorAdapter implements PaymentProcessor {
  constructor(private readonly sdk: VendorSdk) {}

  async pay(amountCents: number): Promise<void> {
    const dollars = (amountCents / 100).toFixed(2);
    const res = await this.sdk.charge(dollars, "usd");
    if (!res.ok) throw new Error("charge failed");
  }
}

// Client depends only on the target interface
const processor: PaymentProcessor = new VendorAdapter(new VendorSdk());
await processor.pay(2500);
