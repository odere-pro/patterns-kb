Visitor

Packages a new operation as an object you push through a stable class hierarchy — so you can teach a fixed set of types new tricks without editing a single one of them.

GoF Behavioral

What it is

The Visitor pattern moves an operation out of the classes it works on and into a separate object. Each element in a class hierarchy exposes a single accept(visitor) method; the visitor carries one method per concrete element type. Add a new operation by writing a new visitor — the element classes never change.

The force it resolves is the tension between two axes of change. A class hierarchy is easy to extend with new types (add a subclass) but painful to extend with new operations (edit every subclass). When the set of types is stable but operations keep multiplying — pretty-printing, type-checking, cost estimation, serialization — Visitor flips the cost model so new operations are cheap and localized.

It works by double dispatch. A single virtual call picks one type; Visitor uses two — accept resolves the concrete element, then the callback resolves the concrete operation. The pair together selects exactly the right code without a chain of type checks.

The price is that adding a new element type now touches every visitor. Visitor is a bet that types will stay put while operations grow.

How it works

Variations

Classic double dispatch
Every element implements accept and calls back the visitor's type-specific method. Fully type-safe, but the object structure must know about the visitor interface.
Acyclic Visitor
Splits the monolithic visitor interface into one small interface per element, so a visitor implements only the types it cares about and new element types don't force a recompile of every visitor.
Reflective / dynamic Visitor
Skips accept and dispatches on runtime type — instanceof checks or pattern matching. Less boilerplate, but you lose the compiler's exhaustiveness guarantee.
Default (base) Visitor
Provides no-op defaults for every element so a concrete visitor overrides only the handful of nodes it needs — handy over large or generated hierarchies.

Trade-offs

Pros

  • Add new operations without touching any element class.
  • Keeps a related operation's logic in one object instead of scattered across the hierarchy.
  • Can accumulate state across a whole traversal — totals, reports, symbol tables.
  • Dispatch is type-safe on the concrete element, with no manual type switching.

Cons

  • Adding a new element type forces a change to every existing visitor.
  • Double-dispatch boilerplate: every element needs an accept method.
  • Visitors often need access to element internals, weakening encapsulation.
  • Overkill when there is only one operation, or when it belongs on the element itself.

When to use it

Reach for it when

  • The set of element types is stable, but you keep adding new operations over them.
  • An operation doesn't belong on the elements — serialization, pretty-printing, type-checking.
  • You want a family of related operations gathered together, separate from the data.

Avoid when

  • New element types appear often — every one drags in a change to every visitor.
  • There is a single operation, or it sits naturally as a method on the element.
  • A plain switch or pattern match over a small closed type set reads more clearly.

Code sketch

TypeScript — a shape visitor 
interface ShapeVisitor<T> {
  circle(c: Circle): T;
  square(s: Square): T;
}
interface Shape {
  accept<T>(v: ShapeVisitor<T>): T;
}

class Circle implements Shape {
  constructor(readonly r: number) {}
  accept<T>(v: ShapeVisitor<T>): T { return v.circle(this); }
}
class Square implements Shape {
  constructor(readonly side: number) {}
  accept<T>(v: ShapeVisitor<T>): T { return v.square(this); }
}

// A new operation — no Shape class changes needed.
class AreaVisitor implements ShapeVisitor<number> {
  circle(c: Circle) { return Math.PI * c.r ** 2; }
  square(s: Square) { return s.side ** 2; }
}

const shapes: Shape[] = [new Circle(2), new Square(3)];
const total = shapes.reduce((sum, s) => sum + s.accept(new AreaVisitor()), 0);
