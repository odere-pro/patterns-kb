Architecture

Microkernel / Plugin

Keeps a deliberately small, stable core that does almost nothing on its own, and lets every real feature — file formats, auth providers, report types — arrive later as an independently built plug-in the core discovers and wires in without a rebuild.

Architecture

What it is

The microkernel (or plug-in) architecture splits a system into a minimal core that supplies only the mechanism every user needs — lifecycle, wiring, a plug-in registry — and a set of plug-ins that supply the actual features. The core publishes a stable extension API; plug-ins implement it and register themselves, and the core dispatches to whichever ones happen to be installed without ever naming a concrete plug-in type.

The force it resolves is an open-ended feature set. A product's core behavior is fixed and must stay small, auditable, and rarely touched, but the features built on top of it keep growing — new file formats, new integrations, new report types — often built by teams or vendors who never see the core's source. Baking every feature into one monolith means every addition risks the whole system and forces a full rebuild. The microkernel instead draws a hard line: the core changes almost never, and everything volatile lives on the other side of a contract.

The name comes from operating system microkernels — Mach, seL4 — where the kernel does only messaging and scheduling and drivers, file systems, and protocol stacks run as ordinary user-space servers behind it. Eclipse's plug-in model, VS Code extensions, and browser extensions apply the same idea one layer up: install a capability without rebuilding, or even restarting, the host.

How it works

Variations

Strategy per extension point
Each plug-in is really a Strategy the core selects at runtime for one extension point — swap the implementation, and the core's call site never changes.
Static vs. dynamic loading
Plug-ins compiled and linked into the core at build time, versus discovered and loaded at runtime from a directory, manifest, or registry — dynamic loading is what makes "install a feature without a rebuild" possible.
Client-server microkernel
The pattern's OS lineage: the kernel supplies only messaging and scheduling, and drivers, file systems, and protocol stacks run as user-space servers behind it — Mach and seL4 push it to its logical extreme.
Sandboxed, out-of-process plug-ins
Run each plug-in in its own process or sandbox — browser extensions, IDE language servers — so a crashing or misbehaving plug-in can't take the host down with it.

Trade-offs

Pros

  • Keeps the always-loaded core small, stable, and easy to audit — churn lives in plug-ins, not the kernel.
  • New capability ships as a new plug-in; the core is never recompiled or redeployed to add it.
  • Independent teams or vendors build against one published contract without touching each other's code.
  • Unneeded features are simply never loaded, so the running system stays lean.

Cons

  • The plug-in API becomes the hardest thing to change — every installed plug-in depends on it.
  • Plug-ins that need to cooperate must go through the core, which can turn it into an unplanned bottleneck.
  • A bug can live in the core, in a plug-in, or in the wiring between them, which slows debugging.
  • Versioning, discovery, and load order for many independently built plug-ins becomes its own subsystem.

When to use it

Reach for it when

  • You're shipping a product whose feature set will keep growing after release, by teams you don't control.
  • Third parties or separate teams need to extend the system without access to, or a rebuild of, the core.
  • You want an always-on core to stay small and stable while volatile feature logic lives elsewhere.

Avoid when

  • The feature set is small, fixed, and known upfront — the plug-in indirection is pure overhead.
  • One team owns the whole codebase — plain modules or packages are simpler than a plug-in contract.
  • Plug-ins would need deep, ad hoc access to each other's internals — the isolation the core provides breaks down immediately.

Prevents the smell of Golden Hammer — with a plug-in per need, no team has to stretch one favorite tool to jobs it was never built for.

Code sketch

TypeScript — a minimal core and one plug-in 
interface Plugin {
  name: string;
  init(core: Core): void;
}

class Core {
  private commands = new Map<string, (arg: string) => void>();

  register(plugin: Plugin): void {
    plugin.init(this); // plug-in wires itself into the core
  }

  addCommand(name: string, handler: (arg: string) => void): void {
    this.commands.set(name, handler); // core stays ignorant of who registered
  }

  run(name: string, arg: string): void {
    const handler = this.commands.get(name);
    if (!handler) throw new Error(`no plug-in handles "${name}"`);
    handler(arg);
  }
}

// A plug-in the core has never heard of at compile time
class MarkdownExportPlugin implements Plugin {
  name = "markdown-export";
  init(core: Core): void {
    core.addCommand("export", (path) => console.log(`exporting to ${path}`));
  }
}

const core = new Core();
core.register(new MarkdownExportPlugin());
core.run("export", "./out.md");
