Architecture

Hexagonal

Wraps the core domain in ports it defines itself, then lets adapters plug in databases, UIs, and frameworks from the outside — so the business rules never depend on any of them.

Architecture

What it is

The Hexagonal architecture — Alistair Cockburn's Ports and Adapters — puts the core domain logic at the center and forces every dependency to point inward. The core defines ports: interfaces stating what it needs from the outside world (a repository, a payment gateway) and what it offers to it (a use case, a command). Adapters implement those ports from the outside — a Postgres repository, a REST controller, a CLI — and are the only code that knows about frameworks, drivers, or wire formats.

The force it resolves is infrastructure churn contaminating business rules. In a typical layered app the domain model ends up importing the ORM, the web framework's request object, or a message broker's client directly, so testing a business rule means booting a database and an HTTP server, and swapping either means touching the core. Hexagonal inverts that: the core owns the interface on its own terms, and the infrastructure has to satisfy it — never the other way round.

The hexagon itself carries no special meaning — Cockburn chose it only so the diagram wouldn't imply a "top" (UI) and "bottom" (database) the way a layered stack does. Each side of the polygon is just another port, treated symmetrically whether it's driven by a user, a scheduler, or another service.

How it works

Variations

Ports & Adapters
Alistair Cockburn's original name for the same pattern — hexagonal is just the customary picture, chosen to avoid implying a top or bottom to the architecture.
Onion Architecture
Draws the same idea as concentric rings — domain model at the center, then domain services, then application services — and states the dependency rule explicitly: nothing points outward.
Clean Architecture
Adds an explicit use-case (interactor) ring between the domain and its ports, and generalizes the dependency rule across any number of layers.
Anti-Corruption Layer
A stricter adapter that translates and validates at a bounded-context boundary, so a foreign or legacy model can't leak its shape into the core.

Trade-offs

Pros

  • Core business logic has zero dependency on frameworks, drivers, or transport — it's plain code you can unit-test with in-memory fakes.
  • Infrastructure is swappable: replace Postgres with DynamoDB, or REST with gRPC, by writing a new adapter, not touching the core.
  • Ports make every integration point an explicit, named contract instead of an implicit coupling.
  • The same core can be driven by several front doors at once — a web API, a CLI, a scheduled job — without duplicating rules.

Cons

  • More upfront ceremony: an interface, at least one adapter, and wiring for every single dependency, even trivial ones.
  • Drawing the port boundary is a real design judgment — too fine and it's noise, too coarse and infrastructure leaks back in.
  • Small, short-lived apps rarely earn back the indirection; it adds files and interfaces for a project that will never change database.
  • Nothing stops a hurried adapter from smuggling infrastructure types back across the port if the team isn't disciplined.

When to use it

Reach for it when

  • The business rules need to outlive a specific framework, database, or transport that's likely to change.
  • You want to unit-test core logic without booting a database, HTTP server, or queue.
  • Several front doors — web, CLI, batch job, another service — must drive the same core logic consistently.

Avoid when

  • The app is a thin CRUD wrapper around one database with no real domain logic worth isolating.
  • The team is small and a straightforward Layered / N-Tier structure already gives enough separation for the project's lifetime.
  • Nobody will enforce the boundary — an unpoliced port is just an interface nobody bothers to implement twice.

Prevents the smell of business rules entangled with frameworks, drivers, and wire formats — the drift toward a Big Ball of Mud where nothing can change in isolation.

Code sketch

TypeScript — a port, a core, and one adapter 
// Port: interface the core defines and owns
interface OrderRepository {
  save(order: Order): Promise<void>;
  findById(id: string): Promise<Order | null>;
}

// Core: depends only on the port, never on SQL or a driver
class PlaceOrder {
  constructor(private readonly orders: OrderRepository) {}

  async execute(order: Order): Promise<void> {
    if (order.items.length === 0) {
      throw new Error("cannot place an empty order");
    }
    await this.orders.save(order);
  }
}

// Adapter: implements the port, owns the persistence detail
class PostgresOrderRepository implements OrderRepository {
  constructor(private readonly db: Pool) {}
  async save(order: Order): Promise<void> {
    await this.db.query("insert into orders values ($1)", [order.id]);
  }
  async findById(id: string): Promise<Order | null> {
    const row = await this.db.query("select * from orders where id = $1", [id]);
    return row ? toOrder(row) : null;
  }
}

// Composition root wires the adapter into the core's port
const placeOrder = new PlaceOrder(new PostgresOrderRepository(pool));
