Architecture

Event-Driven Architecture

Components communicate by producing and reacting to events — facts about something that already happened — instead of calling each other directly, so publishers and subscribers can be built, deployed, and scaled independently.

Architecture

What it is

In an event-driven architecture, components emit events — immutable records that something happened, like OrderPlaced or PaymentCaptured — without knowing or caring who, if anyone, reacts. Interested components subscribe to the event types they care about and react asynchronously, whenever the event reaches them. No producer ever calls a consumer's method; it only asserts a fact into the world.

The force it resolves is coupling through synchronous calls. A service that invokes its collaborators directly must know their addresses, their interfaces, and be available to wait for their replies — one slow or failing dependent stalls the caller, and adding a new consumer means changing the producer's code. Events invert that: the producer publishes once, and zero, one, or a hundred consumers may react, today or after the producer has moved on.

This buys temporal, spatial, and behavioral decoupling. Producer and consumer don't need to be running at the same time, don't need to know each other's location, and the producer's contract is the event's shape, not a method signature tied to any particular subscriber's needs. New reactions — a fraud check, an analytics pipeline, a notification — plug in by subscribing, never by touching the code that raised the event.

That freedom has a cost: the price of "fire and forget" is that no one guarantees delivery order, timing, or even that a listener still exists. Event-driven architecture trades the traceability of a direct call stack for the flexibility of a system that reacts to what happened rather than dictating what happens next.

How it works

Variations

Choreography vs. orchestration
In choreography every service reacts to events on its own, with no central controller. In orchestration a coordinator — often a saga — tells each participant what to do next. Pure EDA favors choreography; orchestration returns when a business process needs an explicit, auditable sequence.
Event notification vs. event-carried state transfer
A thin event says only "this changed, go fetch details," forcing consumers to call back for the rest. A fat event carries the full new state so consumers never need to. Fatter events cut coupling further but risk staleness and duplicated data across services.
Domain Event vocabulary
The events themselves are named and shaped as business facts, not technical deltas — OrderPlaced, not RowUpdated. This is the vocabulary EDA is built out of.
Publish-Subscribe transport
The wiring mechanism underneath: producers publish to a topic, consumers subscribe to it, and a broker handles fan-out, buffering, and delivery — Kafka, SNS/SQS, RabbitMQ, EventBridge.
Event Sourcing persistence
Instead of just reacting to events in flight, the event stream itself becomes the system of record — current state is derived by replaying it. A natural, but optional, pairing with EDA.

Trade-offs

Pros

  • Producers and consumers are decoupled in time, space, and identity — add a reaction without touching the source.
  • Consumers scale, deploy, and fail independently of the producer and of each other.
  • Naturally resilient to partial outages: a down consumer doesn't block the producer, and queued events catch it up later.
  • Models the domain as a stream of things that happened, which often matches how the business actually thinks.

Cons

  • Hard to trace: one business flow scatters across independently-deployed handlers with no single call stack.
  • Eventual consistency is pervasive — consumers see stale state, and races between events are subtle.
  • Debugging and testing require standing up or simulating a broker; "where did this event come from" often means grepping logs across services.
  • Needs real operational discipline: schema versioning, idempotent consumers, dead-letter queues, and a plan for duplicate or out-of-order delivery.

When to use it

Reach for it when

  • Multiple, independently-owned parts of the system need to react to the same fact without the producer knowing who they are.
  • Producers and consumers must be deployed, scaled, and evolved on separate schedules.
  • The domain is naturally a stream of things that happened — orders placed, payments captured, inventory changed.

Avoid when

  • A handful of services run one synchronous business transaction and the caller needs an immediate, consistent answer.
  • The team can't yet operate and monitor a broker, retries, and dead-letter handling.
  • There is exactly one consumer that will ever care — a direct call is simpler and easier to trace.

Code sketch

TypeScript — a minimal in-process event bus 
type Events = {
  OrderPlaced: { orderId: string; total: number };
};

class EventBus<E extends Record<string, unknown>> {
  private handlers = new Map<keyof E, Set<(payload: any) => void>>();

  on<K extends keyof E>(type: K, handler: (payload: E[K]) => void): () => void {
    const set = this.handlers.get(type) ?? new Set();
    set.add(handler);
    this.handlers.set(type, set);
    return () => set.delete(handler); // unsubscribe
  }

  emit<K extends keyof E>(type: K, payload: E[K]): void {
    // producer never learns who ran, or how many
    for (const handler of this.handlers.get(type) ?? []) handler(payload);
  }
}

const bus = new EventBus<Events>();
bus.on("OrderPlaced", (e) => chargeCard(e.orderId, e.total));
bus.on("OrderPlaced", (e) => notifyWarehouse(e.orderId));

bus.emit("OrderPlaced", { orderId: "o-42", total: 4999 });
