Architecture

MVP

A presenter sits between a passive view and the model, pulling data out, pushing formatted results back through explicit calls, and leaving the view with no logic of its own to test.

Architecture

What it is

The Model-View-Presenter pattern splits a screen into three pieces: a model that holds the data, a view that renders it and forwards raw user input, and a presenter that sits between them. The presenter reads and writes the model, decides what the screen should show, and pushes the result into the view through an explicit interface of setters — setTitle(), showError(), setLoading(true). The view itself contains no decision logic; it renders whatever it's told and reports events like clicks straight back to the presenter.

The force it resolves is testing UI logic without a real UI. In classic desktop and early mobile toolkits, instantiating a live view in a test runner was slow or flatly impossible — a real window, a real widget tree. MVP moves every decision a screen makes (what to show on failure, how to format a value, when to enable a button) out of the view and into the presenter, which depends on the view only through a plain interface. Swap in a fake that records calls, and the presenter's entire behavior is testable with no UI framework running at all.

The trade is a granular view interface — one setter per thing the screen can display — and a presenter that calls it explicitly, call by call. There's no observing and no automatic binding: every update on screen is an imperative instruction the presenter chose to send.

How it works

Variations

Passive View
The strict form: the view holds zero logic, not even simple formatting or visibility rules — every display decision is pushed into the presenter and invoked explicitly.
Supervising Controller
A looser form: the view binds directly to the model for simple, low-risk display, and the presenter only steps in for the non-trivial behavior.
Interface-first view
The presenter is written against a view interface, never a concrete class, so tests substitute a fake that records calls instead of booting a real UI toolkit.
Presenter-per-view vs. shared presenter
Usually one presenter backs one screen, but a single presenter can coordinate several views across a multi-step flow like a wizard.

Trade-offs

Pros

  • Presenter logic is fully unit-testable without booting a real UI toolkit — swap in a fake view.
  • The view stays framework-thin, so swapping the UI layer means writing a new view against the same interface.
  • Clean separation of concerns: the view renders, the presenter decides, the model holds data.
  • Well suited to toolkits where the view layer is awkward or slow to instantiate directly in tests.

Cons

  • Every field the view can show needs an explicit setter — boilerplate grows with the screen.
  • Passive View pushes even trivial formatting into the presenter, which can bloat it.
  • Without further decomposition, a presenter for a complex screen becomes an unstructured catch-all.
  • Purely manual wiring — no observable binding — so view and presenter can drift if calls are missed.

When to use it

Reach for it when

  • Your UI toolkit is slow or awkward to instantiate in tests, and you want the screen's logic covered without one.
  • You need to swap the view technology while keeping the same presentation logic underneath.
  • You want one explicit seam where every display decision for a screen lives and can be unit tested.

Avoid when

  • The platform already gives you solid two-way data binding — MVVM removes most of the setter boilerplate MVP requires.
  • The screen is a couple of static fields; a full presenter and view-interface split is pure ceremony.
  • The codebase is already fluent in MVC and letting the view bind to the model directly is an acceptable trade there.

Code sketch

TypeScript — a presenter over a passive view 
interface TodoView {
  setItems(items: string[]): void;
  setError(message: string | null): void;
}

class TodoPresenter {
  constructor(
    private readonly view: TodoView,
    private readonly model: { load(): Promise<string[]>; add(t: string): Promise<void> },
  ) {}

  async onLoad(): Promise<void> {
    try {
      this.view.setItems(await this.model.load());
    } catch {
      this.view.setError("couldn't load todos");
    }
  }

  async onAddClicked(text: string): Promise<void> {
    if (!text.trim()) return;           // decision lives here, not the view
    await this.model.add(text);
    this.view.setItems(await this.model.load());
  }
}
