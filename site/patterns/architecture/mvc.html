Architecture

MVC

Splits an interactive application into a model that owns state and behavior, a view that renders it, and a controller that turns user input into changes to the model — so each piece changes without dragging the others with it.

Architecture

What it is

MVC divides an interactive application into three collaborating parts. The model owns application state and the rules for changing it, and knows nothing about how it's displayed. The view renders a representation of the model and forwards raw user input onward. The controller interprets that input, decides what it means, and updates the model accordingly. When the model changes, the view is told to refresh — directly, or by observing the model — so it never goes stale.

The force it resolves is tangled responsibilities. Without the split, code that decides business rules, code that lays out pixels, and code that parses clicks and keystrokes all live in the same class, and changing one risks breaking the others. Separating them lets a designer touch the view without threatening domain logic, lets the model be tested with no UI at all, and lets the same model drive more than one view — a desktop window and a printable report, or a page and an API.

The pattern comes from Smalltalk-80, where model and view were wired with a live observer link: the model notified its views the instant its state changed, and the UI updated in place. Web frameworks reshaped this into a request/response cycle — a controller handles one HTTP request, updates the model, and picks a view to render the response — trading the constant live push for a stateless round trip.

That reshaping is also where the pattern strains. A controller that both parses input and holds view-specific logic grows fat, and a view that needs to know too much about the model's shape gets brittle. MVP and MVVM exist because those two seams — the controller's role, and how the view learns of change — are exactly where implementations diverge.

How it works

Variations

Classic (Smalltalk-80) MVC
Model and view are wired through a live observer link — the model pushes change notifications straight into every view watching it, in-process and in real time.
MVP
Swaps the controller for a presenter that owns all presentation logic; the view becomes passive, exposing only simple getters and setters the presenter drives.
MVVM
Introduces a view-model that exposes bindable, observable state; the view declares bindings instead of being pushed to or pulled from imperatively.
Request-response (web) MVC
Each HTTP request is one pass through the triad — controller updates the model, then renders a template — with no persistent live link between model and view.
Hierarchical MVC (HMVC)
Nests whole MVC triads inside one another so independent sub-UIs (a widget, a panel) each keep their own model, view, and controller.

Trade-offs

Pros

  • Domain logic in the model is testable without touching any UI.
  • The same model can back multiple views — a page, a widget, an export.
  • Designers and front-end code can change the view without risking business rules.
  • A well-understood vocabulary — most teams and frameworks already speak it.

Cons

  • The controller easily absorbs logic that belongs in the model or the view, and swells.
  • The boundary between view and controller is genuinely fuzzy and gets drawn differently by every team.
  • Live model-to-view notification needs its own wiring — usually another pattern (Observer) underneath.
  • Web frameworks bend the names to fit request/response, so "MVC" means something different in each one.

When to use it

Reach for it when

  • The application has real domain logic worth testing independently of the UI.
  • More than one presentation must show the same underlying state.
  • Input handling, business rules, and rendering are already tangled and need separating.

Avoid when

  • The UI is trivial — a static page or a single form doesn't need three layers.
  • Your framework's binding story fits MVVM far better than a hand-rolled controller.
  • You need the view fully passive and unit-testable in isolation — that's MVP's job.

Code sketch

TypeScript — a minimal triad 
interface State { count: number; }
type Action = "increment" | "reset";

class Model {
  private state: State = { count: 0 };
  private listeners = new Set<(s: State) => void>();

  get(): State { return this.state; }
  set(next: State) {
    this.state = next;
    for (const fn of this.listeners) fn(next);
  }
  onChange(fn: (s: State) => void) { this.listeners.add(fn); }
}

class Controller {
  constructor(private model: Model) {}
  handle(action: Action) {
    const s = this.model.get();
    if (action === "increment") this.model.set({ count: s.count + 1 });
    if (action === "reset") this.model.set({ count: 0 });
  }
}

// View renders on every model change; controller owns input handling.
const model = new Model();
const view = { render: (s: State) => console.log(`count: ${s.count}`) };
model.onChange((s) => view.render(s));

const controller = new Controller(model);
controller.handle("increment"); // view logs "count: 1"
