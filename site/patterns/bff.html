Backend-for-Frontend

Gives each class of client — mobile app, web app, third-party partner — its own backend, shaped around exactly the calls and payloads its screens need, instead of forcing every client through one general-purpose API.

Distributed Routing & Scale

What it is

A Backend-for-Frontend (BFF) is a thin backend service built and owned by the same team as one specific client — the iOS app, the web app, a partner integration — sitting between that client and the shared downstream services. It calls whatever it needs, aggregates the results, and returns exactly the shape that client's screens expect.

The force it resolves is a one-size-fits-none API. Mobile wants a small, denormalized payload to save battery and bandwidth; the web app wants a richer, more graph-shaped response for a dense dashboard; a partner needs a stable, versioned contract that can't shift under it. A single general-purpose API serving all three accretes optional fields, client-detection branches, and version flags until every change risks breaking someone else's consumer.

The BFF splits that API along client lines instead. Each frontend gets a backend that speaks only its own language, evolves on its own release schedule, and is owned by the people who actually feel the pain when it's wrong — the frontend team, not a shared platform team fielding requests from three directions at once.

It doesn't replace the shared services underneath; it sits in front of them as a translation and aggregation layer, so the services stay generic while the client-specific shaping lives at the edge, one BFF per client.

How it works

Variations

Per-platform BFF
One BFF per platform — iOS, Android, web — each tuned to that platform's payload size, latency budget, and release cadence.
GraphQL BFF
The BFF exposes one GraphQL schema instead of several REST endpoints, so each screen queries exactly the fields it needs in a single round trip.
Team-owned BFF
The BFF lives in the same repo and on-call rotation as the frontend it serves, so the frontend team ships both without waiting on a shared backend team.
Edge-deployed BFF
Runs as an edge function (Lambda@Edge, Cloudflare Workers) close to the client, so aggregation happens near the request instead of after a long backbone hop.

Trade-offs

Pros

  • Each client gets a payload shaped exactly for its screens — no unused fields, no client-side reshaping.
  • Frontend teams own their BFF and ship on their own schedule, without queuing behind a shared backend roadmap.
  • Aggregates several downstream calls into one round trip, cutting chattiness for bandwidth-constrained clients.
  • Keeps client-specific branching out of the shared services, which stay generic and simpler to reason about.

Cons

  • Multiplies backend services — one per client type — each with its own deploy, monitoring, and on-call.
  • Cross-cutting logic (auth, error mapping, caching) tends to duplicate across BFFs unless it's factored into a shared library.
  • The same downstream data can drift into slightly different shapes across BFFs if no one's watching for it.
  • Adds one more hop and one more moving part between the client and the services it ultimately needs.

When to use it

Reach for it when

  • You serve genuinely different client types whose data shape, aggregation, or performance needs diverge sharply.
  • A shared general-purpose API has accreted so many client-specific branches that no one client is well served.
  • A frontend team wants to own its own backend contract and ship on its own schedule.

Avoid when

  • There's only one client type — a single well-designed API is simpler, with nothing to specialize for.
  • Clients' needs are similar enough that optional fields or partial responses on one API cover them without duplication.
  • You lack the operational capacity to run, deploy, and monitor another service per client.

Code sketch

TypeScript — a mobile BFF aggregating two services 
// Mobile BFF: one call, shaped for the app's home screen
app.get("/mobile/home/:userId", async (req, res) => {
  const { userId } = req.params;

  // Fan out to the shared backend services in parallel
  const [profile, orders] = await Promise.all([
    userService.getProfile(userId),
    orderService.getRecentOrders(userId, { limit: 3 }),
  ]);

  // Reshape into exactly what the mobile UI renders — nothing more
  res.json({
    name: profile.displayName,
    avatarUrl: profile.avatarSmallUrl,  // small asset, not the desktop one
    recentOrders: orders.map(o => ({
      id: o.id,
      total: o.totalFormatted,
      status: o.status,
    })),
  });
});
