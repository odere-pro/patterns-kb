Network · Resilience

Rate Limiter

Caps how many requests a client can make in a fixed window, rejecting or delaying the rest so no single caller can overrun shared capacity.

Distributed Resilience

What it is

A rate limiter caps how many requests a caller — an API key, a user, an IP address, a downstream service — may make within a fixed time window. Cross the limit and the excess is turned away, typically with an HTTP 429 Too Many Requests and a Retry-After header telling the caller when it's safe to try again.

The force it resolves is finite, shared capacity. A database, a paid third-party API with a quota, a compute cluster — all have a ceiling well below what every client combined could throw at them. Without a cap, one noisy client, a runaway retry loop, or an ordinary traffic spike can starve every other consumer and drag the whole system down with it.

It's deliberately a shedding strategy, not an absorbing one — the excess is dropped, not queued. That's what separates it from Queue-Based Load Leveling, which buffers the overflow instead of rejecting it, and from Backpressure, which asks the upstream caller to slow down cooperatively rather than refusing it outright.

How it works

Variations

Token Bucket
Tokens accumulate at a fixed rate up to a cap; each request spends one, so bursts up to the bucket size are allowed as long as tokens remain.
Leaky Bucket
Requests queue and drain at a constant rate instead of bursting through — it smooths traffic into a steady outflow rather than admitting spikes.
Fixed window counter
Count requests per key inside a whole-second or whole-minute window. Simple, but a burst straddling the window boundary can momentarily double the intended rate.
Sliding window (log or counter)
Weight the current and previous window by elapsed time, or keep a rolling log of timestamps, to smooth out the boundary problem at some extra memory cost.
Distributed rate limiting
Back the counter with a shared store — Redis, Memcached — so the limit holds across a fleet of instances instead of resetting per process.

Trade-offs

Pros

  • Protects shared capacity from being monopolized by one noisy or abusive client.
  • Keeps load on downstream systems and databases predictable and bounded.
  • Cheap to implement and reason about — a counter, a clock, and a key.
  • Gives callers an explicit, machine-readable signal to back off instead of silent failure.

Cons

  • Legitimate bursts get rejected right alongside abusive ones unless tuned carefully.
  • Per-instance counters miscount without a shared store, so distributed enforcement needs infrastructure.
  • Choosing window size and rate is a tuning exercise, not a fixed answer.
  • A hard cap turns a rare spike into visible errors rather than degraded-but-working service.

When to use it

Reach for it when

  • A shared resource — an API, a database, a downstream service — must survive noisy or abusive clients.
  • You need to enforce a fair-use policy or a paid-tier quota.
  • Predictable load matters more than accepting every request that arrives.

Avoid when

  • You need to absorb bursts rather than reject them — a queue buffers, a limiter sheds.
  • The bottleneck is concurrent work in flight, not request rate — isolate it with a Bulkhead instead.
  • The caller can be told to slow down cooperatively — that's Backpressure, not a hard cap.

Code sketch

TypeScript — a per-key token bucket 
class TokenBucket {
  private tokens: number;
  private last = Date.now();

  constructor(private capacity: number, private refillPerSec: number) {
    this.tokens = capacity;
  }

  private refill() {
    const now = Date.now();
    const elapsed = (now - this.last) / 1000;
    this.tokens = Math.min(this.capacity, this.tokens + elapsed * this.refillPerSec);
    this.last = now;
  }

  tryConsume(cost = 1): boolean {
    this.refill();
    if (this.tokens < cost) return false;
    this.tokens -= cost;
    return true;
  }
}

// One bucket per API key
const buckets = new Map<string, TokenBucket>();
function limiterFor(key: string): TokenBucket {
  let b = buckets.get(key);
  if (!b) buckets.set(key, (b = new TokenBucket(100, 10))); // burst 100, refill 10/s
  return b;
}

if (!limiterFor(apiKey).tryConsume()) {
