Case study

Bitly

A URL shortener turns a long link into a short code and redirects visitors back to the original. Almost all the load is redirects, so the design is read-first: generate unique codes cheaply, then serve them from memory and the edge.

Design

Understanding the problem

A URL shortener accepts a long URL and returns a short one — short.ly/abc123 — that redirects to the original. It is a small surface with one dominant property that decides everything downstream: the read-to-write ratio is enormous. A link is created once and then clicked thousands of times, so the interesting engineering is not in creating codes but in redirecting fast, cheaply, and at scale. This is the entry point for the read-heavy playbook covered in depth by the Scaling Reads theme.

Requirements

Functional

  1. Submit a long URL and receive a short URL in return.
  2. Optionally choose a custom alias, and optionally set an expiration date.
  3. Visit a short URL and be redirected to the original.

Out of scope: accounts and click analytics — named explicitly so the design stays narrow.

Non-functional

  • Uniqueness — every short code maps to exactly one long URL.
  • Latency — redirects under 100 ms.
  • Availability — 99.99%, favoured over strict consistency (a stale mapping is harmless; a dropped redirect is not).
  • Scale — 1B stored URLs, 100M daily actives, with reads dwarfing writes by roughly 1000:1.

Back-of-envelope

Reads. 100M DAU × ~5 redirects/day ≈ 500M redirects/day ≈ 5,800 reads/sec on average. Traffic is spiky, so budget for a peak roughly 100× the mean — call it ~600k reads/sec. That number, not the write path, is what the architecture has to survive.

Writes. New links arrive far more slowly — on the order of ~100k/day, about 1 write/sec. Writes are a rounding error next to reads.

Storage. A row is a short code, the long URL, timestamps and an optional alias — round to ~500 bytes. 1B rows × 500 bytes ≈ 500 GB, which fits comfortably on one modern SSD-backed instance. The dataset does not force sharding; the read rate forces caching.

Core entities

Three entities, kept deliberately thin:

  • ShortLink — the mapping itself: short_code (primary key), long_url, created_at, optional expires_at, optional custom_alias.
  • Long URL — the destination the user wants to shorten.
  • User — the creator, present only so custom aliases and expiry can be attributed; out of scope here beyond that.

The interface

A small REST surface — one verb per requirement:

HTTP — create and redirect 
POST /urls
{ "long_url": "https://example.com/some/very/long/path",
  "custom_alias": "optional",
  "expiration_date": "optional" }
→ 200 { "short_url": "https://short.ly/abc123" }

GET /{short_code}
→ 302 Found
  Location: https://example.com/some/very/long/path
  (410 Gone if the link has expired)

The redirect is a 302, not a 301: a 301 is cached by the browser and never returns to the server, which would forfeit expiry and any future analytics. 302 keeps every click flowing through the service.

How the system is built

Because reads and writes have wildly different shapes, split the server in two and scale them independently. The Write Service mints a code and stores the mapping; the Read Service does nothing but resolve codes to URLs, fronted by a cache and, for the hottest links, the edge. The store is a single relational instance with a read replica for failover — the 500 GB dataset does not need sharding.

Deep dives

1 · Generating short codes that never collide

Codes must be unique, short, and cheap to make.

  • Naïve — prefix of the URL. Take the first few characters of the long URL. It collides instantly: two linkedin.com/in/… links share a prefix and become indistinguishable on redirect. Rejected.
  • Hashing. Hash the canonicalized URL (SHA-256), base62-encode it, keep the first 8 characters — 62⁸ ≈ 218 trillion codes. Base62 (a–z, A–Z, 0–9) avoids + and /, which mean other things in URLs. Deterministic hashing gives free deduplication but is guessable without an HMAC salt, and collisions still need a UNIQUE constraint plus bounded retries.
  • Counter + base62 (chosen). Keep a global counter, base62-encode its value. Redis's single-threaded atomic INCR hands every caller a distinct number, so there is no collision to check for — cheaper and simpler than hashing. 1B in base62 is only 6 characters. The catch is that sequential codes are enumerable; mitigate with a reversible XOR-with-secret before encoding, or accept it since short links are meant to be shared.

2 · Making redirects fast

An un-indexed lookup is a table scan — fatal at a billion rows. A B-tree primary-key index on short_code gives O(log n) lookups, but a disk-backed instance tops out near ~100k IOPS while we need ~600k reads/sec. So put the working set in memory: a cache-aside layer (Redis) in front of the store serves hits in ~100 ns — roughly 1000× faster than SSD — and only misses touch the database. For globally popular codes, push the mapping to a CDN and run the redirect at the edge, so those requests never reach the origin at all. The cache TTL is set at or below the link's expiry so stale entries evict themselves.

3 · Scaling to 1B URLs and 100M DAU

Storage is the easy part (500 GB on one instance, with replication for availability). The real coordination problem is the counter: many stateless Write Service instances must agree on one sequence. A single Redis counter with atomic INCR solves it, and batching makes it nearly free — each instance claims 1000 values at once and hands them out locally, cutting Redis traffic by 1000×. Across regions, hand each region a disjoint counter range (region A: 0–1B, region B: 1B–2B) so writes never coordinate across the WAN. If Redis loses a few unreplicated values on failover, that is fine — we need uniqueness, not an unbroken sequence, and the database's UNIQUE constraint via a conditional write is the final backstop.

Limitations & trade-offs

What it buys

  • Redirects served from memory and the edge, comfortably under 100 ms at ~600k req/s.
  • Collision-free codes with no read-time checks, thanks to the atomic counter.
  • Read and write paths scale independently, matched to a 1000:1 load asymmetry.

What it gives up

  • Counter-based codes are enumerable unless obfuscated — a real concern if links are meant to be private.
  • Availability is favoured over consistency: a just-created link may be briefly missing from a replica or edge.
  • Cache and CDN invalidation add moving parts; expiry relies on TTLs lining up rather than instant deletes.

What's expected at each level

  • Mid-level — a working end-to-end flow (submit → generate → store → redirect), one sound uniqueness scheme, why 302 over 301, and the insight that caching helps a read-heavy load.
  • Senior — drives the read-first framing, argues hashing vs. counter trade-offs unprompted, details a caching and invalidation strategy, and separates read/write services under asymmetric load.
  • Staff+ — structures the whole design read-heavy from the first minute, and volunteers multi-region counter ranges, Redis failover behaviour, enumeration risk, and expiry/cleanup as operational realities.
