Case study

YouTube

A video platform lets anyone upload a file and lets everyone else watch it. Only two verbs, but the file is tens of gigabytes, the viewer is on an unreliable connection, and one popular clip is read millions of times for every single write. The whole design lives in that gap between a trivial surface and a punishing load.

Design

Understanding the problem

Strip the platform down and there are exactly two interactions: a user uploads a video, and a user watches one. That tiny functional surface is deceptive — almost all the engineering pressure comes from the shape of the load around those two verbs. Uploads are single files in the tens of gigabytes. Playback has to stay smooth for a viewer whose bandwidth drops mid-stream. And a single video is written once but read on a scale that ranges from a handful to hundreds of millions. So the requirements to pin down first are not what the system does but how much and how reliably. The upload half overlaps heavily with the large-file mechanics of a Dropbox-style design; the watch half is a streaming problem in its own right.

Requirements

Functional

  1. A user can upload a video.
  2. A user can watch (stream) a video.

Out of scope: view counts, search, comments, recommendations, and channel/subscription management — named explicitly so the design stays on the upload-and-playback core.

Non-functional

  • Availability — favoured over consistency; a stale or briefly-missing video is tolerable, an outage is not.
  • Large files — uploads and playback for videos in the tens of gigabytes.
  • Low-latency streaming — playback starts quickly and stays smooth, even on low-bandwidth connections.
  • Scale — roughly 1M videos uploaded and 100M videos watched per day.
  • Resumable uploads — an interrupted upload resumes where it stopped rather than restarting.

Back-of-envelope

Writes (uploads). 1M uploads/day is only ~12 uploads/sec on average — trivial as a request rate. What matters is not the count but the payload: each is a multi-gigabyte transfer, so the write path is bandwidth-bound, not QPS-bound.

Reads (watches). 100M watches/day ≈ ~1,200 watches/sec on average, spiky well above that at peak. But a "watch" is not one request — the player pulls a manifest and then a steady stream of short segment files for the whole runtime, so the edge sees orders of magnitude more requests than the raw watch count. The read-to-write ratio is the defining asymmetry: a viral clip is uploaded once and watched millions of times.

Storage. Two very different piles. Metadata is tiny — ~365M new rows/year at roughly a kilobyte each is a few hundred GB/year, comfortable for a partitioned store. Video bytes are enormous — even a conservative ~1 GB of stored renditions per upload puts new blob data near a petabyte a day. That gap is the load-bearing fact: bytes go to a blob store, metadata goes to a database, and the two are designed independently.

Core entities

A short list is enough to anchor the design:

  • User — either an uploader or a viewer.
  • Video — the media itself: the original source plus the derived segments, each a few-seconds playable unit stored in several formats (a codec + container combination).
  • VideoMetadata — the queryable record: videoId, uploaderId, name, description, the blob-store URLs, a pointer to the manifest, and a chunks list that tracks upload progress.
  • Manifest — the streaming index: a primary manifest listing every available format, and per-format media manifests listing the segment URLs the player fetches in order.

The interface

The naïve first cut — POST /upload carrying { Video, VideoMetadata } — routes multi-gigabyte payloads through the application tier. It doesn't survive contact with the requirements, so the upload endpoint evolves into a request for an upload slot rather than a byte sink:

HTTP — upload, track progress, watch 
POST /presigned_url
{ "name": "...", "description": "...", "chunks": [{ "fingerprint": "...", "status": "NotUploaded" }] }
→ 200 { "videoId": "...", "urls": ["https://blob.example/upload/part/1?sig=...", ...] }
     (client then uploads bytes DIRECTLY to the blob store, one part at a time)

PATCH /videos/{videoId}/chunks
{ "partNumber": 3, "eTag": "..." }        // relay each S3 part ack so the server marks it Uploaded
→ 200 { "status": "Uploaded" }

GET /videos/{videoId}
→ 200 VideoMetadata                     // carries the manifest URL the player needs to stream

Note what left the request body: the video bytes. The server hands back a signed URL and the client streams the file straight to storage. GET /videos/{videoId} returns only metadata — the content lives in the blob store and the manifest inside that metadata tells the player how to reach it.

How the system is built

Every request enters through an API gateway and load balancer that handle routing, authentication, and rate limiting, then reach a stateless Video Service. On upload, the Video Service does not touch the bytes: it writes the metadata row and issues a presigned URL so the client uploads each part directly to the blob store, keeping the application tier thin. That blob store is an object store — the only thing that scales to petabytes of video a day. When an upload completes, the store emits an event that fires the processing pipeline. On watch, the Video Service returns metadata — served through a cache so hot videos never hit the database — and the player then pulls the manifest and segments from a CDN at the edge, so the origin is out of the streaming loop entirely.

Deep dives

1 · Processing a video for adaptive bitrate streaming

Storing the raw file is a non-starter: different devices need different formats, and a single giant file can't be streamed a few seconds at a time. So the upload is post-processed into small segments, each transcoded into several codec + container formats, plus manifest files that index them. That work is a pipeline: split the source with a tool like ffmpeg, transcode each segment (and process audio and transcripts), generate the primary and media manifests, then mark the upload complete. Crucially the segments have no cross-dependencies, so the pipeline is really a DAG whose transcode stage — the expensive, CPU-bound step — fans out across a fleet of worker nodes pulling independent segments in parallel. A workflow orchestrator such as Temporal builds the graph and schedules the workers; intermediate artifacts live in the blob store and workers pass URLs, not bytes, to keep the coordination cheap. The payoff at watch time is adaptive bitrate streaming: the player reads the manifest, picks a rendition for current conditions, and — if bandwidth drops mid-clip — quietly switches to more-compressed, lower-resolution segments instead of stalling.

2 · Resumable uploads

An upload of tens of gigabytes will be interrupted; restarting from zero is unacceptable. So the client splits the file into ~5–10 MB chunks, each with a fingerprint hash, and registers that chunk list in VideoMetadata with every entry marked NotUploaded. It then uploads each part directly to the blob store via multipart upload; the store acknowledges each part with a part number and an ETag, which the client relays back (PATCH /videos/{id}/chunks) so the server verifies it and flips that chunk to Uploaded. Resuming is then trivial: fetch the metadata, see which chunks already landed, and skip them. When the client finally calls CompleteMultipartUpload, the store emits exactly one object-level event that kicks off processing — an idempotent trigger, so retries and duplicate notifications never launch the pipeline twice.

3 · Scaling reads, hot videos, and streaming distance

The Video Service is a stateless tier of presigned-URL and point-lookup handlers, so it scales out behind the load balancer without ceremony. Metadata sits in Cassandra, partitioned by videoId — the design only needs point look-ups, never bulk scans, so a horizontally-partitioned leaderless store is a clean fit. The classic failure mode is the hot key: a viral video pins its partition to one node that then buckles under the read load. Two fixes stack — replicate popular metadata onto a few extra nodes so several can share the queries, and put a distributed cache keyed by videoId in front so the hottest records are served from memory and never reach the database at all. The video bytes have their own hot-spot problem: a bucket lives in one region, so a viewer far away sees slow starts and buffering. The answer is the CDN — push segments and manifests to edge servers near the viewer, and once everything a player needs is at the edge, streaming never touches the origin. Upload bursts, meanwhile, are absorbed by a queue in front of the pipeline whose depth drives elastic worker scaling.

Limitations & trade-offs

What it buys

  • Multi-gigabyte uploads never pass through an app server — presigned direct-to-blob keeps the tier thin and cheap to scale.
  • Adaptive bitrate keeps playback smooth as bandwidth swings — the player downgrades resolution rather than stalling.
  • Read scale is handled at the edge and in cache — hot videos are insulated from both the database and the origin bucket.
  • Transcoding, the one expensive job, is embarrassingly parallel across an elastic worker fleet.

What it gives up

  • The client becomes a heavy participant — chunking, ETag bookkeeping, manifest parsing, and bitrate selection all live on it.
  • Availability over consistency — a just-uploaded video isn't watchable until the pipeline finishes and the caches and CDN warm up.
  • The processing pipeline is operationally heavy — an orchestrator, a worker fleet, and piles of intermediate blobs to garbage-collect.
  • Storing every segment in every format multiplies storage several-fold over the original source file.

What's expected at each level

  • Mid-level — a working end-to-end flow for upload and playback, clear API endpoints and data model, and the two key convergences: hand bytes straight to a blob store, and stream in segments rather than downloading the whole file.
  • Senior — moves quickly through the high-level design to spend real time on the hard parts, explaining how multipart upload enables resumable uploads and how a video is post-processed into parallel-transcoded, multi-format segments for adaptive streaming, with the trade-offs argued unprompted.
  • Staff+ — reasons from practice, naming concrete technologies (Cassandra, S3, Temporal, ffmpeg) and why they fit; anticipates the hot-key and streaming-distance bottlenecks and pre-empts them with caching, replication, and a CDN; and steers the conversation as a peer rather than being steered.
