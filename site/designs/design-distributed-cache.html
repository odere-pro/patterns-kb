Case study

Distributed Cache

A distributed cache stores key-value pairs in memory across many machines so it can hold far more data — and serve far more traffic — than any single node could. The design starts as a one-box hash table and grows outward: partition the keyspace across a fleet, replicate each shard for survival, and shave every millisecond off the path a client takes to reach the right node. It is the reference build of the distributed-cache pattern.

Design

Understanding the problem

Callers want three primitives — read a value by key, write one, delete one — answered from memory in single-digit milliseconds. A single-node cache does that trivially with a hash table, but it is capped by one machine's RAM and one machine's network card. The whole exercise is what happens when the data no longer fits and the traffic no longer fits: the cluster has to cooperatively partition the keys across nodes and keep redundant copies so a lost node does not lose data. This question sits deliberately astride two interview styles — the low-level data-structure design of the single node, and the distributed-systems design of the fleet — so the answer has to earn both halves.

Requirements

Functional

  1. Set, get, and delete a value by key.
  2. Attach an optional time-to-live (TTL) per key so entries expire on their own.
  3. Evict on a Least-Recently-Used (LRU) policy once the cache is full.

Out of scope: a user-configurable cache size, and alternative eviction policies (LFU, FIFO, custom) — worth naming to the interviewer, but LRU is the one to build.

Non-functional

  • Availability — highly available, with eventual consistency accepted; a slightly stale read beats a failed one.
  • Latency — under 10 ms for get and set.
  • Scale — up to 1 TB of data and a peak of 100k requests/second.

Explicitly not in scope: durability across restarts, strong consistency, rich queries, and transactions — a cache, not a database.

Back-of-envelope: how many nodes?

Two independent estimates set the cluster size, and the larger one wins.

By throughput. Take a working assumption of ~20,000 req/s that one node sustains before latency degrades. 100,000 ÷ 20,000 = 5 nodes as a floor; pad for spikes and in-flight failures and call it ~8.

By storage. A 32 GB-RAM instance leaves roughly 24 GB usable for cached data once process and OS overhead are subtracted. 1 TB (1024 GB) ÷ 24 GB ≈ 43 nodes; round up for headroom to ~50.

Storage (50) dwarfs throughput (8), so provision for storage — 50 nodes comfortably covers the request rate as a side effect. Plan for a ~50-node cluster. These figures are deliberately rough: real capacity depends on instance type, value size, read/write mix, and replication overhead, so the honest answer always ends in "benchmark the actual workload."

Core entities

The domain is as thin as it gets — keys and values — but the storage representation carries two extra facts that make the requirements work:

  • Key — the opaque string a caller hands in; also what the router hashes to find the owning node.
  • Value — stored not bare but as a (value, expiry) pair, so a read can check the TTL before returning.
  • Node — internally each entry is wrapped in a linked-list node (key, value, expiry) so LRU order can be tracked in O(1); this is an implementation detail, not part of the API contract.

The interface

Three operations, one per functional requirement. Real clusters speak a compact binary protocol over pooled TCP connections rather than HTTP, but the surface reads the same:

API — set, get, delete 
POST /{key}
{ "value": "…", "ttl_seconds": 3600 }   # ttl optional
→ 200 OK

GET /{key}
→ 200 { "value": "…" }               # 404 if absent or expired

DELETE /{key}
→ 204 No Content

How the system is built

Start at the node, then multiply it. A single node is two data structures working together: a hash table for O(1) lookup by key, and a doubly linked list that orders entries by recency so the least-recently-used one is always at the tail, ready to evict. Following the simplest-thing-first instinct of KISS, the get/set/delete logic is built and made correct on one box before any of the distribution is added — the cluster is just this node, replicated and sharded.

TTL needs two mechanisms. Lazy expiry is free — a read checks the timestamp and drops the entry if it is stale — but lazy expiry alone lets untouched-but-expired entries squat in memory forever. So a background janitor periodically sweeps the list and reclaims them; running it on a coarse schedule (say once a minute) trades a little CPU for tighter memory, and that dial is itself a tuning knob.

Multiplied out, the cluster is ~50 such nodes. A client-side router hashes each key onto a consistent-hash ring to find its owner, opens the request over a pooled connection, and every shard streams asynchronously to a replica so a node loss does not become a data loss.

Pseudocode — the O(1) LRU node 
class Node:  # key, value, expiry — one link in the LRU list

get(key):
    node = table[key]
    if node.expiry and now() > node.expiry:
        evict(node); return MISS
    move_to_front(node)          # O(1): mark most-recently-used
    return node.value

set(key, value, ttl):
    node = table.get(key) or Node(key)
    node.value  = value
    node.expiry = now() + ttl if ttl else None
    move_to_front(node)
    table[key] = node
    if size > capacity:
        evict(tail.prev)         # O(1): drop least-recently-used

Deep dives

1 · Staying available when a node dies

High availability means keeping more than one copy of every shard, and the interesting choice is when the copies must agree. Replication offers a spectrum:

  • Synchronous — bad here. The write blocks until every replica acknowledges. That buys strong consistency we explicitly said we do not need, and it puts a replica's latency on the critical path of a 10 ms budget. Wrong trade for a cache.
  • Asynchronous — good. The primary answers immediately and ships the write to replicas in the background. It honours the latency and availability requirements; the price is a small window where a failover can lose the last few unacknowledged writes — acceptable when the source of truth lives elsewhere. Redis defaults to this, and its WAIT command lets a caller opt into replica acknowledgement per write when it matters ("semi-synchronous").
  • Peer-to-peer — good for scale. Drop the primary/replica distinction; every node owns some ranges and replicates others, gossiping updates around. It scales the furthest and removes the single-writer bottleneck, at the cost of more complex conflict handling. This is the Cassandra-style answer.

The interview move is to name the trade rather than declare a winner; asynchronous replication is the pragmatic default.

2 · Sharding the keyspace evenly

One node cannot hold 1 TB or absorb 100k req/s, so the keys must be split — sharded — across the fleet. The naïve router, hash(key) % N, distributes evenly but is catastrophic under change: move from 50 to 51 nodes and the modulus shifts almost every key to a new owner, so the whole cache misses at once. Consistent hashing fixes this by placing nodes and keys on a circular keyspace (via a hash like MurmurHash) and assigning each key to the first node clockwise; adding or removing a node only remaps the slice of keys in that node's arc, not the entire ring. It doubles as the routing table — the client computes the owner locally and skips a round trip to any central lookup service.

3 · Hot keys — reads and writes

Sharding assumes traffic spreads with the keys. A hot key breaks that assumption: a viral post, a flash-sale inventory counter, and one shard takes traffic the other 49 never see. It is usually a property of how the cache is used, but a strong answer raises it unprompted.

  • Hot reads. Vertically scaling the hot shard barely helps and wastes the rest. Better options fan the reads out: a dedicated cache tier for hot keys, read replicas of the shard, or — best — keeping several copies of a hot key on different nodes (e.g. suffixing it, key#1…key#3) so readers spread across them. This is the read-side story the Scaling Reads theme develops in full.
  • Hot writes. Harder, because copies must converge. Batching writes at the client — coalescing many increments into one flush — cuts the write rate to the shard dramatically; alternatively, shard the hot key itself across nodes with random suffixes and reconcile on read. See Scaling Writes.

4 · Squeezing out the last milliseconds

At single-node scale, "performance" is just the O(1) hash table. Across ~50 nodes it becomes a networking problem: how fast a client finds the owner, and how much chatter each request costs. Three levers, some already in place: consistent hashing removes the routing round trip; request batching folds many operations into one network call; and connection pooling — an object pool of long-lived TCP connections — avoids paying a handshake per request, which is what usually pins the p95/p99 tail.

Limitations & trade-offs

What it buys

  • Scales horizontally to 1 TB and 100k req/s by adding nodes, with each node doing O(1) get, set, and eviction.
  • Survives node loss through async replicas, and reshapes with almost no key movement thanks to the consistent-hash ring.
  • Holds the sub-10 ms budget: client-side routing, batching, and pooled connections keep the network path short.

What it gives up

  • Only eventual consistency — an async failover can serve a stale value or drop the last few unreplicated writes.
  • No durability: a node restart empties its shard until the working set refills from the source of truth.
  • Uniform sharding does not fix skew — hot keys still need copies or suffix-sharding bolted on as a special case.
  • More moving parts to operate: the ring, replication lag, and the TTL janitor all become things that can misbehave under load.

What's expected at each level

  • Mid-level — spends more time on the single-node design and reaches the right data-structure pairing (hash table + doubly linked list) for O(1) operations within the latency budget. Can problem-solve toward basic availability and scale — recognising that replication is needed and sketching a couple of ways to do it.
  • Senior — moves through the low-level design quickly and spends the bulk of the time on scale: weighing replication modes against the requirements, handling hot reads and hot writes, and reaching for consistent hashing as second nature. Justifies decisions from the stated non-functionals rather than reciting options.
  • Staff+ — treats all of the above as assumed. The interesting signal is proactively surfacing the operational realities — replication lag and failover data loss, ring rebalancing during a node swap, janitor CPU versus memory pressure — and being explicit that this particular design is one point in a space of defensible combinations.
