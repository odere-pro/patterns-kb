Network

Resilience 0/9

Circuit BreakerStops calling a service that's already failing
Retry with BackoffRetries transient failures with a growing delay
BulkheadIsolates resources so one failure can't sink all
Health Endpoint MonitoringExposes a route that reports whether a service is healthy
Compensating TransactionUndoes completed steps when a later step fails
Queue-Based Load LevelingA queue absorbs bursts so consumers work at their own pace
Rate LimiterCaps how many requests a client can make in a window
Timeout / DeadlineCaps how long a call may take before giving up
Token BucketCaps the average rate but lets short bursts through

Routing & Scale 0/15

API GatewaySingle entry point that routes to backing services
Backend-for-FrontendOne backend tailored to each client type
AmbassadorA proxy that handles networking concerns for a service
SidecarAttaches a helper process alongside the main one
Load BalancerSpreads requests across many identical instances
Consistent HashingDistributes keys so adding a node reshuffles the least
ShardingSplits data across nodes by a partition key
GatekeeperA dedicated broker validates requests before they reach the service
Valet KeyA token grants temporary, limited access to a resource directly
CDNCaches and serves static content from the edge, near the user
AutoscalingAdds or removes instances automatically as load changes
Reverse ProxyFronts hidden backends, forwarding and terminating client connections for them
Service MeshMoves service-to-service networking into a managed proxy layer
Sticky SessionPins each client to one backend for the life of its session
Stateless ServiceKeeps no per-client state in memory, so any instance can serve any request

Principles — how to write it well

Not a rung and not a lens: each principle is a rule of thumb that holds at every elevation — the maxims that keep code simple, decoupled, and cheap to change. Each page cross-links to the patterns that embody it and the hazards it guards against.

Don't Repeat Yourself (DRY)Every piece of knowledge has one authoritative home in the system Keep It Simple (KISS)Prefer the simplest design that actually meets the requirement You Aren't Gonna Need It (YAGNI)Build capability only when a real present requirement demands it Single Responsibility PrincipleA class should have one reason to change — one actor it answers to Open/Closed PrincipleOpen for extension, closed for modification — add behavior by adding code, not editing it Liskov Substitution PrincipleA subtype must stand in for its base type without breaking the caller Interface Segregation PrincipleMany small, client-specific interfaces beat one fat general-purpose one Dependency Inversion PrincipleHigh-level policy and low-level detail both depend on an abstraction the policy owns Composition over InheritanceBuild behavior by assembling small objects, not by inheriting a deep class hierarchy Law of DemeterA method should talk only to its immediate collaborators, never reach through them Separation of ConcernsDivide a system so each part addresses one distinct concern, understood in isolation

Lenses

Not another rung on the ladder — these reshape how you build at whatever elevation you're already working.

Concurrency 0/8

Producer-ConsumerDecouples generating work from processing it
Thread PoolReuses a fixed set of worker threads
Future / PromiseA placeholder for a result that isn't ready yet
Actor ModelIsolated actors that only communicate via messages
ReactorOne loop dispatches events to their handlers
Monitor ObjectSynchronizes access to an object's own methods
Read-Write LockMany readers, or one writer — never both
BackpressureA slow consumer signals upstream to slow down

Messaging 0/14

Publish-SubscribePublishers and subscribers never know about each other
Message QueuePoint-to-point channel, each message consumed once
Message RouterDirects a message to the right channel based on rules
Content-Based RouterRoutes a message by inspecting its actual content
Message TranslatorConverts a message between two different formats
SplitterBreaks one message into many for individual processing
AggregatorCombines many related messages back into one
Competing ConsumersMany consumers pull from one channel to share load
Dead Letter ChannelUndeliverable messages land somewhere visible, not lost
Correlation IdentifierTags related messages so replies can be matched up
Scatter-GatherBroadcasts a request, then merges the responses
Wire TapCopies messages to a side channel for inspection, unchanged
Claim CheckSends a reference to large data, not the data itself
IdempotencyThe same request applied twice has the same effect as once

Caching 0/5

Cache-AsideThe app checks the cache first, fills it on a miss
Read-ThroughThe cache loads on a miss, the app never sees the gap
Write-ThroughWrites hit the cache and the store together, synchronously
Write-BehindWrites hit the cache now, the store catches up later
Refresh-AheadProactively refreshes hot entries before they expire

Domain-Driven Design 0/6

EntityDefined by identity, not by its attributes
Value ObjectDefined only by its attributes, no identity
AggregateA cluster of objects treated as one consistency unit
Domain EventSomething meaningful that happened in the domain
Bounded ContextA model's boundary of validity and meaning
Anti-Corruption LayerTranslates between two different domain models

Functional 0/6

FunctorA type you can map a function over
MonadSequences computations that carry extra context
Pipeline / CompositionBuilds behavior by composing small functions
ImmutabilityNever mutate — always return new data
Lens / OpticsA composable getter/setter into nested data
CurryingPre-fill some arguments, get back a function

Testing 0/9

Dummy ObjectPassed in to satisfy a signature, never actually used
Test StubReturns canned answers, doesn't care how it's called
Test SpyRecords how it was called so the test can check later
Mock ObjectPre-programmed with expectations, fails if they're unmet
Fake ObjectA working lightweight stand-in, like an in-memory database
Page ObjectWraps a UI screen's structure behind one testable interface
Test Data BuilderFluent construction of valid objects for test setup
Golden MasterCompares fresh output against a saved known-good snapshot
Arrange-Act-AssertSet up state, perform the action, check the outcome

Security 0/7

Authentication EnforcerCentralizes verifying who is making the request
Authorization Enforcer (RBAC)Centralizes checking what a role is allowed to do
Secure Session ManagerIssues, tracks, and expires session state safely
Intercepting ValidatorValidates and sanitizes input before it reaches logic
Secure LoggerRecords events for audit without leaking sensitive data
Single Access PointOne well-guarded entry, instead of many small ones
Least PrivilegeEvery component gets the minimum access it needs, no more

Known Hazards

Anti-patterns — not to practice, just to recognize on sight. Every one is what the patterns above exist to prevent.

God ObjectOne class knows and does almost everything
Spaghetti CodeControl flow too tangled to follow
Big Ball of MudNo discernible architecture at all
Anemic Domain ModelData objects with no real behavior of their own
Golden HammerOne familiar tool applied to every problem
Boat AnchorDead code or hardware kept around unused
